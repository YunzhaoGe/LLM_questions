# 视频理解链路（时序采样 → Token 化（帧/Patch）→ 长视频理解的算力怎么控）

把“视频输入”如何一路变成“语义输出”讲清楚：时序采样 → Token 化（帧/Patch）→ 时序建模 → 读出与任务头 → 成本来源 → 算力控制 → 评测与失败模式。

---

## 1. 时序采样：原始视频到片段

1. 帧率与时长规范化：把不同 fps、不同长度映射到统一时间网格（为了让训练与推理在一致时间尺度上工作）
2. 片段切分（clip）：把长视频切成固定窗口或随机窗口（为了把输入长度变成可预算的序列）
3. 采样与镜头边界：用均匀/分层采样并避开跨镜头拼接（为了在有限帧数里提升信息密度并减少时序不连续噪声）

---

## 2. Token 化：像素到序列

1. 粒度选择：帧级 token 省预算，patch 级 token 保细节，tubelet 兼顾时空压缩（为了在“保真”和“token 数量”之间做取舍）
2. 表征来源：连续特征、离散码本或时空 latent（为了匹配任务需求与系统预算）
3. 稳定性边界：tokenizer 的时间稳定性会影响后续时序建模可靠性（为了避免把闪烁/拖影伪影当成事件线索）

---

## 3. 时序建模：从“看见”到“看懂变化”

1. 编码：把 token 序列映射成多层表征（为了从纹理逐步抽象出语义）
2. 时间关系：建模跨帧的动作、变化与因果线索（为了回答“发生了什么”，而不只是“长什么样”）
3. 多尺度时间：短窗负责局部动作，长窗负责事件与叙事（为了兼顾细节与长程依赖）

---

## 4. 读出与任务头：长序列到输出

1. 汇聚：把长序列压缩到少量表示（为了让输出长度与计算开销可控）
2. 对齐：把表示对齐到任务形式（分类、检索向量、问答、字幕等）（为了让下游系统可直接消费）
3. 不确定性处理：对信息缺失与遮挡保留“不确定/需要更多片段”的余地（为了减少自信但错误）

---

## 5. 成本来源：长视频为什么会贵

1. Token 数：时间与分辨率上升会导致序列长度快速增长（为了保真你不得不保留更多 token）
2. 交互范围：跨很远时间做信息交换会显著放大计算与显存压力（为了理解长程因果与一致性）
3. 数据与稳定性：快切、抖动、压缩伪影会增加训练与泛化难度（为了学到稳定规律需要更干净数据与更稳配方）

---

## 6. 算力控制：把“长”拆成可预算问题

1. 采样提信息密度：关键片段优先、镜头级结构化输入（为了减少无效帧与无意义跨镜头建模）
2. Token 压缩与分层：时空 tokenizer、低清长时序 + 高清短时序、动态 token 分配（为了降低 token 基线并把预算对齐到变化）
3. 建模近似：滑窗为主 + 少量全局摘要（为了让成本更接近随时长线性增长并保留长程线索）

---

## 7. 评测与失败模式

1. 静态偏置：只看外观，答不出事件变化（为了定位采样与时序建模是否不足）
2. 长视频遗忘：前情无法影响后续判断（为了暴露典型长时依赖问题）
3. 伪影误导：压缩块效应、抖动被当成动作线索（为了检验数据与 tokenizer 时间稳定性）

---

## 8. 一句话总结

1. 视频理解的难点是“在预算内看懂变化”，不是“把每帧看清楚”
2. 算力控制的主战场是 token 数与时间交互范围
3. 可落地的路径通常是：采样提信息密度 → token 压缩与分层 → 局部时序建模 → 少量全局摘要
