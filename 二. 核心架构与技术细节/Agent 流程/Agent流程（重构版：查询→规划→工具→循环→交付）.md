# Agent 流程（查询 → 规划 → 工具 → 循环 → 交付）

目标：起点是用户指令，终点是交付可用结果并记录轨迹。流程主线是：初始化与装载（1）→ 规划决策（2）→ 工具与观察（3/4）→ 循环迭代（5）→ 最终交付（6），工程级优化（7）是贯穿全程的可选增强。每一步都同时回答“做了什么/为什么这么做”，并配一句“就像……”的类比，帮助建立直观的 Agent 心智模型。

---

## 1. 初始化与上下文装载（让 Agent “开机就绪”）

1. 组件初始化：加载 LLM 后端（支持 function calling/JSON schema）、工具集（web_search/API/file 等）、状态管理器与关键配置（max_steps、temperature、预算/超时）（为了把“能想、能做、能记”三件事一次性备齐，并把成本与风险提前纳入系统约束；就像给机器人装上大脑、工具箱和笔记本，还把油量和行程上限写进仪表盘）
2. 会话与权限接入：读取历史对话、已有工具结果、用户偏好与安全策略，并绑定工具权限与数据访问范围（为了让 Agent 在多轮中保持一致，同时不越权调用工具/数据；就像接手工单先读历史，再把门禁卡权限刷清楚）
3. 输入规范化：把用户指令转成结构化任务定义（目标、约束、输出格式、验收口径，必要时补充上下文）（为了让后续“完成性评估”可判定，避免做对了事却交错了格式；就像把需求写成可验收的交付标准）

---

## 2. 规划与决策（把问题拆成可执行步骤）

1. 初始计划生成：把任务拆成步骤（Planner-Executor/Plan-and-Execute），或使用 ReAct 的“边想边做”（为了给复杂任务提供路线图，并允许在不确定处用工具补证据；就像先画施工图，同时允许现场勘测后微调）
2. 当前步决策：结合 system 规则 + 历史状态 + 成功条件，决定下一步的行动类型（工具/检索/计算/生成/校验）（为了让每一步都能对齐“可验收交付”，而不是只对齐“看起来合理”；就像每个工序都对齐最终验收标准）
3. 计划自适应：根据 observation 动态重排步骤、剪枝无效分支、必要时并行探索多个候选路径（为了减少走弯路并提升复杂任务成功率；就像遇到封路就改路线，必要时多条路线同时探路）

---

## 3. 工具调用（把“想”变成“拿到证据”）

1. 工具选择与路由：由 LLM 决定用哪个工具与参数，必要时用工具路由/规则做兜底（例如 `web_search(query=...)`）（为了在“该查证据/该算数/该读文件”之间做稳定选择；就像把问题分配给最合适的专家）
2. 结构化调用与校验：用 JSON schema/function calling 约束参数与返回结构，并对输出做基本校验与标准化（为了减少“调用形态不稳定”导致的连锁失败；就像表单不仅要填对字段，还要把回执整理成统一格式）
3. 并发与容错：对可并行的工具请求并发执行，并处理超时/限流/空结果的重试、降级与切换（为了降低端到端延迟并提升鲁棒性；就像多条线同时查资料，且任何一条失败都有备选方案）

---

## 4. 观察与状态更新（把工具结果变成“可继续推理的状态”）

1. Observation 回填：把工具输出写回状态（原文、关键字段、来源与时间），并把“结论”与“证据”分离存放（为了让后续校验能追溯到证据，而不是只剩下摘要结论；就像工单里既要写结论，也要贴原始附件）
2. 记忆分层：短期上下文保留细节，长期记忆用摘要/向量索引沉淀可复用信息（为了在长会话里兼顾“可回忆”和“可计算”，同时控制 token；就像随身记事本记录当天细节，档案库保存长期可检索要点）
3. 证据质量控制：对冲突、过期、低可信来源打标并触发再检索/交叉验证（为了让 Agent 的“下一步推理”建立在更干净的证据上；就像质检不过的材料要复检或退货）

---

## 5. 循环迭代与反思（直到完成或触发终止）

1. 完成性评估：判断“是否已满足用户指令定义的成功条件”（目标、约束、格式与验收口径），否则进入下一轮（为了让流程有明确收敛标准；就像验收不过就继续返工）
2. 终止条件：达到目标、达到 max_steps/预算、触发 stop 信号或检测到无效循环（为了防无限循环与资源浪费，并保证任务在可控成本内结束；就像自动驾驶既有目的地也有油量与里程上限）
3. 反思与纠错（可选增强）：用 critic/reflection 或 verifier 发现缺证据、逻辑断点与不满足约束处，并回到相应步骤补齐（为了把“看似合理”变成“可验证正确”；就像审稿人指出引用不足，就回去补引用与证据）

---

## 6. 最终合成与交付（把“过程”变成“可用结果”）

1. 最终汇总：基于全部 observation 与中间结果生成最终答案，并按用户要求输出结构（自然语言/JSON/表格等）（为了把分散证据组织成用户可直接使用的交付物；就像把材料整理成一份可直接提交的报告或表单）
2. 交付前校验：做 groundedness/一致性检查，必要时进行结构化输出校验与引用补全（为了减少幻觉，并保证输出满足格式约束与可审计；就像交付前核对每条结论都能在附件里找到依据）
3. 返回与可观测：流式或一次性返回，并记录轨迹（actions/observations/errors/latency）与关键指标（成功率、工具失败率、成本）（为了用户体验与线上可运维；就像生产线有交付单也有仪表盘）

---

## 7. 工程级优化（不改变语义，但决定真实稳定性与成本）

1. 并行与批处理：并行调用多工具、服务端做 continuous batching，并把独立子任务拆成可并发执行的 DAG（为了降低端到端延迟并提高吞吐；就像把项目拆成可并行施工的多条工线）
2. 缓存与复用：对检索与工具结果做缓存，对重复前缀与常见中间结论做复用，对长历史做摘要与可检索记忆（为了减少重复开销并降低 token/工具成本；就像常用资料做成速查表，且把阶段性成果归档复用）
3. 可观测与护栏：全链路 trace、评测与回放、失败率/超时监控、工具白名单与权限控制（为了能定位问题、持续迭代，并防越权与误用；就像生产线既有摄像头回放也有门禁与质检规范）
