# 视觉 Agent / Computer Use 精准点击链路（屏幕观测 → 目标定位 → 点击执行 → 反馈校验）

把“Computer Use 如何观测屏幕并实现精准点击”讲清楚：采集屏幕观测（截图 + A11y/DOM）→ 统一坐标系与可点击目标集合 → grounding 定位目标元素 → 生成点击动作参数与事件序列 → 注入 OS 输入 → 采样反馈并校验/回退。

---

## 1. 观测采集：把桌面/网页变成可输入的信号

1. 获取截图：从系统或浏览器拿到当前屏幕帧（为了让模型看到“像素级真实界面”）
2. 获取结构：读取可访问性树（A11y Tree）、DOM 或 UI 控件层级（为了让模型拿到“可点击对象”的语义与层级）
3. 同步时间：将截图与结构快照绑定同一时间点，并记录窗口尺寸/缩放/滚动状态（为了避免动作落在过期坐标上）

---

## 2. 坐标统一：把“元素边框”对齐到“真实点击点”

1. 坐标换算：把 DOM/A11y 的 bounds 映射到屏幕像素坐标，处理 DPI、缩放、窗口偏移与滚动偏移（为了避免点在“逻辑坐标”而不是“物理像素”）
2. 可见性裁剪：用视窗范围裁剪或标记不可见区域，识别被遮挡/在折叠区/在弹层后面的元素（为了避免点到看不见或点不到的对象）
3. 输入坐标系固定：统一规定“点击点”的参考系（屏幕/窗口/视口）并随同观测一起传递（为了让后续动作注入不发生坐标漂移）

---

## 3. 目标集合：把“屏幕上能点的东西”列成候选清单

1. 抽取可交互节点：从 DOM/A11y 里筛选可点击/可聚焦/可输入的节点（button/link/input/menuitem 等）（为了把搜索空间收敛到“可操作对象”）
2. 生成候选条目：为每个候选生成描述信息（文本、role、层级路径、邻近标签、bounds）（为了让模型能用语义与布局共同定位）
3. 关联视觉证据：把候选条目与截图局部区域绑定（可视为“框 + 文本”）（为了在结构缺失或标签不完整时仍能回到像素验证）

---

## 4. 目标定位（Grounding）：从“我要点哪个”到“点哪里”

1. 语义匹配：用用户指令中的关键词、同义词与上下文线索在候选清单中检索目标（为了找到“最像的那一个按钮/链接/输入框”）
2. 消歧判定：当候选相似时，结合相对位置（在谁旁边/在表单里/在弹窗上）、层级结构与视觉图标做二次筛选（为了避免点到“同名按钮”的错误目标）
3. 落点生成：把目标元素的 bounds 转成点击点（中心点/可点击区域内偏移点）并给出置信度（为了把目标元素转成可执行坐标）

---

## 5. 点击动作生成：把“一个点击”变成可执行输入序列

1. 视口就绪：若目标不在视口内，先滚动/切换标签/唤起窗口让目标进入可见范围（为了让点击发生在可交互状态）
2. 事件序列化：把点击动作展开成可控事件序列（移动到点位 → 按下 → 释放，必要时加 hover/focus）（为了兼容不同 UI 对输入时序的依赖）
3. 参数校验：检查点位是否在窗口内、是否落在可点击区域、是否明显偏离目标框（为了在执行前拦截“错点/越界”）

---

## 6. 点击执行：把动作注入 OS/浏览器并等待生效

1. 注入输入：通过系统/浏览器的输入注入接口发送鼠标事件到目标窗口（为了让“模型的决定”真正作用于界面）
2. 等待稳定：等待 UI 帧稳定与事件冒泡完成，处理加载动画与异步刷新（为了避免马上采样到半更新状态）
3. 二次采样：立刻重新采集截图与结构快照（为了拿到“点击之后”的新观测用于校验）

---

## 7. 校验与回退：保证“点对了”，而不是“点了”

1. 成功判定：用结构变化与视觉变化验证是否达成目标（按钮状态变化、弹窗出现、URL/标题变化、焦点落到输入框等）（为了把结果从主观观感变成可判定信号）
2. 失败归因：区分错目标、坐标漂移、目标被遮挡、点击被拦截（覆盖层/权限弹窗）、页面未加载完成（为了选对修复动作）
3. 回退重试：按原因触发纠错（重新采样再定位、先关闭遮挡弹窗、换一个落点偏移、改用结构节点触发点击、先滚动到元素）（为了在真实网页噪声下仍能稳定命中）
