# 打破 O(N²) 魔咒：百万上下文时代的注意力序列并行架构

长文本处理已是行业标配，但传统注意力计算在长序列下面临算力与显存的平方级爆炸。虽然张量并行和流水线并行能解决部分问题，但序列并行（Sequence Parallelism，如 Ulysses）才是应对超长上下文架构缺陷的直接方案。

---

## 1. 架构瓶颈：注意力子层的 O(N²) 现象

1. 算力爆炸：自注意力机制需要序列中每个 Token 与其他所有 Token 计算点积，计算复杂度随序列长度呈平方级增长（为了在长文本下维持完整的全局依赖）。
2. 显存墙限制：除了计算量，保存庞大的注意力打分矩阵（Attention Score）与中间激活值会瞬间耗尽单卡显存（为了支持反向传播或后续的上下文状态保留）。
3. 通信开销瓶颈：当单卡无法容纳超大矩阵时，跨卡分配会带来极大的网络传输负担（为了实现多卡协同计算并维持正确的矩阵维度）。

---

## 2. 并行局限：传统方案在极长序列下的劣势

1. 张量并行（TP）：沿模型隐藏层维度切分参数，虽能分担显存，但每次计算都需要全局聚合（All-Reduce），通信开销随序列长度剧增（为了保证多卡之间的参数维度一致性）。
2. 流水线并行（PP）：按 Transformer 层切分模型，处理长序列时由于单层计算时间大幅拉长，流水线气泡（Pipeline Bubble）造成的算力闲置被显著放大（为了避免等待前置节点的计算结果而导致 GPU 空转）。
3. 序列切分需求：长文本场景必须直接沿序列维度（Sequence Dimension）进行切分，让不同的 GPU 处理序列的不同片段（为了真正突破单卡显存对输入上下文长度的物理限制）。

---

## 3. 核心机制：Ulysses 序列并行的计算重构

1. 序列切分与分配：将超长输入序列沿长度维度等分为多个子片段，均匀分配到分布式集群的多个设备上（为了降低单卡需要处理的序列长度基数）。
2. 局部投影计算：每个设备先独立对分配到的序列片段进行 Query、Key、Value 的线性投影计算（为了最大化利用单卡的计算能力，避免此阶段产生通信）。
3. 维度转换通信：通过 All-to-All 通信，将原本按“序列切分”的数据重新排布为按“注意力头（Attention Head）切分”（为了让每个设备获取完整序列的子集特征，从而独立完成完整序列的局部注意力打分）。

---

## 4. 参数一致性：位置编码与掩码的跨节点同步

1. 坐标锚定：即使序列被物理切分，每个 Token 仍需保留其在全局序列中的绝对或相对位置参数（为了保证旋转位置编码 RoPE 的数学等价性）。
2. 掩码重组：自回归因果掩码（Causal Mask）在按注意力头切分后，需要根据子序列的全局偏移量重新对齐（为了避免在分布式计算中错误地看到未来 Token）。
3. 状态恢复：注意力计算完成后，再次通过 All-to-All 通信将按头切分的结果恢复为按序列切分的状态（为了将特征向量无缝送入后续的 MLP 前馈网络子层）。

---

## 5. 工程权衡：通信带宽与显存占用的博弈

1. 显存收益：序列并行将 O(N²) 的显存占用均摊到多个节点，使得百万级 Token 推理成为可能（为了打破单节点硬件显存天花板）。
2. 网络压力：All-to-All 通信高度依赖集群的节点间带宽（如 NVLink 或 InfiniBand），网络拓扑设计成为核心考量（为了防止通信耗时掩盖并行计算带来的收益）。
3. 混合部署策略：在实际生产中，通常将序列并行（SP）与张量并行（TP）及数据并行（DP）混合使用（为了在不同维度的计算墙与通信墙之间找到全局最优解）。
