# PagedAttention 与 vLLM 显存调度链路（新请求 → 页表映射 → KV 写入/读取 → 追加/抢占 → 回收）

把“一个请求如何在 KV cache 显存里稳定增长而不 OOM”讲清楚：前缀命中复用 → 逻辑块与物理页（页表）→ prefill 把 KV 写进页 → decoding 读取历史页并追加新页 → 显存压力下的抢占/换出/重算 → 结束回收与碎片抑制。

---

## 1. 请求入场：为 KV cache 增长准备运行时状态

1. 构造请求状态：将 prompt token、采样参数与停止条件封装为请求状态（为了让引擎能跟踪“已生成到哪、还需要多少 KV”）
2. 记录资源画像：估算该请求的潜在最大长度/交互性优先级/可抢占性（为了在显存紧张时决定谁先拿页、谁先让页）
3. 进入调度批：把多个请求合并为执行批并对齐运行时元数据（为了让同一套页表与注意力内核能并行处理多条序列）

---

## 2. 前缀复用：命中时直接“继承已有 KV 页”

1. 形成前缀键：将 prompt token 序列映射成可检索的前缀表示（为了在缓存里快速定位“相同开头的历史”）
2. 找最长命中：在前缀索引中找到最长公共前缀对应的 KV 页集合（为了把 prefill 的一段计算直接复用掉）
3. 切分增量段：把未命中部分作为本次 prefill 的输入，并把“已有页表”作为起始状态（为了让新 KV 追加在旧页之后）

---

## 3. 页表抽象：让注意力从“连续内存假设”变成“按页间接寻址”

1. 固定粒度分块：把序列按固定大小切成逻辑块（block），块内顺序连续、块与块之间可不连续（为了让“增长”不依赖大块连续显存）
2. 建立块表（block table）：维护每条序列的“逻辑块 → 物理页”映射（为了让注意力内核能通过块表去 gather 对应 KV）
3. 维护增长指针：记录每条序列当前写到第几页/页内偏移与总长度（为了在 decoding 时把新 token 的 KV 写到正确位置）

---

## 4. 物理分配：从页池拿页，把块表补齐

1. 申请页框：从全局显存页池中分配空闲物理页（为了在显存不足时仍能做可控的资源管理）
2. 填写页表：将新分配的物理页写入映射表，并同步到内核可读的数据结构（为了让注意力计算能正确定位 KV）
3. 处理不足：若空闲页不足，触发抢占/换出/延迟执行等策略（为了避免直接 OOM 导致整批失败）

---

## 5. Prefill：把增量 prompt 写成“可复用的 KV 页”

1. 前向生成 KV：对未命中段运行 prefill，得到每层 K/V（为了补齐这段 token 的历史记忆）
2. 分页落盘：把每层 K/V 写入对应的物理页，并推进增长指针与块表（为了让后续 decoding 只读页表就能找到全部历史）
3. 产出首步输入：得到最后位置隐藏状态与最新块表/增长指针（为了开始 decoding 并持续追加）

---

## 6. Decoding：每一步都“读旧页 + 写新页”

1. 历史读取：注意力内核按块表 gather 历史页里的 K/V 来计算下一 token（为了不依赖 KV 的连续内存布局）
2. 追加写入：把新 token 的 K/V 写入当前页，页满则从页池再拿一页并扩展块表（为了让序列在显存里稳定增长）
3. 循环推进：重复“按块表读历史 → 采样 → 追加写页”的闭环直到停止（为了完成整段生成）

---

## 7. 显存压力：当页不够时，系统如何“不断服务而不是崩溃”

1. 选择牺牲者：基于优先级、剩余长度、交互性等指标选出要被抢占的请求（为了把资源让给更重要的请求）
2. 执行处置：把其 KV 页换出到 CPU/SSD，或直接丢弃并在恢复时重算 prefill（为了用“算力换显存”保持系统可用）
3. 恢复续跑：当资源回补后，按保存的映射与进度恢复请求，必要时重建缺失的 KV（为了让长请求在压力下仍能推进）

---

## 8. 回收与碎片抑制：让页池长期稳定可用

1. 释放页：请求完成或取消后，把其占用的物理页归还到页池（为了让后续请求复用显存资源）
2. 清理索引：更新前缀缓存的引用计数与过期策略，移除不再可复用的条目（为了控制缓存膨胀与命中质量）
3. 抑制碎片：通过分页池、紧凑数据结构与批级调度策略减少碎片与抖动（为了让吞吐与 TTFT 更稳定）
