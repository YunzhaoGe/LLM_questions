# 为什么系统会在检索过程中不断改写问题？——从用户输入到 Rewrite 输出的一条真实主线

很多人第一次听到 query rewrite，会以为这只是一个小动作：

用户提一个问题，系统把句子改写得更像搜索词，然后再去检索。

这个理解不能说错，但它太浅了。真正难的地方，从来不是“把一句话换个说法”，而是：

- 系统怎么知道这句话缺了什么
- 系统怎么知道应该先补哪个
- 系统怎么知道补完之后真的比原问题更适合检索

所以这篇文章不再泛泛讲“query rewrite 是什么”，也不再讲一堆抽象模块。我们只做一件事：

**用一个真实风格的例子，把“用户输入 → 系统判断 → 系统补充 → rewrite 输出”完整走一遍。**

---

## 一、先说结论：Rewrite 的目标不是“更漂亮”，而是“更可执行”

在工程里，一个好的 rewrite 结果，不是更长的句子，也不是更专业的句子，而是一个更适合检索系统执行的问题表示。

它至少要做到三件事：

- 明确系统到底该找什么
- 缩小无关结果的范围
- 帮系统决定该优先查哪些数据源

如果一个 rewrite 结果只是把原话说得更复杂，但检索范围、数据源选择、结果排序都没变化，那它大概率不是真正有用的 rewrite，只是 paraphrase，也就是换一种说法复述原话。

---

## 二、先把场景说清楚：我们到底在改写什么问题

下面我们用一个非常典型的企业内部问答场景来讲。

假设你的系统后面接了三类知识源：

- 产品流程文档
- 历史工单
- 发布变更记录

用户在内部助手里提了这样一个问题：

> “为什么这个客户一直开户失败？”

这句话非常像真实用户会说的话。它对人类同事来说也许够用了，因为人脑会自动脑补很多背景。但对检索系统来说，它远远不够直接执行。

为什么？

因为系统现在只知道：

- 有一个客户
- 在做开户这件事
- 结果失败了
- 用户想问原因

但系统还不知道：

- 是哪个地区的客户
- 是个人客户还是企业客户
- 失败发生在流程哪个阶段
- 有没有报错码
- “一直”到底是一周、一个月还是某次版本发布之后

所以接下来真正要做的，不是直接回答，而是先把这句话一步步压缩成一个可执行的检索输入。

---

## 三、第一步：先拆骨架，不要急着改写

**这一步是什么：** 把用户原始问题拆成最基础的结构化信息。

**为什么要做：** 因为如果系统连“用户已经明确说了什么”都没有拆开，后面的所有补充都会变成无约束发挥。

最常见的错误，就是系统在第一步就偷偷把“开户失败”理解成了“身份校验失败”或者“地区配置异常”。这都太早了。它们最多只能是候选解释，不能在一开始就当成事实。

**怎么做：** 第一版实现时，最稳妥的方法不是让模型自由发挥，而是做一次固定字段抽取。

对这句话，系统至少应该先拆出：

- 对象：这个客户
- 动作：开户
- 现象：失败
- 任务意图：原因分析
- 时间描述：一直
- 原始文本：为什么这个客户一直开户失败

这里有一个很重要的工程细节：**原始文本必须保留。**

因为后面你再怎么补地区、补错误码、补流程阶段，都必须能回头检查一件事：我是不是还在围绕“开户失败”这个原问题工作，还是已经被某个候选假设带偏了。

**这一步的输出是什么：** 输出不是 rewrite 结果，而是一张“问题骨架卡片”：

- 用户问的是原因排查
- 原句里只有对象、动作、现象和模糊时间
- 还没有足够的信息支持直接检索

---

## 四、第二步：拿一张模板来查缺口

**这一步是什么：** 把当前问题对齐到一张“原因排查类问题模板”上，明确知道它缺了哪些关键槽位。

**为什么要做：** 程序没法处理“感觉这个问题有点泛”这种判断。程序需要的是：

- 缺的是哪一类信息
- 哪些缺口会影响检索
- 哪些缺口可以后补

也就是说，缺口判断必须相对于一个模板来做，而不是凭直觉。

**怎么做：** 第一版系统完全不需要支持所有问题类型。你只要先支持“原因排查”这一类，就已经很有价值。

这类问题的一张常见模板可以是：

- 对象是谁
- 现象是什么
- 时间范围是什么
- 环境是什么
- 错误信号是什么
- 想诊断的是原因、规则还是流程阶段

然后把当前问题往这张模板里填。

一填你就会发现：

- 已知：对象、现象、任务类型
- 模糊：时间里的“一直”
- 缺失：地区、客户类型、错误码、失败阶段、版本

接着还要做一层分级。不是所有未知项都要立刻补，通常要分成三类：

- 必须补：不补就几乎没法有效检索
- 建议补：不补也能搜，但结果会发散
- 暂不补：暂时不影响首轮判断

在这个例子里，一个很典型的分级会是：

- 必须补：地区、错误码或失败阶段
- 建议补：时间范围、客户类型
- 暂不补：客户昵称、提问人部门

**这一步的输出是什么：** 到这里，系统终于能明确说出一句程序员爱听的话：

- 我现在不是“觉得问题太泛”
- 而是明确知道“我缺环境信息和错误信号”

这一步之后，rewrite 才真正开始有了方向。

---

## 五、第三步：先别急着补，先判断补哪个最值钱

**这一步是什么：** 对缺口做优先级排序。

**为什么要做：** 真实问题永远缺很多信息。如果系统见一个补一个，最后只会得到一条越来越长、但不一定越来越有用的 query。

所以系统必须回答：**如果我现在只能补一个条件，补哪个最值？**

**怎么做：** 一个非常实用的工程判断标准是看这个缺口会不会改变三件事：

- 会不会显著缩小召回范围
- 会不会改变数据源路由
- 会不会把高概率候选原因拉开

放到这个例子里，为什么“地区”优先级高？

因为一旦补进地区，通常会同时发生三件事：

- 文档从全局开户流程文档缩到地区规则文档
- 工单从全量历史问题缩到本地区相似问题
- 候选原因从“所有开户失败原因”缩到“该地区常见失败原因”

为什么“错误码或失败阶段”也很关键？

因为它会直接决定系统接下来查的是：

- 流程说明
- 异常工单
- 某条具体校验链路

而像“客户昵称”这种信息，通常不会改变检索范围，也不会改变候选原因区分，所以优先级就低。

**这一步的输出是什么：** 经过排序，系统大概率会得出这样的结论：

- 第一优先补：地区
- 第二优先补：错误码或失败阶段
- 第三优先补：时间范围
- 第四优先补：客户类型

注意，这一步结束时，系统知道了“补什么最值钱”，但还不知道“具体补什么值”。

---

## 六、第四步：先列候选原因，再反推到底缺什么

**这一步是什么：** 建立一棵很小的候选原因树。

**为什么要做：** 因为系统只有知道自己在区分什么，才知道缺口是什么。

如果系统根本不知道当前混淆的是“地区配置问题”和“版本变更问题”，它就不会知道地区和发布时间为什么重要。

**怎么做：** 第一版不要指望模型凭空生成完美原因树。最实用的做法，是先手工维护一棵受控的小树。

对“开户失败”这种问题，一棵很小的树就够用了：

- 身份校验失败
- 权限校验失败
- 地区配置异常
- 最近版本变更引入
- 上游依赖异常

然后再提前写好“区分这些原因需要什么条件”：

- 地区配置异常 vs 版本变更引入：需要地区、发布时间
- 权限校验失败 vs 身份校验失败：需要错误码、失败阶段
- 个例异常 vs 批量异常：需要相似工单分布

这里最关键的一条原则是：

**候选原因只能指导补条件，不能直接当成最终结论。**

也就是说，系统现在可以怀疑“可能是地区配置异常”，但不能因此直接把主查询改成“查地区配置异常”。它只能据此得到一个更保守的动作：优先去补地区条件，并查一查相关变更记录。

**这一步的输出是什么：** 系统得到的不是答案，而是：

- 当前最值得关注的几条候选路径
- 每条路径之间的分界条件
- 也就是下一步真正该补的字段

到这里，rewrite 已经不再是“把原话补长一点”，而是在为分叉判断补条件。

---

## 七、第五步：做一轮轻量侦察检索，专门找“可补的值”

**这一步是什么：** 做首轮试探性检索，但目标不是找答案，而是找候选补充值。

**为什么要做：** 到现在为止，系统大概知道自己缺“地区”“错误码”“版本”，但还不知道这些字段的具体值是什么。

如果没有这一步，系统只能抽象地说“我缺地区信息”，却不知道到底该补“中国区”“日本区”还是“国际区”。

**怎么做：** 这一步要尽量轻、尽量宽、尽量便宜。

一个很典型的做法是：

- 用原问题做一次宽召回
- 同时查文档、工单、变更记录
- 只看前几条结果
- 重点抽高频出现但用户没说过的实体和术语

假设这轮侦察后，系统在结果里频繁看到了这些词：

- 中国区
- v3 身份校验链路
- 错误码 1021
- 实名校验失败

这时候，系统拿到的不是答案，而是一批可补候选项。

注意，这些词仍然不能直接当事实。它们现在的身份是：

- “中国区”是候选环境值
- “v3 身份校验链路”是候选流程值
- “错误码 1021”是候选错误信号
- “实名校验失败”是候选阶段描述

**这一步的输出是什么：** 这一轮检索后，系统终于不再只是知道“缺什么类型”，而是知道“可能补什么值”：

- 候选地区：中国区
- 候选链路：v3 身份校验链路
- 候选错误信号：1021
- 候选失败阶段：实名校验

这一步非常像侦探先做外围摸排。不是为了立刻破案，而是为了知道下一步去哪条路最值得深挖。

---

## 八、第六步：把候选值写回去，但要分清谁是硬约束，谁是软提示

**这一步是什么：** 真正进入 rewrite 生成阶段。

**为什么要做：** 因为系统现在不只是知道缺口，还拿到了可补的候选值。接下来最重要的，不是把这些词全堆进一句话，而是决定它们应该以什么身份进入查询。

**怎么做：** 最实用的办法，是把补进来的内容分成三层：

- 硬约束：直接缩小检索范围
- 软提示：影响排序，但不强制过滤
- 子查询：专门验证某条候选路径

放到这个例子里，一个比较稳妥的分配是：

- 硬约束：地区 = 中国区
- 硬约束：时间 = 最近两周或最近一个版本周期
- 软提示：v3 身份校验链路
- 软提示：实名校验失败
- 子查询一：最近版本发布后是否新增中国区身份校验限制
- 子查询二：错误码 1021 是否集中出现在个人客户开户流程

为什么“v3 身份校验链路”不直接当硬约束？

因为它来自侦察检索，还是候选值，不一定是用户已知事实。如果现在就把它写成强过滤条件，很可能把一批真正相关、但没有明确提到 v3 的证据给过滤掉。

所以这一步最重要的不是“补得多”，而是“补得分层”。

**这一步的输出是什么：** 到这一步，系统已经形成一个初步 Query 包：

- 主查询还保留用户原始意图
- 硬约束负责缩范围
- 软提示负责提排序
- 子查询负责查分叉

这时候的 rewrite，已经不再是“重写一句话”，而是在生成一个查询计划。

---

## 九、第七步：验证这次 Rewrite 真的带来了收益

**这一步是什么：** 做一次小型验收，判断 rewrite 是否有效。

**为什么要做：** 因为如果没有这一关，系统会天然倾向于“再补一点总是好的”。但实际工程里，多补一点往往意味着：

- 过滤过头
- 某条候选假设被误写成事实
- 结果没有更集中，只是 query 更长了

**怎么做：** 最务实的做法，是直接对比 rewrite 前后的检索结果，看三件事有没有变化：

- 结果是不是更集中
- 路由是不是更清晰
- 候选原因是不是更容易区分

例如，rewrite 前的 Top-K 可能是：

- 开户流程总览
- 开户常见问题
- 如何提交资料
- 开户失败处理指南

这些结果都不算错，但都太泛。

rewrite 后，如果 Top-K 开始变成：

- 中国区开户规则说明
- 错误码 1021 相关工单
- v3 身份校验链路变更记录

那说明 rewrite 确实改变了检索面。

如果 rewrite 后结果还是一堆泛泛的流程文档，那说明系统补进来的条件没有真正改变召回范围和排序逻辑。

**这一步的输出是什么：** 系统会得到一个非常重要的判断：

- 现在已经够具体，可以进入正式检索
- 还是不够，需要继续补条件

这一步就是 rewrite 的“停表点”。

---

## 十、第八步：输出最终的 Rewrite 结果

**这一步是什么：** 把整个过程产出的结构化结果交给下游检索链路。

**为什么要做：** 因为下游不是人类读者，而是检索器、排序器、查证模块和回答模块。它们真正需要的，不是一句“更顺口的话”，而是一份可执行的查询包。

**怎么做：** 对这个例子，一个更像工程产物的最终 rewrite，通常会长这样：

- 主查询：排查某客户在中国区开户持续失败的原因
- 硬约束：地区 = 中国区；时间 = 最近两周
- 软提示：v3 身份校验链路、实名校验失败、错误码 1021
- 子查询一：最近版本发布后是否新增中国区身份校验限制
- 子查询二：错误码 1021 是否集中出现在个人客户开户流程
- 子查询三：历史工单中是否已有同类修复记录
- 数据源建议：优先查工单和变更记录，再补查流程文档

最好再带两类元信息：

- 条件来源：来自用户原话、模板默认，还是侦察检索
- 条件置信度：它是硬事实，还是待验证候选

**这一步的输出是什么：** 这时的最终输出，已经不是一句改写后的文本，而是一份真正可执行的 Query 包。

把它和原问题对比一下，差别就很明显了：

- 原问题：为什么这个客户一直开户失败
- Rewrite 输出：围绕中国区、最近两周、错误码 1021、v3 身份校验链路，优先查工单和变更记录，验证这是不是地区规则或版本变更导致的持续开户失败

前者是人类说法。  
后者是机器能干活的说法。

---

## 十一、把整条链路再压缩成一句程序员能记住的话

如果把刚才那条主线压缩一下，整个 Query Rewrite 实际上就是这八步：

- 拆骨架
- 对模板
- 排缺口优先级
- 列候选原因
- 做轻量侦察检索
- 把候选值分层写回
- 做 rewrite 前后回看
- 输出 Query 包

你会发现，它的核心从来不是“模型怎么把话说得更漂亮”，而是：

**一个模糊问题进入系统后，怎样被一步步压缩成一个带范围、带判别力、带执行计划的查询包。**

---

## 十二、如果你今天就准备动手做第一版

第一版最值得做的，不是一个全能改写器，而是这五样东西：

- 一张原因排查模板
- 一棵小型候选原因树
- 一轮轻量侦察检索
- 一套硬约束/软提示/子查询分层规则
- 一个 rewrite 前后结果对比器

只要这五样东西在，同一句“为什么这个客户一直开户失败”进来，你的系统就已经不再是在“换个说法”，而是在稳定地产生一个更适合检索的输入。

---

## 结语：为什么这条主线比“改写句子”更重要

很多人第一次接触 query rewrite 时，会以为它的核心任务是把一句模糊的话改得更专业。

但从这个例子里你应该能看出来，真正值钱的不是“改写”两个字，而是整条收束链路：

- 先知道原问题哪里不够
- 再知道为了区分候选原因该补什么
- 再知道这些补充到底有没有带来收益

所以如果要把这篇文章压成一句话，我会这样说：

**Query Rewrite 的本质，不是把用户输入换一种写法，而是把它一步步变成一份机器真的可以执行的检索计划。**

---

## 附：把同一条主线放进 RAG 场景里，再走一遍

前面那版主线，更像一个通用检索系统里的 rewrite 过程。  
但你如果真在做 RAG，脑子里的问题通常会更具体：

- 我后面连的是向量库，不是一个抽象“检索系统”
- 我要召回的是 chunk，不是整份文档
- rewrite 的目标，不只是“更可执行”，还要“更容易打中正确 chunk”

所以同样的主线，放进 RAG 场景里，要重新看一遍。

这次我们还是不用抽象说法，直接用一个更贴近 RAG 的例子。

假设你在做一个企业内部知识库助手，后面接了这些内容：

- 产品帮助中心文档
- 各地区运营规则说明
- 发布说明
- 历史事故复盘文档

用户问：

> “为什么日本区可以下单，中国区一直提示风控校验失败？”

这句话和前面的“开户失败”很像，都是人类觉得足够、RAG 觉得不够的问题。  
因为对于向量检索来说，它的问题不是“能不能看懂”，而是：

- 该拿这整句话去做 embedding 吗
- 该把它拆成几个子查询吗
- 该优先召回规则文档、发布说明，还是事故复盘
- 哪些词应该成为强约束，哪些词只应该用来做扩展召回

这就是 RAG 场景下 query rewrite 的真正作用：**不是把问题改写给人看，而是把问题改写成更适合召回 chunk 的检索输入。**

---

## 一、RAG 场景里的原始问题，为什么天然不适合直接检索

先看原问题：

> “为什么日本区可以下单，中国区一直提示风控校验失败？”

如果你直接把这整句拿去做一次向量检索，系统很可能会发生三种常见失误：

- 召回一堆“下单流程总览”，因为“下单”这个词权重很高
- 召回一堆“风控是什么”的解释文档，因为“风控校验失败”被当成通用概念
- 漏掉真正关键的地区规则文档和发布说明，因为原问题里没有明确写“地区差异规则”“中国区风控开关”“最近版本变更”

这就是 RAG 和一般搜索很不一样的地方。

在 RAG 里，系统最终打中的不是“答案”，而是一批 chunk。  
而 chunk 的命中，通常高度依赖 query 里有没有出现：

- 正确的术语
- 正确的约束
- 正确的拆分方式

所以 RAG 场景下的 rewrite，目标不是把一句话润色成更自然，而是把一句话拆成更适合召回的检索面。

---

## 二、第一步：先判断这个问题适不适合“一次召回”

### 这一步是什么

先做一次查询分型，判断这个问题该走单轮 RAG，还是要走 rewrite + 多路召回。

### 为什么要做

因为不是每个 RAG 问题都值得进入复杂链路。

像“某个字段是什么意思”这种问题，一次召回通常就够了。  
但“日本区可以、中国区不行，为什么还提示风控校验失败”这种问题，天然就带着：

- 对比结构
- 地区约束
- 失败现象
- 原因追问

这类问题如果还走一次召回，往往会把几个信息面混成一团。

### 怎么做

第一版最务实的办法，是先做一个很轻的分型判断：

- 是否出现对比关系，比如“一个能，一个不能”
- 是否同时包含场景 + 现象 + 原因追问
- 是否明显需要跨两类以上文档求证

这个例子里，答案几乎是全中。

所以系统此时应该做的，不是立即检索，而是先标记：

- 这是一个“对比型原因排查问题”
- 不适合单 query 一次召回
- 需要进入 rewrite 流程

### 这一步的输出是什么

系统得到一个最关键的前置结论：

- 这不是“拿整句去搜”的题
- 这是“要拆开检索意图”的题

---

## 三、第二步：先把原问题拆成 RAG 真的关心的几个检索面

**这一步是什么：** 把用户的一句话，拆成多个对召回有意义的子面向。

**为什么要做：** 因为在 RAG 里，一句长问题往往同时混着好几种检索意图：

- 找规则
- 找异常
- 找差异
- 找变更

如果不拆，embedding 很容易把整句话平均掉，最后每个意图都打中一点，但没有一个打得够深。

**怎么做：** 对这个问题，系统至少应该拆出四个检索面：

- 地区对比面：日本区 vs 中国区
- 业务行为面：下单
- 异常现象面：风控校验失败
- 原因追问面：为什么会这样

拆完之后，系统才能继续问：这四个面里，哪些该留在主查询里，哪些要拆成子查询去召回。

**这一步的输出是什么：** 系统不再面对一句大白话，而是面对四个可管理的检索意图。

这对 RAG 非常重要，因为后面不管你做 dense retrieval、hybrid retrieval 还是 rerank，最怕的都是“一个 query 里混了太多意图”。

---

## 四、第三步：把用户语言翻译成知识库语言

**这一步是什么：** 把用户说法映射成知识库里真正会出现的术语。

**为什么要做：** RAG 最大的问题之一，是用户语言和知识库语言经常不是一套口径。

用户会说：

- 下单失败
- 风控卡住了
- 日本区可以，中国区不行

但知识库里真正写的可能是：

- 交易准入校验
- 风险规则引擎
- 中国大陆站点风控开关
- 区域化订单拦截策略

如果 rewrite 不做术语映射，向量召回很可能只能命中泛化 chunk，而打不中最关键的规则说明。

**怎么做：** 系统这一步通常会做两层映射：

- 同义映射：把“下单”映射到“订单提交”“交易准入”
- 领域映射：把“风控校验失败”映射到“风险规则引擎拦截”“风控开关”“区域订单限制”

这里要注意一个边界：

映射的目标不是替换掉用户原话，而是扩展检索口径。  
也就是说，系统不应该只保留“风险规则引擎”，把“风控校验失败”丢掉；更合理的是同时保留两套词。

**这一步的输出是什么：** 系统得到一组术语扩展：

- 用户词：下单、风控校验失败、日本区、中国区
- 知识库词：交易准入、风险规则引擎、区域风控开关、地区订单限制

这一步之后，rewrite 才真正开始具备“打中知识库语言”的能力。

---

## 五、第四步：不要只重写一句话，而要拆成三类 Query

**这一步是什么：** 把原问题拆成主查询、对比子查询、原因子查询。

**为什么要做：** 这是 RAG 场景和普通搜索最不一样的地方之一。

普通搜索可能只需要一个改写后的 query。  
但 RAG 往往更适合同时准备几条不同功能的 query，然后分别召回，再合并。

因为这个问题本质上不是单跳问题，而是：

- 先找日本区规则
- 再找中国区规则
- 再找风控失败相关说明
- 再找最近是否有变更

如果把这些都塞进一个 query，很容易谁都不够强。

**怎么做：** 对这个例子，一个很合理的拆法是：

- **主查询**：中国区下单风控校验失败的原因
- **对比查询**：日本区与中国区的下单风控规则差异
- **变更查询**：最近版本是否调整过中国区下单风控开关或地区限制

注意，这里不是为了“多搜几次而多搜几次”，而是因为这三条 query 对应的是三种完全不同的 chunk 类型：

- 主查询更容易打中异常说明和故障处理文档
- 对比查询更容易打中地区规则和配置差异文档
- 变更查询更容易打中发布说明和事故复盘

**这一步的输出是什么：** 系统产出的已经不只是一个 rewrite 句子，而是一个小型 query set。

对 RAG 来说，这比“把原句修得更漂亮”有用得多，因为它直接改变了召回路径。

---

## 六、第五步：给每条 Query 指定召回目标，而不是一锅端

**这一步是什么：** 为每条 rewrite 后的 query 绑定更合适的数据源和召回策略。

**为什么要做：** RAG 的真正问题，往往不是“搜不搜”，而是“往哪里搜”。

同一句 query，如果你拿它去统一搜全库，系统很可能召回很多噪声 chunk。  
而如果你先给 query 做路由，效果通常会稳定很多。

**怎么做：** 对这个例子，比较合理的路由是：

- 主查询优先去历史工单和异常处理文档
- 对比查询优先去地区规则文档和配置说明
- 变更查询优先去发布说明和事故复盘

这一步的工程意义非常直接：

- 不是所有 query 都进同一个向量库分片
- 不是所有 query 都用同一个 Top-K
- 不是所有 query 都值得用同样的 rerank 规则

也就是说，rewrite 在 RAG 里不仅改写 query 文本，还在悄悄决定检索路线。

**这一步的输出是什么：** 系统得到的是一组“query + route”：

- 这条 query 去哪里召回
- 召回的 chunk 大概应该长什么样
- 哪一类结果优先进入后面的 rerank

---

## 七、第六步：看召回回来的 chunk，再做一次二次 Rewrite

**这一步是什么：** 基于首轮召回结果，进一步收窄 query。

**为什么要做：** 这是 RAG 里最容易被忽略、但又最像真实系统的地方。

很多人理解 rewrite 时，只想到“检索前重写一次”。  
但复杂问题里，真正有价值的 rewrite，往往发生在**首轮召回之后**。

因为你看过第一批 chunk 后，才真正知道：

- 知识库里用的到底是哪套术语
- 哪个地区词最关键
- 哪个错误词反复出现
- 哪个版本变更最值得深挖

**怎么做：** 假设首轮召回后，最常出现的是这些信息：

- 中国区订单准入规则
- 风险规则 7B
- v5.2 版本新增地区风控开关
- 日本区未启用该规则

那系统第二轮 rewrite 就会比第一轮更收敛。

这时原来的主查询：

- 中国区下单风控校验失败的原因

就可以进一步收窄成：

- 中国区订单准入中风险规则 7B 触发的原因

而原来的对比查询：

- 日本区与中国区的下单风控规则差异

也可以进一步收窄成：

- 日本区是否未启用中国区 v5.2 新增的风险规则 7B

**这一步的输出是什么：** 第二轮 rewrite 不再只是“更长”，而是明显更贴近首轮召回出来的知识库语言。

这就是 RAG 场景下 query rewrite 的一个核心特征：

**rewrite 不是一次性的，而是被检索结果反向校正的。**

---

## 八、第七步：最终给 RAG 管线输出的，不是一句话，而是一套检索计划

**这一步是什么：** 把整个 rewrite 过程的结果，整理成后续 RAG 管线可直接消费的输入。

**为什么要做：** 因为 RAG 下游要做的不只是召回，还包括：

- rerank
- 去重
- chunk 组装
- 证据排序
- 最终回答

如果 rewrite 的输出只是一句自然语言，下游其实很难稳定利用它。

**怎么做：** 对这个例子，一个更像 RAG 工程产物的最终输出，通常会包含这些层次：

- 主查询：定位中国区下单风控校验失败的直接原因
- 对比查询：比较日本区与中国区的订单准入规则差异
- 变更查询：检查 v5.2 是否新增中国区地区风控开关
- 术语扩展：下单 / 订单准入 / 风险规则引擎 / 地区风控开关
- 路由信息：主查询查工单，对比查询查规则文档，变更查询查发布记录
- 首轮高置信候选：风险规则 7B、中国区启用、日本区未启用

如果再往 RAG 管线里走一步，这些内容还会影响：

- 哪些 chunk 先送去 rerank
- 哪些 chunk 应该按“地区差异”组织
- 哪些 chunk 应该作为“版本变更证据”保留

**这一步的输出是什么：** 最终输出已经不是“改写后的问题”，而是一套可以直接驱动 RAG 召回与证据组织的检索计划。

---

## 九、把 RAG 版主线压缩成一句最重要的话

在普通理解里，query rewrite 好像只是给搜索框润色一下输入。  
但在 RAG 场景里，它真正做的是三件更重的事：

- 把一个大问题拆成多个更容易命中 chunk 的 query
- 把用户语言翻译成知识库语言
- 把 query 和数据源、召回目标、二次收窄过程绑在一起

所以，如果要把这版 RAG 主线压成一句话，我会这样说：

**RAG 场景里的 Query Rewrite，本质上不是改写一句问题，而是把一句模糊问题拆成一套更容易召回正确 chunk 的检索计划。**
