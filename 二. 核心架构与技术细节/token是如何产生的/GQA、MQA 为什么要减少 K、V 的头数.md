# 为什么不是所有头都要各存一份 KV：GQA、MQA 为什么要减少 K、V 的头数

上一篇我们已经讲过，多头注意力的价值，在于让同一个位置能从多个低维视角并行读取上下文。可一旦把这个思路真正带进大模型推理，工程师很快就会发现一个麻烦：头数一多，Q、K、V 虽然都被拆开了，但真正越来越贵的，往往不是 Query，而是那些必须一路留在内存里的 Key 和 Value。

这个问题在长上下文和自回归生成里尤其明显。模型每生成一个新 Token，都会临时现算当前步的 Q，再去和历史里累积下来的 K、V 做匹配和读取。也就是说，Q 更像一次性的查询请求，而 K、V 更像必须持续留在现场的“历史档案”。上下文越长、层数越深、头数越多，这份历史档案就越重。

GQA 和 MQA 的核心动机，就是正视这个不对称现实：既然真正昂贵的是 K、V，不如减少 K、V 的头数，让多个 Query 头共享它们。这样做不是否定多头注意力，而是在保留多视角查询能力的同时，压低缓存和带宽成本。

## 1. 标准多头注意力里，为什么 K、V 会成为推理负担

在标准多头注意力里，Q、K、V 通常都会按相同头数展开。表面上这很整齐，但在推理阶段，它们的“生命期”并不对称。

1. **Q 通常是当前步临时现算的**：它像刚发出的提问，用完这一轮相关性计算就结束了。
2. **K、V 则要被留下来供后续所有位置继续访问**：它们会形成历史缓存，后面每生成一个 Token 都要继续拿出来参与匹配和读取。
3. **所以头数一多，真正持续膨胀的是 K、V 的存储和访存代价**：不是数学定义变了，而是推理时“谁要常驻内存”这件事，决定了成本会集中落在谁身上。

这很像开会做记录。Q 像当前发言者临时提出的问题，说完就过去了；K、V 像会议纪要，后面每个人都还要翻阅。问题不是“提问多不多”，而是“纪要要保存多久、被查多少次”。

## 2. 为什么长上下文会把这个问题放大

如果上下文不长，标准多头的成本有时还不至于特别刺眼。但一旦长度往上拉，K、V 的代价就会越来越难忽视。

1. **历史越长，要保存的 K、V 就越多**：因为每一个旧位置都得留下可被后续访问的痕迹。
2. **后续每一步生成都要重新读取它们**：这不只是存储问题，还会变成持续的带宽压力。
3. **所以瓶颈常常不是“算不出 Q”，而是“搬不动 K、V”**：尤其在现代推理系统里，很多时候真正拖慢速度的是内存访问，而不是纯计算。

这也是为什么近两年的长上下文讨论，越来越多地从“注意力公式”转向“KV Cache 怎么放、怎么读、怎么压”。当上下文足够长时，谁占缓存、谁消耗带宽，往往比纸面结构更决定工程体验。

## 3. MQA 到底改了什么：为什么多个 Query 头要共享同一组 K、V

MQA 的全称是 Multi-Query Attention。它最核心的动作很简单：不是每个 Query 头都配自己独立的一组 K、V，而是让很多 Query 头共享同一组 K、V。

1. **Query 仍然可以保持多头**：也就是说，当前位置依然可以从多个视角发起查询。
2. **但 Key 和 Value 不再按同样头数一一复制**：多个 Query 头会去对接同一份 K、V 表示。
3. **这样一来，缓存体积和读取流量就会明显下降**：因为历史里常驻的那部分不再随着 Query 头数等比例膨胀。

直觉上看，MQA 像一群记者各自从不同角度提问，但大家共用同一套档案管理员和同一份资料库。提问视角还是多样的，只是后台不再给每位记者各配一整套重复档案。

## 4. MQA 的收益为什么很直接，但代价也很真实

MQA 之所以有吸引力，是因为它打中的正好是长上下文推理里最痛的地方。但它不是白拿收益。

1. **收益很直接**：共享 K、V 以后，缓存更小，带宽压力更轻，推理时更容易跑得动。
2. **代价也很直接**：多个 Query 头面对的是同一份 K、V 后台，意味着可被匹配和读取的表示多样性会减少。
3. **所以它更像一种强压缩**：把后端资料系统做得更轻，但也可能牺牲一部分精细分工能力。

换句话说，MQA 不是“更聪明的标准多头”，而是“更便宜的多头近似”。在资源吃紧、上下文很长的场景里，这种交换常常有吸引力；但如果你特别在意表达细腻度，它就未必总是最稳的选择。

## 5. GQA 为什么常被看成更稳的折中

GQA 的全称是 Grouped-Query Attention。它没有像 MQA 那样把所有 Query 头都压到同一组 K、V 上，而是做了一个中间方案：把多个 Query 头分组，每一组共享一组 K、V。

1. **它保留了部分共享**：所以缓存和带宽压力能明显低于标准多头。
2. **它又没有共享到只剩一组 K、V**：因此不同 Query 组之间，仍然保有一定的后台表示差异。
3. **这让它经常成为效果和成本之间的折中点**：比标准多头更省资源，比 MQA 更保守，也更容易让人接受。

你可以把它理解成图书馆分区。不是每位读者都配一个独立馆员，那样太贵；也不是全馆只剩一个总馆员，那样太挤。更现实的办法，是一组读者共用一个分区馆员。这样既节省人力，又不至于所有需求都堵在同一个入口。

## 6. 一个常见误区：减少 K、V 头数，不等于放弃多头查询

很多人第一次听到 GQA、MQA，会以为“那不就是把多头注意力砍掉了吗？”这其实不太准确。

1. **多头查询能力通常还在**：尤其在 GQA 里，多个 Query 头依然存在，它们还是会从不同视角发问。
2. **被压缩的是后台可共享部分**：也就是历史里的 K、V 不再按最细粒度为每个 Query 头单独准备。
3. **所以变化的重点不是‘有没有多头’，而是‘多头的后端资源如何组织’**：这是一个工程成本重构问题，不是简单的功能开关。

这个纠偏很重要。否则很容易把 GQA、MQA 理解成“把模型做弱了”，而看不到它们真正重写的是缓存结构和访存成本。

## 7. 为什么这类设计在最近两年更受重视

GQA、MQA 并不是凭空突然冒出来的概念，它们之所以在最近两年被更频繁地讨论，和大模型的使用场景变化直接相关。

1. **上下文越来越长**：对话、多轮 Agent、代码仓库理解、长文档问答，都在把序列长度往上推。
2. **推理系统越来越在乎吞吐和成本**：很多场景的核心问题已经不是“能不能跑”，而是“同样资源下能服务多少请求”。
3. **注意力优化越来越围绕 KV Cache 展开**：Flash 解法、分页缓存、状态压缩、共享 K/V，本质上都在回答一个问题：历史信息到底该以多高成本留在系统里。

所以，GQA 和 MQA 的流行，不只是注意力理论的一个小修小补，而是长上下文时代把“KV 太贵”这件事彻底推到了台前。

## 8. 现实里的边界：它们不是唯一答案

虽然 GQA、MQA 很重要，但也不能把它们讲成“注意力优化的终局”。

1. **它们解决的是 K、V 头数带来的缓存与带宽问题**：并没有一口气解决所有长上下文难题。
2. **不同模型会根据目标做不同取舍**：有的更偏省显存，有的更偏稳效果，有的会继续结合别的缓存或压缩方案。
3. **近两年的一些更激进路线还会进一步改写注意力状态表示**：比如不再满足于“共享多少 K、V”，而是继续追问“历史信息能不能被压缩成更紧凑的形式”。

也就是说，GQA、MQA 更像是主流工程世界里一组非常实用的中间答案，而不是所有问题的最后答案。

这一篇放在 11.2 和 11.4 之间，逻辑上其实很自然。

1. **11.2 刚讲完为什么要多头**：我们已经理解了“多视角读取”为什么有价值。

## 10. 一句话复盘

GQA、MQA 之所以要减少 K、V 的头数，是因为在长上下文自回归推理里，真正持续占用缓存和带宽的不是临时现算的 Query，而是必须长期保留下来的 Key 和 Value；MQA 选择让更多 Query 头共享同一组 K、V，以更激进的方式换取更低成本，GQA 则用分组共享做出更稳的折中，它们共同体现的是同一个工程判断：多头查询能力很宝贵，但不一定值得为每个头都长期背一整套独立的历史档案。
