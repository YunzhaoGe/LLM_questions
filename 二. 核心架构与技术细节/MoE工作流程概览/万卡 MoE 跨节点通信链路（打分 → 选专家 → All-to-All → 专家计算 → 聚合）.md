# 万卡 MoE 跨节点通信链路（一个 token → 选专家 → 发出去 → 算完 → 收回来）

把“一个 token 在 MoE 层里如何跨节点去专家、再回到原位置”讲清楚：gate 打分 → Top-k 选专家 → 把路由变成 All-to-All 的发送计划 → 分发到专家所在 GPU/节点 → 专家前向 → 回收结果 → 加权聚合并写回原 token 顺序。

---

## 1. MoE 层入口：明确“这个 token 从哪来、最终要回哪去”

1. 输入到达：上一子层输出的隐藏状态进入 MoE 层（为了开始“按 token 分流到不同专家”）
2. 标记来源：为每个 token 记录其原始位置与来源设备/节点（rank）（为了让计算完成后能准确回收并放回原位）
3. 形成 token 视图：将 batch/序列视图展平成 token 列表并保留还原索引（为了让路由、打包、通信都能按 token 粒度执行）

---

## 2. Gate 打分：给每个 token 评估“该去哪个专家”

1. 门控投影：将每个 token 的隐藏向量送入门控网络，得到对所有专家的打分（为了量化“哪个专家最合适”）
2. 归一化处理：对打分做 softmax 或等价变换，得到可解释的路由权重（为了把打分变成可用于加权的系数）
3. Top-k 选择：为每个 token 选出 k 个专家及其权重（为了把稠密选择变成稀疏分派以节省算力）

---

## 3. 路由表：把“token → 专家”的关系变成可执行计划

1. 生成路由项：为每个 token 产出 k 条路由记录（token_id、expert_id、weight 等）（为了把抽象选择变成可调度单位）
2. 映射到设备：将 expert_id 映射到其所在设备/节点，并把路由项按目标 rank 分桶（为了把“选专家”变成“发给谁”）
3. 生成发送计划：为每个目标 rank 计算 counts 与 offsets，并确定打包顺序（为了让 All-to-All 一次性把 token 发到正确位置）

---

## 4. All-to-All 分发：跨 GPU/节点把 token 送到专家所在处

1. 打包发送缓冲：将要发送的 token 向量按目标设备顺序拷入发送缓冲（为了让通信尽量连续、少碎片）
2. 执行集体通信：通过 NCCL 等通信库触发 All-to-All（或等价的多次点对点）交换（为了把 token 送到“专家本地”）
3. 到达与落位：接收端按预先计算的 offsets 写入接收缓冲并得到本地 token 列表（为了让专家计算能直接读取）

---

## 5. 专家计算：在本地做 FFN（或更复杂专家）的前向

1. 按专家切片：把接收来的 token 再按 expert_id 切成若干小批（为了让每个专家做自己的矩阵乘）
2. 执行前向：每个专家对分到的 token 做上投影/激活/下投影（或等价结构）（为了得到专家对该 token 的变换结果）
3. 产出结果：得到与输入 token 一一对应的专家输出向量（为了进入回收与聚合阶段）

---

## 6. All-to-All 回收：把专家输出送回 token 的原始所在设备

1. 打包回传：将专家输出按“原始 token 所属设备/节点”重新分组并拷入发送缓冲（为了让回收通信同样连续）
2. 执行回收通信：再次触发 All-to-All，把结果送回最初持有该 token 的设备（为了让后续层继续在原 batch 上执行）
3. 还原顺序：按路由表记录的 token_id 与偏移，把回收结果还原到原 token 顺序（为了保证与残差/后续操作对齐）

---

## 7. 聚合与残差：把 k 个专家的结果合成一个输出

1. 加权求和：按 gate 给出的权重对 k 个专家输出做加权合并（为了得到该 token 的最终 MoE 输出）
2. 合并回 batch：将 token 视图的输出写回原 batch/序列张量形状（为了与 Transformer 其余部分接口一致）
3. 残差连接：将 MoE 输出与输入残差相加（并可选做 dropout 等）（为了维持深层网络的稳定训练/推理）

---

## 8. 容量与拥塞：保证“发得出去、算得回来”

1. 容量裁剪：为每个专家设置容量并在打包阶段裁剪/回退溢出 token（为了避免某些专家队列爆炸导致尾部延迟失控）
2. 负载平衡：通过路由噪声、辅助损失或调度策略让 token 更均匀分布（为了减少 All-to-All 的热点与拥塞）
3. 通信重叠：把打包/通信/专家计算做流水化与重叠（为了让跨节点传输不吞噬 MoE 的省算力收益）
