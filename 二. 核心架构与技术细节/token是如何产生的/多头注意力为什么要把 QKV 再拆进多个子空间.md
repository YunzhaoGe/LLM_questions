# 为什么不是一次“大而全”的注意力：多头注意力为什么要把 QKV 再拆进多个子空间

前面两篇我们已经顺着主干流程走到了这里：隐藏状态先被投影成 Q、K、V，再交给注意力机制去做“查询—匹配—读取”。很多初学者在这一步会自然冒出一个问题：既然已经有了 Q、K、V，为什么还要再把它们继续拆成多个 head？直接做一次足够大的注意力，不是更简单吗？

表面上看，确实可以这么想。可一旦你把注意力理解成“当前位置拿着自己的问题，去上下文里找信息”，就会发现一个现实难点：同一个位置通常不只需要一种信息。它可能既要看主谓关系，也要看长距离指代，还要留意语气转折、时间条件、结构边界。如果所有关系都塞进同一套检索空间里，模型就很容易把不同任务搅在一起。

多头注意力的核心直觉，就是不要逼一套 QKV 同时用一种方式解释所有关系，而是把它拆成多个较小的子空间，让不同头各自形成不同观察角度。这样做不是为了把结构弄复杂，而是为了让“信息读取”这件事更有分工，也更不容易互相打架。

## 1. 为什么不能只做一次大注意力

如果只做一次单头的大注意力，理论上模型也能读上下文，但它会把所有关系都挤进同一个表示空间里。

1. **不同关系会争抢同一套表达通道**：句法关系、语义关联、长程依赖、局部搭配，都会混在同一套打分逻辑里。
2. **当前位置的需求往往是复合的**：同一个 Token 既可能想找“谁在修饰我”，也可能想找“我指代的是谁”，还可能想找“前面哪一段限制了我”。
3. **单一大头很难同时把这些角度都表达得足够清楚**：不是说完全做不到，而是不同关系更容易互相干扰，最后谁都学得不够稳定。

这就像一个记者同时负责采访现场、记录时间线、分析人物关系、整理背景资料。不是完全不能做，而是信息会互相挤占注意力，最终每个维度都容易变得粗糙。

## 2. 拆成多个头，本质上是在并行开多个观察窗口

多头注意力最重要的，不是“头变多了”，而是“观察视角被拆开了”。

1. **每个头都会拿到自己那一份 Q、K、V 子空间**：它不是在整块 hidden size 上无边界地看，而是在一个更小、更专注的低维空间里做匹配。
2. **低维空间更容易形成明确偏好**：一个头可以偏向局部搭配，另一个头可以偏向远距离关系，还有的头更容易学出结构边界或位置敏感模式。
3. **多个头并行运行，相当于同一位置同时发起多次不同角度的检索**：不是“多次重复同一个动作”，而是“用不同镜头看同一段上下文”。

你可以把它想成一支转播团队：同一场比赛不会只放一个固定镜头，而是有近景镜头、全景镜头、跟拍镜头、回放镜头。比赛还是那场比赛，但不同镜头能抓住完全不同的信息。

## 3. 为什么每个头都只在低维子空间里工作

很多人会以为，拆头只是把一整块向量随便切几刀。其实这里有一个很关键的设计直觉：每个头不应该什么都看一点，而应该在更小空间里形成更集中的表达能力。

1. **低维子空间会迫使每个头做选择**：它没有无限空间去兼顾所有模式，所以更容易形成某种稳定关注偏好。
2. **这种限制反而能带来分工**：因为不同头在不同参数初始化和训练反馈下，会逐渐发展出不完全相同的职责。
3. **多个小头拼起来，往往比一个超大头更灵活**：因为它允许模型用多个“专长视角”共同描述上下文，而不是期待一个单体结构包办一切。

这有点像一个项目组。不是把所有问题都扔给一个全能型成员，而是让几个人分别盯需求、架构、测试、上线风险。单个人可能也能勉强做完，但多人分工通常更稳，也更细。

## 4. 不同头通常会学出哪些差异

这里要先说清一个边界：不是每个模型、每一层、每一个头都会整齐地长成固定功能清单。很多时候我们说的是“常见倾向”，而不是绝对规则。

1. **有些头偏局部搭配**：它们更容易关注相邻词之间的组合关系，比如短语内部的紧密依赖。
2. **有些头偏长距离连接**：它们更适合把很远的位置拉进当前读取过程，比如跨句指代或远距离条件约束。
3. **有些头偏结构边界**：它们会对分隔符、句末、段落边界、特殊标记更敏感。
4. **有些头偏位置或顺序模式**：特别是在加入位置编码之后，不同头对相对距离的敏感方式可能不同。

所以，多头注意力不是简单复制 N 份同样的注意力，而更像让同一个位置同时问出 N 个略有不同的问题。

## 5. 一个常见误区：多头不等于“每个头都必须有明确职业”

很多科普文章为了好懂，喜欢把每个头都说成一个职位，比如“这个头负责语法，那个头负责指代，另一个头负责时间”。这种说法有启发性，但不能讲得太死。

1. **真实情况往往更混合**：一个头可能同时掺着局部语法、位置偏好和语义线索，不一定能被贴成单一标签。
2. **头的功能也会随层数变化**：浅层和深层的头，关注模式常常不同，不是所有层都在做同一种事。
3. **还有些头可能没有那么有用**：实际模型里常见功能重叠、弱贡献甚至近乎失效的头。

所以，理解多头最稳妥的方法，不是给每个头硬贴工牌，而是先接受一个更真实的事实：它提供的是“多视角并行读取能力”，至于每个视角最后长成什么样，要看训练和模型规模。

## 6. 为什么会出现分工、冗余和退化同时存在

多头机制最有意思的地方，是它并不会保证“头越多越完美分工”。现实里经常是三种现象一起出现。

1. **分工会出现**：一部分头确实会表现出比较稳定的关注倾向，这是多头设计最想要的结果。
2. **冗余也会出现**：两个头可能学出相近模式，因为训练并不会强制所有头都完全不同。
3. **退化同样会出现**：有些头贡献很弱，甚至长期接近“可有可无”的状态。

这并不说明多头机制失败了。更准确地说，它像一个带冗余的工程系统：不是每个部件都满功率发光，但整体上这种并行冗余能提高鲁棒性，也给模型留下更多自组织空间。

## 7. 现实里的取舍：头数不是越多越好

一听到“多视角并行”，很多人会直觉觉得头越多越强。这个结论并不稳。

1. **头更多，不代表每个头都足够有信息量**：如果每个头的维度被切得太小，单头表达能力反而会变弱。
2. **头更多，也会带来更高的实现复杂度**：虽然这里先不展开后续拼接和回并，但只从拆分本身看，更多头意味着更多子空间组织和更复杂的并行管理。
3. **工程实践会在头数、单头维度、缓存成本之间做平衡**：这也是为什么后面会出现 GQA、MQA 这类变体，不是因为多头思路错了，而是因为标准多头在长上下文推理里开始显出成本压力。

换句话说，多头注意力不是“无限加头”的游戏，而是“用多少个观察窗口最划算”的设计问题。

## 8. 近两年的变化，为什么大家又重新审视“头”这件事

多头注意力提出已经很多年了，但在最近两年，它又变成了工程讨论热点，原因主要来自长上下文和推理成本。

1. **长上下文让头的代价更可见**：头数一多，后续与 K/V 相关的缓存、访存和带宽压力就更容易成为瓶颈。
2. **很多新变体开始重新分配“谁需要多头，谁可以共享”**：这正是 GQA、MQA 这类设计会流行的背景，它们不是否定多视角，而是在重写多视角的成本结构。
3. **可解释性研究也在反复观察头的行为**：过去大家更关注“注意力能不能工作”，现在更关心“哪些头真的有用、哪些头只是冗余、哪些层最值得保留完整多头结构”。

所以，“头”不只是一个教材名词，它已经变成结构表达能力、推理效率和可解释性三者交汇的关键接口。

## 9. 和第 14 章的边界放在哪里

这一篇讲到这里就够了，因为它的任务是回答“为什么要拆头”，不是回答“拆完之后怎么收回来”。

1. **这一章只讲拆分动机和多视角意义**：为什么一个位置需要多个观察窗口，为什么不同头会形成不同关注模式。
2. **这一章不展开回并过程**：多个头各自读出的结果如何拼接、如何再投影回主通道，这些都留给后面的章节。
3. **这样主线才不会缠在一起**：先理解“为什么分”，再理解“怎么合”，逻辑上更顺，也不容易重复。

这就像先理解“为什么要设多个小组”，再去理解“小组产出的内容怎么汇总成最终决策”。两个问题看起来挨着，但并不是同一个问题。

## 10. 一句话复盘

多头注意力之所以要把 Q、K、V 再拆进多个子空间，不是为了把一套注意力复制很多遍，而是为了让同一个位置能够同时从多个低维视角并行读取上下文；这样，模型更容易在不同头里发展出不同的关注模式，哪怕其中会同时伴随分工、冗余和退化，这种多视角结构依然比“一次大而全的单头读取”更灵活，也更接近真实语言关系的复杂性。
