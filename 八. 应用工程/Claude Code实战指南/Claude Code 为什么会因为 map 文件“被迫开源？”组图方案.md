### 第 2 张：source map 到底是什么

**标题建议：**

先搞清楚：source map 不是源码，它是“对照表”

**这一张承载的信息：**

- source map 本质是一个 JSON 文件
- 作用是把“生成后的代码”映射回“原始源码”
- 为什么需要它：
  - TypeScript 会编译成 JavaScript
  - bundler 会合并模块
  - 代码会被压缩、重命名、tree-shaking
  - JSX/TSX/Sass 会被转换
- 它服务的对象是调试器，帮助恢复报错位置、断点位置和源码视角

**适合直接上图的表达：**

- 线上代码 = 译文
- source map = 译后索引
- 调试器 = 通过索引回看原文的人

**画面结构建议：**

- 左侧放“原始源码 → 构建转换 → 线上产物”
- 右侧放“source map 把两边连起来”

### 第 3 张：一个 map 文件里到底有什么

**标题建议：**

source map 里最关键的 5 个字段

**这一张承载的信息：**

- `sources`：原始源码文件路径列表
- `mappings`：生成代码位置到源码位置的映射关系
- `names`：变量名、函数名等标识符表
- `sourceRoot`：原始路径前缀
- `sourcesContent`：最敏感，可能直接内联原始源码全文

**这一张的重点结论：**

真正危险的通常不是 map 文件本身，而是它有没有把 `sourcesContent` 一起发出去。

**适合做成视觉重点的句子：**

当 `sourcesContent` 存在时，map 文件可能已经不是“索引”，而是“带全文附件的索引库”。

**画面结构建议：**

- 采用卡片式结构
- 前 4 个字段简写说明
- 最后一块 `sourcesContent` 用高亮色单独强调

### 第 4 张：为什么 map 文件能“还原源码”

**标题建议：**

源码是怎么被“还原”出来的？

**这一张承载的信息：**

- 情况 A：map 里带了 `sourcesContent`
  - 直接读路径
  - 读源码正文
  - 按索引配对
  - 批量写回目录
- 这时候严格说不算高难度逆向，更像“解包”
- 一个很重要的类比：
  - 你以为发的是索引
  - 实际上传的是带正文的压缩包目录

**这一张的重点结论：**

如果 `sourcesContent` 已经内联了原始文件全文，那么源码很多时候不是被“推导”出来的，而是本来就在包里。

**画面结构建议：**

- 用 4 步箭头流程图
- 底部加一句“不是破解，而是解包”

### 第 5 张：不是所有 map 都等于源码泄露

**标题建议：**

一个常见误区：有 map，不等于 100% 泄露源码

**这一张承载的信息：**

- 高风险：带 `sourcesContent` 的 source map
- 中风险：不带正文，但暴露路径、模块结构、标识符、映射信息
- 较复杂情况：正文不在 map 中，但能从别处拿到原始文件
- 判断风险大小，要看这几个条件是否同时成立：
  - map 是否可公开访问
  - 是否有 `sourcesContent`
  - `sources` 是否暴露完整目录结构
  - 是否覆盖核心业务代码

**这一张的重点结论：**

风险判断不是只看“有没有 `.map`”，而是看“map 里装了什么、外界能拿到什么”。

**画面结构建议：**

- 适合做成“三档风险”对比图
- 右下角放 4 个判断条件

### 第 6 张：为什么很多工程会把源码内联进 map

**标题建议：**

为什么团队会把源码一起打进 map？

**这一张承载的信息：**

- 因为这样调试最方便
- 调试器拿到 map 后几乎可以直接工作
- 不需要继续到处找原始源码路径
- 开发体验提升，但生产环境风险大幅上升

**这一张的核心冲突：**

- 对开发者：越完整越方便
- 对生产发布：越完整暴露越多

**适合做成图上的一句话：**

很多泄露事故不是因为没人懂 source map，而是把开发阶段的便利配置，原样带到了生产发布。

**画面结构建议：**

- 左右对比式排版
- 左边“开发调试收益”
- 右边“生产暴露成本”

### 第 7 张：Claude Code 事件为什么会引爆讨论

**标题建议：**

Claude Code 这次真正刺眼的点是什么？

**这一张承载的信息：**

- 争议核心不是“高难度破解”
- 而是公开包里被发现带了体积很大的 `cli.js.map`
- 如果其中内联了大量原始源码内容，那么外界几乎不需要反编译
- 就可以还原目录结构和大段源码文本

**这一张的重点结论：**

真正让人警惕的，不是攻击者技术太高，而是源码暴露这件事变成了一个低门槛、机械化过程。

**适合直接上图的总结句：**

这不是按开源协议开源，但从“可见性”上，已经接近公开浏览源码。

**画面结构建议：**

- 上方写“公众误解：被破解了”
- 下方写“实际本质：把带源码内容的 map 发到了公开渠道”

### 第 8 张：如果做安全排查，应该怎么查

**标题建议：**

想自查风险，按这 4 步看

**这一张承载的信息：**

- 第一步：确认产物里有没有 `.map`
  - 看静态资源
  - 看 `sourceMappingURL`
  - 看 HTTP 头
  - 看 npm 包/安装包/镜像
- 第二步：判断它只是映射，还是“源码容器”
  - 看有没有 `sourcesContent`
  - 看路径是否完整
  - 看体积是否异常大
- 第三步：评估可恢复程度
  - 只能看结构
  - 能恢复大量模块
  - 能按目录重建主要源码树
- 第四步：确认是不是生产级暴露
  - 测试环境还是正式分发渠道
  - 单个 bundle 还是全量核心产物
  - 短时窗口还是长期可获取

**这一张的重点结论：**

排查的关键不是“有没有 map”，而是“能恢复到什么程度、暴露到什么范围”。

**画面结构建议：**

- 用 4 步流程图最合适
- 每一步配 2 到 4 个关键词

### 第 9 张：防守建议 + 总结收束

**标题建议：**

怎么避免类似事故再次发生？

**这一张承载的信息：**

- 区分“生成 source map”和“公开 source map”
- 开发、CI、生产环境不要使用同一套发布策略
- 即使保留 map，也尽量避免把源码全文内联进去
- 把 source map 检查加入发版流程：
  - 是否包含 `.map`
  - 是否存在 `sourcesContent`
  - 体积是否异常
  - 是否暴露内部路径和私有模块名
- 不要只盯 CDN，也要检查 npm 包、桌面应用包、扩展包、混合资源包

**适合做收束金句：**

source map 不是原罪，问题在于你是否知道它包含什么、谁能访问它。

**最后一句建议直接用于尾图：**

source map 本来是开发者的“维修手册”，一旦把带全文的版本发到公开渠道，它就可能变成任何人都能翻阅的“施工图纸”。

**画面结构建议：**

- 上半部分放 4 条防守动作
- 下半部分放总结金句，作为整组图收尾

- 页眉统一：左上角放系列名，右上角放页码，例如“01/09”
- 标题统一：主标题都控制在 1 到 2 行，避免忽长忽短
- 信息块统一：所有卡片采用一致圆角、边框和间距
- 强调色统一：普通信息一个主色，风险信息一个警示色
- 金句统一：每页底部都保留一条结论句，形成节奏感
- 留白统一：上下边距尽量固定，保证整组图观感整洁
