# LLM 推理流程（Prompt → 第一个 Token）

目标：从用户输入 prompt 开始，串起分词与编号、向量化与位置处理、Transformer 的注意力与前馈计算（含因果 mask、残差与 KV cache），直到输出 logits 并用解码策略选出第一个 token。每一步都同时回答“做了什么/为什么这么做”，并配一句“就像……”的类比，帮助建立直观的全局理解。

---

## 1. 输入与预处理

1. Tokenization：把 prompt（在聊天模型里通常先按 chat template 拼成一段带特殊 token 的文本）切分为 token 序列 `tokens = [t0, t1, ..., t(n-1)]`（为了把连续字符串转换成可计算的离散符号；现代 LLM 常用 byte-level BPE 以兼容任意字符与代码；就像先把对话按统一格式排版，再把一段话拆成一块块可拼装的“乐高零件”）
2. Vocabulary lookup：把 token 映射为 `token_ids = [id0, id1, ..., id(n-1)]`（为了把符号映射成整数索引，便于 embedding table 以 O(1) 查表；就像把“单词”换成字典里的页码/编号，方便快速定位）
3. Padding/Batching（可选）：如果是 batch 推理，会对齐长度并生成 `attention_mask`（为了并行算多个样本；同时防止 padding 位置参与注意力与损坏分布；就像把不同长度的句子装进同样大小的盒子，但用隔板标记“这些空位别当内容”）
4. Truncation/Windowing（常见）：超过上下文长度时做截断、滑窗或保留关键片段（为了把输入控制在模型可处理的长度内，并尽量保留最有用的信息；就像行李超重时先保留证件和必需品）

---

## 2. Embedding 与位置信息

1. Token embedding：查表得到 `X = Embedding(token_ids)`，形状通常是 `[n, d_model]`（batch 情况为 `[b, n, d_model]`）（为了把离散 id 映射到连续向量空间，让后续注意力/MLP 能做连续变换与相似性计算；就像把“编号”变成一串坐标，让相近含义在地图上离得更近）
2. 位置处理（取决于模型；为了让模型区分“同一个 token 出现在不同位置”；就像同一句话里“第 1 段”和“第 10 段”含义会不同）：
   - 绝对/可学习位置嵌入：`X = X + PosEmbed(positions)`（为了把位置信息直接注入到每个 token 的表示里；就像给每个词贴上“第几号座位”的标签）
   - RoPE/ALiBi：一般不改 `X`，而是在注意力内部对 Q/K（或注意力分数）施加位置相关变换/偏置（为了让注意力对相对距离更敏感，并能更好外推到更长上下文；就像更关心“离我多远”，而不是“绝对坐标是多少”）
   - 长上下文扩展（常见）：对 RoPE 做 scaling（如 NTK/YaRN 思路）或改用更强的位置策略（为了在更长输入下保持稳定与可用性；就像把地图比例尺调大，仍能看清远处的路标）

---

## 3. Transformer 前向：Attention 子层（计算 Q/K/V 与相关性打分，prefill）

这一步的产物是：对 prompt 每个位置的隐藏状态 + 为后续解码准备的 KV cache。

对每一层 `l = 1..L`：

1. Norm（常见为 Pre-Norm 的 RMSNorm/LayerNorm）：`H = Norm(X)`（为了稳定数值尺度与训练/推理的激活分布，降低深层网络的数值风险；就像把音量旋钮拉到合理范围，避免忽大忽小）
2. 线性投影得到 Q/K/V（为了把同一个隐状态映射到“查询/键/值”三个子空间，分别承担匹配与聚合的角色；工程上常做 fused QKV projection 提升吞吐；就像“我在找什么(Q) / 目录索引(K) / 具体内容(V)”三件事分开做，但在流水线上一次性加工完）：
   - `Q = H Wq`
   - `K = H Wk`
   - `V = H Wv`
3. 多头拆分：把 Q/K/V reshape 为 head 维度（为了让不同 head 学到不同“关系子空间”，并行建模多种依赖；现代推理常用 MQA/GQA 让多组 Q 共享更少的 K/V 头以省显存与带宽；就像同一段话同时从“语法”“指代”“主题”多个角度去读，但目录索引可以多人共用）：
   - 典型形状：`Q: [n, n_q, d_head]`, `K/V: [n, n_kv, d_head]`（GQA/MQA 下 `n_kv <= n_q`）
4. 位置处理（若使用 RoPE）：对每个 head 的 Q/K 应用 RoPE（为了把位置信息以相对关系的形式注入注意力匹配；就像把“相对距离”编码进匹配规则里）

---
## 4. Transformer 前向：Attention 子层（mask/softmax/聚合与输出，prefill）

对每一层 `l = 1..L`（承接上半部分第 1-4 步）：

1. 因果约束（Causal mask；为了保证自回归：当前位置不能使用未来信息；就像读书时把后面的答案页遮住）：
   - 注意力分数：`S = (Q K^T) / sqrt(d_head)`
   - 在 `S` 上施加因果 mask（并与 padding mask 结合，如果有 batch）（为了屏蔽不允许 attend 的位置；就像把“不该看的位置”直接打上禁止标记）
2. Softmax + 聚合：`A = softmax(S)`，`O = A V`（为了把注意力分数归一化成权重分布并完成加权求和；就像把“关注度”变成一组加起来等于 1 的投票权重，再按票数汇总成结论）
3. 高效注意力实现（工程常见）：使用 FlashAttention/SDPA 这类 fused kernel，避免显式 materialize `S` 与 `A` 的完整矩阵（为了显著降低显存与提升吞吐，TTFT/吞吐都受益；就像不把整张表格打印出来再算，而是在流水线上边算边汇总）
4. 输出投影：`O = O Wo`（为了把多头输出映射回模型维度，继续走残差主干；就像把多个小组的结论汇总成同一种格式的报告）
5. 残差：`X = X + O`（为了保留原信息通道，并让深层网络更易优化/更稳；就像在改稿时保留原稿主线，再把增补内容叠加上去）

---

## 5. Transformer 前向：MLP/FFN 子层 + KV Cache（prefill）

承接上半部分（Attention 子层）之后，同一层还会继续做：

1. MLP/FFN 分支（实现常见 SwiGLU/GEGLU + 两线性，也可能是 GELU 的两层 FFN；为了提供逐 token 的非线性变换与通道混合能力，补足注意力的表达；就像对每个位置做一次“本地加工/润色”）：
   - `M = MLP(Norm(X))`
   - `X = X + M`
2. KV cache 写入（为后续解码用）：把本层的 `K,V`（按 head）缓存起来（为了让后续每生成一个 token 时，不必重复计算整段 prompt 的 K/V；单步解码的注意力从“重算整段”变为“只算新 token 对历史的注意力”（近似 O(T)）；就像先把材料做成可检索的笔记，后面只查索引不从头通读）

经过 L 层后，得到最后一层输出隐藏状态 `X_L`，其中我们关心最后一个位置 `x_last = X_L[n-1]`（代表“读完整段 prompt 后的当前状态”）。

---

## 6. 得到 logits（只用最后一个位置）

1. Final norm（取决于模型结构）：`h = FinalNorm(x_last)`（为了把最终表征对齐到输出头期望的数值尺度；就像出门前把指南针校准一下）
2. LM head：`logits = h W_vocab`（形状 `[vocab_size]`，是否有 bias 取决于实现）（为了得到对词表中每个 token 的打分；就像把“当前想法”翻译成一张“候选词得分榜”）

`logits` 就是“第一个输出 token 的未归一化分数”。

---

## 7. 解码决策（产出第一个 token）

1. Logits 处理（可选；为了把“基础模型的偏好”对齐到产品目标与可控性要求；就像在出答案前先调一调“说话风格/谨慎程度”的旋钮）：
   - Temperature：`logits = logits / T`（为了控制随机性：T 越低越确定，T 越高越发散；就像控制“敢不敢跳出常规”的胆子大小）
   - Logits bias、重复惩罚、bad words mask 等（为了做安全/风格/重复控制；就像给某些词加减分，或直接拉黑不让选）
   - 结构化/约束解码（常见）：grammar/JSON schema 等约束候选集合（为了让输出满足格式与安全边界；就像填表只能在规定的格子里写字）
2. 候选截断（可选；为了降低长尾噪声、提升可读性/一致性；就像先列一个“候选短名单”再做决定）：Top-k / Top-p / Min-p / Typical 等
3. 采样或贪心（为了从分布里选出“第一个输出 token”；就像“掷骰子选一个”或“永远选最高分”）：
   - Sampling：从 `softmax(logits)` 得到的分布采样（为了多样性；就像按概率抽签）
   - Greedy：`argmax(logits)`（为了确定性与可复现；就像永远点“推荐第一名”）
4. 得到 `next_token_id`，并可 detokenize 为首个输出片段（字符/子词）（为了把离散 token 还原成用户可见文本；就像把乐高编号重新拼回可读的文字）

到这里，“从 prompt 到第一个 token”就完成了。后续生成会进入 decode 循环：每一步只输入“新产生的 token”，复用 KV cache，直到遇到 stop token 或达到长度上限。

---

## 8. 工程级推理优化（不改变语义，但决定真实速度）

1. Continuous batching：动态把不同请求的 prefill/decode 拼到同一个 batch（为了提高 GPU 利用率、降低空转；就像拼车把同方向的人凑一辆车跑）
2. Paged KV cache（PagedAttention 思路）：把 KV cache 按页管理，减少碎片与峰值显存（为了让长上下文与高并发更稳；就像把书按分页装订，随取随放不占一整张桌子）
3. Prefix cache / Prompt cache：对重复前缀（系统提示词、检索模板等）复用缓存（为了把“每次都要读一遍的公共段落”变成一次性成本，显著降低 TTFT；就像常用材料提前做成模板，开工直接套用）
4. 并行与内核优化：张量并行/流水线并行 + fused kernel（GEMM、RMSNorm、RoPE、attention）（为了把计算与通信做满并减少 kernel launch 开销；就像把多道工序合并成一条装配线）
