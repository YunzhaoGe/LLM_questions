# 从 Next-Token Prediction 到 Reinforcement Learning：国产推理/Agent 模型的 RL 创新地图（工程视角）

SFT 仍然是大模型“行为塑形”的主力：指令遵循、格式输出、工具调用 schema、拒答边界，大多依赖高质量监督数据完成。变化发生在另一个层面：当目标从“说得像”升级为“做得成”，业界把更多后训练预算投入到了 RL（以及可验证反馈的自动化闭环）上。

这篇文章不讨论“RL 是否更高级”，只做三件事：

- 把近期国产模型里可见的 RL 创新方向拆成几条“技术轴”
- 解释每条轴在工程上解决的瓶颈、带来的收益、付出的代价
- 给出一个选型视角：不同侧重点会把模型带向不同形态

为了让讨论落到“可识别的工程对象”上，文中会反复出现两个框架名：

- **Forge（MiniMax）**：面向 Agent 的可扩展 RL 训练框架与算法体系（与 M2.5 强绑定）[Forge: Scalable Agent RL Framework and Algorithm](https://www.minimax.io/news/forge-scalable-agent-rl-framework-and-algorithm)
- **Slime（z.ai / 智谱 GLM-5）**：面向大模型 post-training 的异步 RL 基础设施（与 GLM-5 强绑定）[GLM-5 Overview](https://docs.z.ai/guides/llm/glm-5)

---

## 0. 一个统一的坐标系：RL 不是魔法，是“优化目标”的重写

把 SFT 理解为行为克隆更准确：给定上下文，最大化示例中的 token 似然。

$$
\mathcal{L}_{SFT} = - \sum_t \log P(x_t \mid x_{<t})
$$

SFT 的强项是“把正确的说法记下来”。弱项不是“贪婪”，而是目标天然偏局部：它没有直接优化“任务最终是否成功”，也不会主动为可验证的终局目标付出中间步骤的试错成本。

RL/偏好学习把目标改成“最大化回报”。在 LLM 场景里，回报来源常见三类：

- **Outcome reward（结果奖励）**：答案对不对、测试过不过、任务是否完成
- **Preference reward（偏好奖励）**：A 比 B 更好（人评或模型评）
- **Process reward（过程奖励）**：推理过程是否符合某些可解释的规范（更难做稳）

工程上真正决定效果的通常不是“用了 RL”四个字，而是两个问题：

- **Reward 来自哪里（是否可验证）**
- **Rollout 怎么规模化（训练系统是否跟得上）**

---

## 1. 创新方向一：把 Reward 变得“可验证”（Verifier-first）

推理/代码任务里，最便宜也最强的反馈不是人评，而是可自动判定的信号：

- 编译器、单测、lint、类型检查
- 数学/逻辑题的答案校验器
- 结构化约束（JSON schema、DSL 解析成功与否）

这类信号的特点是：**低噪声、高吞吐、可规模化**。它天然适合 outcome-style 的训练：不关心模型怎么想，只关心它能不能把事做成。

代表模型（示例）：

- DeepSeek-R1（推理/代码任务强调可验证信号的价值）
- MiniMax M2.5（在大量真实环境里做 RL，天然依赖“任务是否完成”的反馈）[MiniMax M2.5 News](https://www.minimax.io/news/minimax-m25)

### 影响（你会在模型输出里感受到）

- 更强的“自校验”倾向：先写方案、再写实现、最后自测
- 更稳定的工具调用与多步任务完成率
- 更少“说得很像但跑不通”的答案

### 代价

- 可验证信号覆盖面有限：真实世界很多任务没有标准答案
- 训练容易学到“投机策略”：绕过测试、过拟合 benchmark harness，需要更强的环境多样性来对冲

---

## 2. 创新方向二：把 RL 训练做得更便宜（算法工程：PPO 之后）

经典 PPO 需要价值函数（critic）估计优势（advantage）。对超大模型而言，critic 的显存/吞吐成本会把训练“按死”。

近期很多做法的共同点是：**降低 critic 依赖，或者让 baseline 来自同批次样本的相对比较**。你在一些技术报告里会看到类似“组内比较/相对优势/多样本归一化”的思路（例如 DeepSeek 系列公开提到的 GRPO 路线）。

这条轴解决的是一个很现实的问题：**不是大家不想做 RL，而是做不起**。

代表模型（示例）：

- DeepSeek-R1（公开讨论过 GRPO 一类的“弱 critic/相对优势”思路）

### 影响：RL 降本

- 同样预算下，能进行更多 post-training 迭代
- 更容易把 RL 从“推理”扩展到“Agent 长任务”（因为 rollout 更贵）

### 代价：RL 降本

- 算法更依赖采样质量与奖励设计，reward 噪声会更快放大
- 训练稳定性变得更工程化：分布漂移、奖励黑客、长尾任务会直接影响收敛

---

## 3. 创新方向三：把“搜索”搬到推理时（Test-time compute + Verifier-guided）

很多推理型模型的体验差异，来自推理阶段的计算分配，而不完全来自训练。

常见工程形态包括：

- Best-of-N：采样 N 次，用 verifier/规则筛选最优
- 多轮自修复：失败后改写，直到通过测试或达到预算
- 树搜索式解码（或其近似）：把生成当作搜索问题，用外部打分器引导

这条轴的本质是：**把“回溯/前瞻”放在系统层完成**。模型本身不需要内置 MCTS，但系统可以让它表现得像会回溯。

代表模型（示例）：

- MiniMax M2.5（公开强调更少搜索轮次与 token 效率，属于把“解题路径”当作优化对象的一类工程落点）[MiniMax M2.5 News](https://www.minimax.io/news/minimax-m25)

### 影响：推理时搜索

- 成功率上升，尤其在“能检验”的任务上
- 延迟、成本、token 消耗上升，且更依赖工程调度（预算控制）

### 代价：推理时搜索

- 产品体验更难稳定：同一问题不同次可能花费不同
- 需要一整套“评估与停止策略”，否则会无意义地消耗算力

---

## 4. 创新方向四：Agent 化训练（把“工具/环境”纳入 RL 回路）

聊天任务的反馈通常来自文本偏好；Agent 任务的反馈来自环境状态变化：文件是否写对、命令是否成功、网页是否找到证据、交付物是否可用。

这类训练更像“强化学习的本体”：状态、动作、回报、长时序信用分配。难点集中在三处：

- **动作空间大**：工具种类多、参数复杂、错误类型丰富
- **长 horizon**：一两步做错会导致后面全错
- **环境多样性**：一个 harness 过拟合就会出现“在评测里很强、换脚手架就崩”

代表模型与框架：

- **MiniMax M2.5 + Forge**：官方材料明确把“Agent RL 的系统吞吐、稳定性、框架灵活性”作为核心矛盾，并给出了 Forge + CISPO 的工程化解法（面向任意 Agent scaffold 的大规模 RL）[Forge: Scalable Agent RL Framework and Algorithm](https://www.minimax.io/news/forge-scalable-agent-rl-framework-and-algorithm)

### 影响：Agent 化训练

- 更强的任务拆解与工具使用稳定性
- 更关注“用更少轮次完成任务”（token/round efficiency）

### 代价：Agent 化训练

- 训练基础设施复杂度陡增：需要可复现、可并行、可评估的环境体系
- 安全与合规更难：环境越真实，越要管理越界行为

---

## 5. 创新方向五：RL 训练系统的可扩展性（异步化、解耦、吞吐优先）

LLM 的 RL 训练被 rollout 吞吐卡住是常态：生成轨迹、环境执行、打分、再回传更新，任何一步慢都会拖累整个 pipeline。

这解释了一个现象：当模型规模上到 MoE/百亿激活参数级别，很多团队的“创新点”会落在训练系统而不是公式上。

代表模型与框架：

- **z.ai / 智谱 GLM-5 + Slime**：公开文档明确提到为更大规模、更复杂的 RL post-training 引入 “Slime” 异步 RL 框架（强调训练流程效率与吞吐）[GLM-5 Overview](https://docs.z.ai/guides/llm/glm-5)。媒体报道也将其解读为“异步 RL 基础设施”以缓解长尾瓶颈（信息来源属于二手解读，仅作方向参考）[VentureBeat: GLM-5](https://venturebeat.com/technology/z-ais-open-source-glm-5-achieves-record-low-hallucination-rate-and-leverages)。

### 影响：异步训练系统

- 让 RL 迭代次数变多成为可能（同预算下做更多试错）
- 更容易把 RL 从短任务扩展到长任务/多回合环境

### 代价：异步训练系统

- 工程风险更大：数据一致性、延迟反馈、策略滞后都会带来训练不稳定
- Debug 成本高：需要对“生成-评估-学习”链路全链路可观测

## 6. 把“侧重点”翻译成“模型性格”：一张工程选型表

| 侧重点                       | 主要手段                                 | 代表（示例）                   | 你会得到的能力                  | 典型代价                  |
| ---------------------------- | ---------------------------------------- | ------------------------------ | ------------------------------- | ------------------------- |
| Verifier-first（可验证奖励） | 编译/测试/校验器驱动的 outcome RL        | DeepSeek-R1、MiniMax M2.5      | 更可靠的推理与代码可执行性      | 覆盖面有限，易产生投机    |
| RL 算法降本                  | 弱化 critic、组内相对优势、采样效率优化  | DeepSeek-R1（GRPO）            | 更多 RL 迭代、更快收敛          | 对 reward 噪声更敏感      |
| 推理时搜索                   | Best-of-N、自动修复、verifier 引导       | MiniMax M2.5（token/轮次效率） | 成功率上升，特别是可检验任务    | 延迟/成本上升，需预算控制 |
| Agent 环境交互               | 多环境 RL、任务分解奖励、轮次/token 效率 | MiniMax M2.5 + Forge           | 工具使用稳、长任务完成率提升    | 环境系统复杂、安全压力大  |
| 异步训练系统                 | actor/learner 解耦、rollout 吞吐优化     | GLM-5 + Slime                  | RL 可规模化到更大模型与更长任务 | 工程复杂度与不稳定性上升  |

## 7. 写这类“RL 创新观察”文章的一个实用模板

如果目标是观察国产模型的 RL 创新方向，而不是复述名词，可以按同一套问题去拆每个模型：

- 它的 reward 更像 outcome、preference 还是 process？
- 它的 verifier 是什么，覆盖哪些任务，噪声如何控制？
- 它如何降低 RL 成本（算法、采样、数据、系统）？
- 它把多少能力放在推理时（search/repair），多少放在训练时？
- 它为了 Agent 做了哪些“环境工程”，如何防过拟合 harness？
- 它的 trade-off 是什么（成本、延迟、稳定性、安全）？

---

## 8. 结语：RL 不是替代 SFT，而是把“做成事”的指标引入训练闭环

更贴近工程现实的结论是：

- SFT 决定“会不会按要求说话与用工具”
- RL/可验证反馈决定“在约束下能不能把事做成”
- 推理时搜索决定“在预算内把成功率抬到多高”

写国产模型的 RL 创新，最有信息密度的不是“用了什么算法”，而是它把资源押在了哪条轴上，以及因此获得了什么能力、牺牲了什么指标。
