# 所有 Transformer Block 跑完后，模型如何选出下一个 Token 并反查回文本

如果把前面的所有 Transformer Block 理解成一条层层加工的流水线，那么当最后一层也跑完时，模型手里拿到的仍然不是文字，而是一份“最后当前位置的高级隐藏表示”。这份表示已经吸收了整段上下文的信息，但它本身还不能直接显示给人看。后面还要再经过几步：先做最后一次归一化，再把这个向量投到整个词表空间里，得到每个候选 Token 的分数；然后按照解码策略真正选出一个 Token ID；最后再由分词器把这个 ID 反查成可见文本片段。

所以，从“所有 Block 都算完了”到“用户看到一个新字符或新词”之间，其实还有一小段很关键的尾部链路。可以把它理解成：前面的深层网络负责把上下文想明白，后面的输出头负责把“内部想法”翻译成“词表里的一个具体编号”，再把这个编号还原成人类可读的文本。

---

## 1. 先确认起点：最后留下来的仍然是隐藏状态，不是文字

1. 所有 Transformer Block 跑完后，模型得到的是最后一层的输出隐藏状态。它本质上仍是一组高维向量，而不是汉字、单词或者句子。
2. 对当前这一步生成来说，最关键的通常是“最后一个位置”的那条向量。因为自回归生成要决定的是：**接下来应该吐出哪个新 Token**。
3. 这一步的意义在于，它把整段上下文先压缩进了当前位置的表示里。换句话说，模型不是直接从原文里选字，而是先把上下文变成内部状态，再从内部状态里做下一步决策。

---

## 2. Final Norm：进入输出头之前，通常还要再整理一次数值

1. 在很多现代 LLM 里，所有 Block 结束后，往往还会再经过一次最终归一化，比如 Final RMSNorm 或类似结构。
2. 它的作用和前面各层里的归一化有些相似，都是先把数值尺度整理稳一点，避免最后输出头读到一份过于飘动的表示。
3. 这一步不会凭空增加新的语义内容，更像是正式出结果前的“最后校准”。如果把前面的层看成反复加工，那么这里就像出厂前再做一次统一校表。

---

## 3. LM Head：把内部向量翻译成“整个词表的候选分数”

1. 归一化之后，这个最后位置的隐藏向量会进入输出投影层，通常也叫 LM Head。它会把一个隐藏维度上的向量，映射成“词表大小”那么长的一串分数。
2. 这些分数的含义不是“最终答案已经确定”，而是：**如果下一步要输出一个 Token，词表里每个候选项分别有多匹配当前上下文**。
3. 到这里，模型才第一次真正站到“词表选择题”的层面上。前面的 Transformer Block 更像在理解和加工上下文；这里则像把内部思考翻译成一张覆盖全词表的候选名单。

---

## 4. Logits：这时候拿到的是偏好强弱，还不是概率本身

1. LM Head 输出的这串分数，通常叫 logits。你可以把它理解成“每个 Token 的原始倾向值”。
2. 它们能表示谁更被偏好，但还不是那种已经整理成总和为 1 的概率分布。所以此时更准确的说法是：模型给所有候选 Token 排了一次强弱顺序。
3. 这一步很重要，因为很多人会误以为“分数最高的那个词已经注定会被输出”。其实还没有。后面还要看解码策略怎么处理这些分数。

---

## 5. 解码决策：模型不是“想到什么就直接说什么”，而是要先做一次选择

1. 当 logits 产生后，系统会进入解码阶段，也就是从整张候选表里真正挑出一个 `next_token_id`。
2. 如果采用贪心策略，那么通常就是选当前分数最高的那个 Token。这种方式更稳定、更确定，但也更容易显得保守。
3. 如果采用采样策略，那么系统会先把分数整理成概率，再按概率抽取一个候选。这样生成会更有变化，但波动也更大。
4. 在很多真实产品里，这一步还常常叠加温度、Top-k、Top-p、重复惩罚、结构化约束等机制。它们并不是在“改变模型学到的知识”，而是在控制输出时的风格、风险和可用性。

---

## 6. 选出一个 Token 之后，生成其实还没有结束

1. 当前这一步解码，得到的只是一个新的 Token ID，而不是整段最终回答已经完成。
2. 如果这个新 Token 不是停止标记，系统就会把它接到上下文末尾，然后再进入下一轮生成：重新做嵌入、继续走后续层、再产出下一个 Token。
3. 也就是说，用户看到的一整句话，并不是模型一次性整句想完后整体吐出来的，而是很多轮“算一个 Token、接回去、再算下一个 Token”串起来的结果。

---

## 7. 反查回文本：分词器会把 Token ID 重新翻译成人类可读片段

1. 一旦某个 `next_token_id` 被选出来，分词器就会去词表里反查它对应的 Token 片段。
2. 这里反查出来的，不一定总是一个完整汉字、单词或一句话。它可能是一个字、半个词、带空格的片段，甚至是标点和控制符。
3. 这也是为什么模型内部生成单位是 Token，而用户界面上看到的文字效果，往往是多个 Token 连续拼接后才逐渐变得自然可读。

---

## 8. 为什么“反查回文本”不等于“逐字符打印”

1. 很多人会直觉上觉得，模型是不是每次都吐一个字出来。这个理解并不准确，因为 Token 和字符不是一一对应关系。
2. 有些 Token 可能正好对应一个汉字，有些对应一个完整英文单词，还有些可能只是词的一部分，或者带着前导空格。
3. 所以 detokenize 更像“按词表片段重新拼装”，而不是像老式打字机那样单纯逐字符输出。你看到的自然语言，其实是 Token 片段经过连续还原后的显示结果。

---

## 9. 一个常见误区：模型输出文字，不代表它内部真的“存着整句文本”

1. 模型内部主干网络保存的不是待输出的现成句子，而是隐藏状态、分数偏好和逐步展开的生成过程。
2. 它并不是先在脑子里写完完整答案，再逐字打印出来；更常见的情况是，每一步只根据当前上下文和内部状态决定下一个 Token。
3. 这也是为什么解码策略会明显影响最终文风。因为“同一份内部倾向”在不同选择规则下，可能会展开成不完全一样的自然语言表面形式。

---

## 10. 这条尾部链路里，近两年有哪些变化特别重要

1. 结构化解码变得越来越常见。很多系统不再只追求“像人说话”，还要求输出 JSON、函数调用参数或严格格式，所以“怎么从 logits 里约束候选集合”变得越来越关键。
2. 流式输出几乎成了标准体验。用户不再总是等整段答案结束才看到文字，而是边生成边看到 detokenize 后的片段，因此“Token 级输出如何平滑显示”成了产品层的重要问题。
3. 推测式解码、前缀缓存和更高效的采样实现越来越受关注。它们不改变“最后还是要选 Token 再反查文本”这个本质，但会明显改变用户感受到的首字延迟和整体速度。

---

## 11. 一句话复盘

当所有 Transformer Block 都跑完后，模型并不会直接吐出文字，而是先拿到最后位置的隐藏状态；再经过 Final Norm 和 LM Head，把这份内部表示翻译成整个词表上的 logits；接着通过解码策略从中选出一个新的 Token ID；然后由分词器把这个 ID 反查成可显示的文本片段；如果还没遇到停止条件，这个新 Token 又会被接回上下文，推动下一轮生成。也就是说，用户看到的自然语言，其实是“深层隐藏状态 → 词表打分 → 选择 Token → 反查文本”这条尾部链路一轮轮滚动出来的结果。
