# 推理 Tokenization 全流程（Prompt → token_ids）

目标：从用户输入开始，串起 chat template 拼接、（可选）文本规范化、预分词、子词编码（BPE/Unigram + 字节回退）、特殊 token 处理、截断与 mask 生成，直到产出可直接喂给模型的 `token_ids` 与相关张量。每一步同时回答“做了什么/为什么这么做”，并配一句“就像……”的类比，建立可复用的直觉。

---

## 1. 输入拼接与规范化（得到可分词的文本）

本节作用：把“多轮消息”变成一段稳定、可复现的文本输入，并在不破坏语义的前提下减少编码噪声，为后续分词提供统一起点。

1. Chat Template Formatting：把 system/user/assistant 等消息按模板拼成单一文本串，并插入 `<BOS>、<EOS>、<USER>、<ASSISTANT>` 等特殊标记（为了标准化多轮对话格式，让模型能区分角色与边界；就像先把散乱的对白整理成一份分镜清晰的“剧本”）
2. Unicode Normalization（可选）：对拼接后的文本做 NFKC 等归一化（为了减少编码变体带来的碎片化，让“同一个字符”不要被当成不同符号；但对代码、日志、符号敏感的场景常保持原样以避免语义被“纠正”；就像把不同字体统一成印刷体，但遇到公式和代码要保留原样）

---

## 2. 预分词与字节化（把文本变成可合并的基础单元）

本节作用：把“字符级文本”转成分词器可处理的中间表示（片段 + 字节符号），确保任何输入都能进入子词编码阶段，避免因罕见字符导致流程中断。

下面以“支持任意字符的子词编码”为例（BPE/Unigram 的细节不同，但工程主线一致；现代 LLM 常配合字节回退来覆盖罕见字符）。

1. Pre-Tokenization：按空格/标点/规则把文本粗切成片段（为了在不丢信息的前提下给后续合并提供边界与起点；就像先把长句粗切成几大块“原料”）
2. Byte-Level Encoding：将片段编码为 UTF-8 字节序列（为了覆盖任意 Unicode 字符、表情符号与代码，避免遇到生僻字符就“分不出来”；就像把文本转成通用的“二进制砖块”，什么字符都能搭）
3. Initial Byte Splitting：把字节序列拆成最细粒度符号（用于 byte-level BPE 或 byte fallback；为了确保任何输入都有可表示的起点；就像先把零件拆到最小的螺丝与垫片，再决定怎么组合）

---

## 3. 分词器资产加载（一次性准备，推理时复用）

本节作用：把“分词规则”与“映射表”加载到内存中，确保同一段文本在不同机器/不同时间得到一致的 token 化结果，并把分词开销压到足够低以适配线上并发。

1. Vocabulary Loading：加载词表（token → id）（为了把最终符号落到一个有限集合，便于后续 embedding 查表；就像打开仓库清单，确认有哪些标准零件可用）
2. Subword Model Loading：加载子词模型参数（BPE merges/ranks 或 Unigram 的 token 及分数）（为了让编码过程可复现训练时的子词体系；就像拿到“有哪些标准模块、以及怎么拼装/怎么选”的工艺表）
3. Special Tokens Config：加载/注册特殊 token（如 BOS/EOS/PAD 等）（为了让模型得到边界与控制信号；就像给流程加上“起点/终点/占位符”这些固定路标）
4. Fast Tokenizer Runtime（工程常见）：使用 Rust/SIMD 实现与缓存初始化结果（为了把 tokenization 从“CPU 瓶颈”变成可忽略的开销；就像把手工分拣换成流水线分拣机）

---

## 4. 子词编码（推理时的实际分词核心）

本节作用：把字节/片段序列压缩成“模型词表里的 token 序列”，在可逆与可覆盖的前提下尽量缩短序列长度，从而降低后续注意力计算的成本并提升吞吐。

注意：这里把“子词编码”展开说明。BPE 与 Unigram 的推理算法不同：

1. BPE：训练阶段基于语料统计产出 merges；推理阶段用**预先训练好的 merges 优先级**做确定性的贪婪合并。
2. Unigram：推理阶段通常用动态规划在候选 token 中找最优分割（而不是 merges）。

1) BPE Pair Ranking：枚举相邻 pair，并查表得到它们的 merge rank（为了知道“哪些相邻符号允许合并、且应该先合并哪一对”；就像检查相邻零件是否有标准卡扣，并比较卡扣优先级）
2) BPE Greedy Merging Loop：反复选择 rank 最小（优先级最高）的 pair 合并，直到不存在可合并 pair（为了把序列压缩成更长、更语义化的子词，同时保持确定性；就像按工艺表把零件组合成标准模块，直到无法继续拼装）
3) Unigram Best-Path（如果使用 Unigram）：在所有可匹配 token 中找一条“覆盖整段文本”的最优路径（为了在多个切分方案里选一个整体最合理的；就像用导航在多条路线里选总耗时最短的一条）
4) Unknown / Fallback Handling：对极端输入的兜底（现代 LLM 常用 byte fallback，通常不依赖 `<UNK>`）（为了保证任意输入都能表示，不让分词流程崩掉；就像再奇怪的零件也能用“单颗螺丝”方式装进清单）

---

## 5. 输出组织与张量化（得到模型可用的输入）

本节作用：把 token 序列变成模型真正接收的输入张量（ids + mask + 边界控制），并对长度与批处理对齐做最后的工程处理，确保推理可并行且结果可控。

1. Special Token Insertion：按需要插入/保留 BOS/EOS、分隔符等（为了标记序列边界与角色切换；就像在剧本里插入“场记板：第几幕/谁在说话”）
2. Token ID Mapping：把 token 映射为 `token_ids = [id0, id1, ..., id(n-1)]`（为了得到可计算的整数索引；就像把零件名称换成仓库里的货号）
3. Truncation / Padding：超过最大上下文时截断或滑窗，不足时补 PAD（为了满足模型长度约束并支持 batch；就像把行李装进规定尺寸的箱子，超出就取舍，不够就用泡沫填充）
4. Mask Generation：生成 `attention_mask`（padding mask）以及推理侧可能需要的其他 mask（为了让模型忽略 PAD 位置；因果约束通常在注意力内部实现；就像用遮挡板把“空位”盖住，不让它参与讨论）
5. Tensor Creation：把 `token_ids` 与 mask 转为框架张量（PyTorch/TF 等）（为了进入 embedding 查表与后续矩阵计算；就像把纸面清单录入系统，流水线才能开工）
