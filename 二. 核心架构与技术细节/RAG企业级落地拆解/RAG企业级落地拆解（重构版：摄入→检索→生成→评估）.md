# RAG 企业级落地全流程（摄入 → 检索 → 生成 → 评估）

目标：从企业知识源开始，串起离线摄入（清洗/切块/向量化/建索引/权限元数据）、在线查询（检索/重排/上下文组装/缓存与路由）、LLM 生成（减少幻觉、引用对齐与格式化输出）、以及评估监控与反馈闭环。每一步都同时回答“做了什么/为什么这么做”，并配一句“就像……”的类比，帮助建立可落地的 RAG 系统心智模型。

---

## A. 离线摄入（1–3）

1. 数据源采集：从 PDF/Word/网页/数据库/CSV/内部知识库/实时流等拉取原始数据，并做结构化抽取（文本/表格/图片）（为了覆盖信息面并保留原始出处与版本；就像把分散在各部门的资料统一收进档案室，并保留来源标签方便追溯）
2. 清洗与规范化：去噪（HTML、重复段落、脚注）、脱敏（PII）、以及必要的编码与语言处理（Unicode、语言检测、OCR 纠错等）（为了避免“脏内容”污染向量语义与召回排序；就像先把原材料里的砂石杂质挑掉，不然后面做出来的零件精度全会飘）
3. 切块与加元数据：把长文档切成 chunks（固定长度/语义边界/层次结构、parent-child、多粒度切块等），同时附加来源、时间戳、权限标签、章节路径等元数据（为了在召回粒度与上下文完整之间平衡，并把权限过滤与引用对齐做成“系统能力”；就像把一本大书拆成可引用的小卡片，还在卡片背面写上书名、页码、目录位置和“谁能借阅”）

---

## B. 建索引与查询入口（4–6）

4. 向量化与建索引：用 embedding 模型把 chunk 编码为向量，并与稀疏索引（BM25/learned sparse）一起写入检索系统（向量侧常用 HNSW/IVF/PQ，也可能用多向量/late-interaction 结构）（为了把召回做成可组合的“多路检索”，兼顾语义、关键词与长文档精匹配；就像既有按主题摆放的书架（Dense），也有按书名/关键词的目录卡（Sparse），需要时还能按段落做精确定位）
5. 查询接收与预处理：接收用户 query，做意图识别/路由（选知识库、选检索策略、选是否走 RAG），并做可选重写（扩展同义词、拆子问题、HyDE 等）与缓存命中（语义缓存/答案缓存/检索缓存）（为了把“用户口语问题”变成更适合检索且更省成本的请求；就像把口头报修翻成标准工单，并先查知识库看是否已有同类已解决方案）
6. 检索（多路召回）：Dense（向量近邻）+ Sparse（BM25/learned sparse）+ Hybrid 融合，并可叠加多跳/迭代检索、图谱/关系检索，以及 metadata filter 与去重/多样性约束（为了既广覆盖又能对关键词、实体与关系做“硬命中”；就像先圈片区（Dense），再按门牌号/姓名精查（Sparse），必要时顺着关系网找到“相关联系人”的资料）

---

## C. 候选到证据（7–9）

7. 重排序（Re-ranking）：对 top-K 候选用 reranker（Cross-Encoder、专用 reranker，或 LLM ranker）重新打分排序（为了把“看起来差不多”的候选按语义精确相关性拉开差距；就像先粗筛出 50 份简历，再让面试官逐份对照岗位要求打分）
8. 上下文组装：把 top-M chunks 组装成可给 LLM 的 context，附带引用与来源；必要时做压缩/摘要/去冗余（为了在有限上下文窗口里装进“最有用的证据”，并降低 token 成本与延迟；就像把一堆材料整理成一页 briefing，只保留对决策有用的要点并标注出处）
9. 增强 Prompt 构建：把 system 规则 + context + 用户问题组装成 augmented prompt，并明确“仅基于上下文/无证据则说不知道/必须给引用/引用要可追溯到 chunk”等约束（为了让模型把检索结果当作主要事实来源，并把“回答-证据”绑定起来；就像写论文时要求“每个结论都要有参考文献，且能定位到页码”）

---

## D. 生成与把关（10–12）

10. LLM 推理与输出：调用 LLM 生成（通常采用流式输出），并选择合适解码策略（为了在延迟、成本与输出稳定性之间取平衡；就像演讲时先把前几句话顺畅说出来，后面再边组织边补充细节）
11. 后处理与验证：做 groundedness/一致性检查（自评或外部 verifier）、引用校验（答案中的引用是否真出自上下文）、结构化提取（JSON/schema）、以及必要的安全与合规过滤（为了把不可靠或不合规的输出挡在系统边界内；就像稿件发表前不仅要审校文风，还要逐条核对参考文献是否真的支持结论）
12. Offline 评估：用标注集或合成集评估 Faithfulness、Answer Relevancy、Context Precision/Recall 等，并把“检索、重排、生成、引用”拆开看指标（为了能定量比较不同 embedding/reranker/chunk/检索融合策略的收益；就像质检不仅看成品合格率，还要追溯是原料、工艺还是装配环节造成的偏差）

---

## E. 在线迭代与闭环（13–15）

13. Online 监控与 A/B：监控 E2E 延迟、TTFT、tokens 成本、失败率、用户反馈，并做 A/B 对比不同配置，结合 trace 记录每步命中与引用覆盖（为了用真实流量验证改动是否有效且能快速定位回归来源；就像小批试产时把每道工序的参数都记下来，出问题能立刻定位哪台机器）
14. 反馈闭环：把用户纠错/点赞/工单结果回流到知识库与模型侧（增量摄入、更新索引、微调 embedding/reranker，或 active learning 选样复标）（为了让系统对业务变化持续自适应；就像售后数据反哺研发，下一版产品才会越做越稳）
15. 增量/Delta 摄入：只处理新增与变更数据，并记录版本、回滚点与权限变更（为了避免全量重刷带来的成本与服务抖动，同时保证权限收敛；就像只补丁更新改动文件，并同步更新访问控制列表）

---

## F. 运营与规模化（16–18）

16. 多模态扩展：对图像/表格/版式等做多模态解析与 embedding，统一进入检索体系（为了覆盖企业知识里大量“非纯文本”的信息；就像不仅能搜文档正文，还能搜截图、表格与流程图）
17. 成本分层：用便宜模型做粗筛（embedding/轻量 reranker），用更强模型做精排/压缩/验证（为了把算力花在“最接近最终答案”的少量候选上；就像先用门禁刷卡筛人，再对少数可疑对象做人工复核）
18. 可观测性：监控摄入成功率、chunk 分布、召回/重排命中、答案忠实度与引用覆盖率，并把可观测数据与权限审计关联（为了快速定位问题是在“数据”“检索”还是“生成”，以及是否出现越权引用；就像仪表盘不仅告诉你发动机、轮胎还是油路出了问题，还能告诉你是否有人把油箱盖偷打开了）
