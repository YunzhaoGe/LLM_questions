# 从 token 到 action：LLM 如何接到外部执行面

如果你已经知道 token、tool calling 和 computer use 这些概念，那理解“从 token 到 action”时，最重要的一点不是“模型会不会规划任务”，而是“它为什么敢把一串 token 接到真实执行面上之后，还尽量不把外部世界操作偏”。因为外部执行面不是普通输出通道，它后面接的是浏览器、窗口、文件、邮件草稿、权限确认。也就是说，这条链路不是单纯在做“把语言变成动作”，而是在做“把只会生成 token 的模型，接到一个会真实改动系统状态的环境里，同时尽量不让动作跑偏”。

可以把这条主线理解成：先把用户请求固定成一张可执行任务单，再把模型输出收束成动作候选；接着把这些候选动作挂到一组真实可用的执行把手上，然后根据当前屏幕和窗口状态把动作落到具体目标；真正执行时，不是直接完成一个大任务，而是拆成一串很小的原子操作；每做完一步，再把外部环境的新状态重新编码回模型上下文，必要时还要在关键动作前插入权限检查，最后才让任务在真实世界里留下正确痕迹。

为了把这条链路讲清楚，本文固定一个 mini-case。用户给 Agent 的任务单记作 T0：确认下周一与客户 A 的会议时间；搜索客户 A 最近一周的公开新闻；把结论写入共享盘里的《客户A-周会.pptx》；最后生成一封发给销售团队的邮件草稿。后文不再更换例子。我们只跟踪 T0 在系统里产出的几类对象：会话状态 S、动作候选 A、能力目录 C、观察包 O、执行环境 E，以及最终落地到外部系统里的文件和邮件草稿。

---

## 1. 先明确目标：这条链路接的不是普通文本，而是会真实改动外部环境的任务单

1. 它一开始处理的对象是什么：不是一句“帮我做点事”的模糊请求，而是一张待执行任务单 T0。进入运行时之后，T0 会先变成会话状态 S0，里面至少要绑定目标列表、上下文、可用能力、权限边界和完成条件。
2. 为什么这比普通对话更难：因为普通对话答偏了，后果通常还停留在文本里；但 computer use 一旦走偏，后果会直接落到浏览器、文件和邮件上。也就是说，这里不是“说错一点”，而是“可能真的点错、写错、改错”。
3. 所以后面真正要保住的是什么：不是让模型把任务描述得很完整，而是让这张任务单在后续每一步里都保持同一条主线。对 T0 来说，后面所有动作都得继续围绕“查会议信息、搜新闻、改 PPT、起草邮件”这四个目标展开。

---

## 2. 不是上来就执行：它会先把语言目标改写成一份稳定的会话状态

1. 为什么不能直接拿用户输入往下做：原始请求里通常混着目标、顺序、隐含约束和风险边界。如果不先整理，模型每一轮都可能重新解释任务，今天先搜新闻，下一轮又突然先写邮件，动作顺序会漂。
2. 运行时先做什么：它会把 T0 改写成 S0。你可以把 S0 理解成一份任务快照：当前要做什么、已经做到了哪一步、哪些动作可以自动做、哪些动作需要确认、现在有哪些工具和执行面可用。
3. 这一步的意义：一旦语言请求被压成稳定状态，后面每轮推理读到的就不再是原始自然语言，而是“同一张任务单的当前快照”。这也是为什么信息图第一张最适合画 T0 到 S0 的状态迁移，而不是先画浏览器界面。

---

## 3. token 不会直接变成点击：它会先被收束成动作候选

1. 这一步到底发生了什么：模型连续生成 token 之后，运行时不会立刻把它们发给鼠标和键盘，而是先收束成动作候选 A1。对 T0 来说，A1 可能是“先读取会议安排，再搜索客户 A 新闻，再定位《客户A-周会.pptx》”这样的结构化提议。
2. 这里一个很容易误解的点：很多人会觉得，只要模型说出了“打开浏览器并搜索客户 A”，系统就已经知道怎么执行了。这个说法并不完全对，因为自然语言里并没有明确标出执行面、参数边界、失败回退方式，也没有说明这一步到底该走 GUI、API 还是文件系统。
3. 为什么这一步值钱：因为动作候选层第一次把“语言里的意图”变成“运行时可以继续验证和路由的对象”。从信息图角度看，这一步最适合画成一束 token 流进入一个解析器，再输出一张动作卡片 A1。

---

## 4. 不是每个动作都走同一条路：系统会先把世界暴露成一组可用把手

1. 这一步的主对象是什么：是能力目录 C0。它不是一个抽象概念，而是运行时当前真的能给模型使用的那些执行把手。对 computer use 来说，常见把手包括读屏、取窗口树、移动光标、点击、输入、滚动、读写文件、请求用户确认。
2. 为什么必须显式暴露这一步：因为模型并不知道当前机器上到底能不能直接读日历、能不能访问共享盘、能不能调起邮件客户端。没有 C0，模型只能凭训练语料“想象”世界；有了 C0，它才会在一个真实、受控、有边界的动作空间里选下一步。
3. 这一步进入下一步时，会把 A1 和 C0 放在一起做路由判断。也就是说，系统接下来要解决的问题已经不再是“模型想做什么”，而是“它想做的事，当前到底应该接到哪个真实执行面上”。

---

## 5. 真正的分叉点在这里：同一个目标，可能落到不同的外部执行面

1. 对 T0 来说，哪些执行面会出现：读取会议安排更像是结构化读取，理论上更适合 API 或日历连接器；搜索客户 A 新闻可以走搜索 API，也可以走浏览器；修改《客户A-周会.pptx》可能走文件系统加 Office 集成，也可能直接走 GUI；生成邮件草稿既可能走邮件 API，也可能落到桌面客户端。
2. 为什么这一步很关键：因为不同执行面的代价完全不同。API 稳定、结构化、便于校验；GUI 通用，但最脆弱；文件系统适合改持久状态，但不等于完成业务动作。系统如果不先做执行面选择，就容易把一切问题都粗暴丢给 computer use 的 GUI 能力。
3. 所以这里真正发生的状态变化是：A1 不再是一张中性动作卡片，而是被拆成带执行面标签的动作队列。对信息图来说，这一步最适合画成一张分流图：同一个目标，从中间分到 API、GUI、文件系统和权限层。

---

## 6. 一旦走到 GUI，问题就变了：系统不再面对接口，而是在面对当前屏幕

1. 这一步先要拿到什么：是观察包 O1。它的内容不是普通返回值，而是执行环境 E0 的可读表示，比如截图、窗口标题、可访问性树、焦点位置、按钮标签、文件列表。对 T0 而言，O1 可能显示浏览器已打开、共享盘窗口在前台、《客户A-周会.pptx》出现在文件列表里。
2. 为什么 GUI 环境特别难：因为 GUI 没有像 API 那样稳定的字段接口。模型不能只说“下一步点搜索框”，它必须先知道搜索框在不在、当前是不是正确窗口、界面有没有刷新、焦点是不是已经落在输入区。也就是说，动作能不能落下去，先取决于有没有正确观察到当前世界。
3. 这一步的意义：它把原本抽象的动作候选 A2，压缩成“面向当前屏幕状态的下一步动作”。信息图如果画这一章，主角不再是任务单，而是屏幕截图加标注框：按钮、输入框、窗口标题和焦点位置。

---

## 7. 真正执行时，不是完成大任务，而是把高层意图拆成一串原子操作

1. 这一步具体怎么变：一旦 O1 确认了落点，动作候选就会被翻译成执行环境里的原子事件序列。对 T0 来说，“搜索客户 A 新闻”不会作为一个整体动作直接发生，而是被拆成切到浏览器、点击搜索框、输入客户 A 名称、回车、等待页面刷新这几步。
2. 为什么必须拆得这么细：因为 GUI 世界理解的是点击、输入、滚动、等待，不理解“帮我搜一下新闻”这种高层业务语义。模型擅长决定“做什么”，但真正保证这一步能落在正确按钮上的，是执行层对坐标、窗口、时序和失败重试的处理。
3. 这一步之后，执行环境会从 E0 变成 E1。也就是说，外部世界真的被改动了：页面从空白态变成结果页，文件从未打开变成已加载，邮件窗口从关闭变成草稿态。对信息图来说，这一章最适合画时间轴，而不是概念框图。

---

## 8. 外部世界不会返回“理解”，它只会返回一批新的环境证据

1. 一个容易被忽略的事实：执行层做完动作之后，返回给系统的不是“任务推进了 20%”这种高层结论，而是新的环境证据。比如页面刷新后的新闻列表、PPT 新增的一页摘要、邮件客户端里的未发送草稿、或者一个错误弹窗。
2. 为什么这一步重要：因为 LLM 仍然只能继续吃 token，它并不能直接理解“刚才鼠标点击成功了”这种机器状态。所以运行时必须把这些异构结果重新编码成新的观察包 O2，并且尽量压掉不必要的噪声。
3. 这一步的本质：不是简单记录日志，而是把外部世界重新翻译回模型能消费的上下文。对 T0 来说，O2 可能被整理成“已抓到三条新闻、PPT 第 7 页新增摘要块、邮件草稿已生成未发送”这样的状态更新。

---

## 9. 真正驱动多步 Agent 的，不是长思维链，而是“动作 → 观测 → 再动作”的闭环

1. O2 回来之后会发生什么：它不会直接变成最终答案，而是被回写进会话状态，形成 S1。S1 记录的不是单纯历史文本，而是任务单 T0 在当前时刻的执行快照：哪些子任务完成了，哪些还没完成，下一步应该继续搜新闻、继续改 PPT，还是转入邮件草稿确认。
2. 为什么很多人容易误解这里：大家会把 Agent 的强弱归结为“模型能不能想很多步”。这个说法不够准确，因为只靠内部想很多步，并不能保证外部世界真按那个计划发生了。真正稳定的多步行为，来自每一步都先执行、再观察、再继续。
3. 所以这条链路真正像什么：更像一条不断回写状态的闭环控制链，而不是一段一次性展开到底的长文本。信息图画到这里时，最适合把箭头闭合起来，形成 T0 → A → E → O → S 的循环。

---

## 10. 关键动作前必须插入权限层：因为这时错的不是句子，而是外部后果

1. 到了哪一步风险会突然变大：通常是发邮件、提交表单、覆盖共享盘文件、关闭窗口前保存修改这些动作。对 T0 来说，生成邮件草稿可以自动做，但真正把邮件发给销售团队，往往必须经过用户确认。
2. 为什么这一步不能省：因为前面几步就算出错，很多时候还停留在可回退的观察和草稿状态；但一旦跨过权限层，错误会直接变成真实后果。也就是说，这里系统要保住的不是“语言正确性”，而是“动作后果的可控性”。
3. 所以权限层真正做的事，不是增加一个多余流程，而是把 A3 从“模型建议的下一步”改写成“允许执行、拒绝执行或等待确认的动作”。对信息图来说，这一步最适合画成一条横跨各执行面的策略闸门。

---

## 11. 最终写进外部世界的，不是一段回答，而是一组真实被改动的对象

1. 当 T0 完成时，外部系统里会留下什么：日历信息已被读取，三条新闻已整理成摘要，《客户A-周会.pptx》已被更新，邮件草稿已生成并停留在待发送状态。也就是说，任务单不再只是语言里的目标，而变成了外部对象上的真实痕迹。
2. 为什么这一步才是最终判据：因为前面所有中间环节都只是为了保证这里不跑偏。只要最终文件没改对、新闻没抓全、草稿状态不对，那么前面再漂亮的 token 流、动作候选和观察包，也只是没有落地的中间过程。
3. 所以从工程角度看，这条链路真正保住的不是“模型说得像会做事”，而是“模型通过运行时之后，真的把正确对象改成了正确状态”。这也是最适合信息图收尾的一张图：四个外部对象并排展示它们的最终状态变化。

---

## 12. 一句话复盘

如果按主线流程来理解，从 token 到 action 不是“模型突然学会用电脑”这么简单。更准确的流程是：先把用户请求固定成一张可执行任务单 T0，并压成稳定的会话状态 S0；再把模型输出的 token 收束成动作候选 A1，而不是直接送去执行；接着根据当前可用能力目录 C0，把这些动作路由到 API、GUI、文件系统等真实执行面；如果动作走到 GUI，就先拿观察包 O1 读取当前屏幕和窗口状态，再把高层意图拆成原子操作去改动执行环境 E；做完之后，外部世界返回新的环境证据，运行时再把它们压成 O2 并回写到会话状态里，驱动下一轮动作；碰到发邮件、覆盖文件这类高风险步骤时，还要先经过权限层，把动作后果卡在可控范围内；最后真正落到外部系统里的，是被正确修改的文件、草稿和界面状态。也就是说，这条链路真正保住的，不是 token 本身，而是“同一张任务单在多轮执行中始终落到正确对象、正确位置和正确后果上”。

这篇会专门把：

- 为什么 GUI 执行面比 API 更脆弱
- 为什么观察包比规划本身更关键
- 为什么权限层其实是这条链路的稳定器

这三件事串起来讲清楚。
