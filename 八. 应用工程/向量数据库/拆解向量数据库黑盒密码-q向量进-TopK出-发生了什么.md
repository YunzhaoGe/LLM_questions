你把一个查询向量 q 交给向量数据库，顺手带上 k=10、一个过滤条件和若干“召回/延迟”参数。几毫秒到几十毫秒后，系统吐出 Top‑K 的结果列表。看似简单的一进一出，背后其实是一条被工程折叠起来的流水线：路由、分片、段管理、近似检索、过滤、精排、合并、裁剪、返回。

这篇文章的目标只有一个：把“q 向量进 → Top‑K 出”这条链路拆开，回答两个关键问题：

- 为了快，系统到底“省略”了哪些计算？
- 为了准，它又在哪些阶段“补回”了这些计算？

## 0. 先建立一张心智地图

把向量数据库当作一个分布式的“候选生成器 + 排序器”最贴切。Top‑K 不是一步算出来的，而是两步：

1. 召回（Recall / Candidate Generation）：用近似方法快速找到一批“看起来可能近”的候选（通常是 k 的几十倍到几百倍）。

2. 重排（Rerank / Refine）：对候选做更精确的距离计算与过滤，最后裁剪出 Top‑K。

流水线常见形态如下：

```
请求进入
  ↓
校验与归一化（可选）
  ↓
路由到分片 / 节点
  ↓
选择可见数据段（segments）
  ↓
ANN 索引检索（HNSW / IVF / …）→ 候选 IDs（M 个）
  ↓
过滤（pre / post）→ 候选 IDs（M' 个）
  ↓
精算距离（读取原始向量）→ (id, exact_score)
  ↓
Top‑K 选择 + 分片结果合并
  ↓
返回
```

其中，黑盒的“密码”主要藏在 ANN 索引检索这一格：它如何在不看全量向量的前提下，把搜索空间缩到可控范围。

## 1. 输入的 q，真的是“一个向量”吗？

表面上你只传了一个数组，但系统实际收到的是一个“检索请求对象”，典型包含：

- 向量 q：维度 D、数据类型（float32/float16/int8）、是否已归一化
- k：期望返回条数
- metric：L2 / Inner Product / Cosine（Cosine 常被实现为“归一化后的内积”）
- filter：metadata 条件（范围、等值、集合、时间窗、权限、租户等）
- search params：例如 HNSW 的 efSearch、IVF 的 nprobe、是否启用 rerank、候选放大倍率等

这里埋着第一个“黑盒坑”：

### 1.1 Cosine vs Inner Product：很多系统用“归一化”偷换概念

- Cosine 相似度需要向量归一化（单位长度），否则角度相似性会被模长干扰。
- 一些系统为了速度，会把 Cosine 变成：先把向量归一化，再做内积。

这意味着：归一化究竟发生在客户端、写入时、还是查询时，会直接影响一致性与性能。查询时归一化会增加一点点 CPU，但能保证即使库里向量没归一化也可用；写入时归一化省查询成本，但要求写入路径严格一致。

### 1.2 数据类型与带宽：你以为算力是瓶颈，实际常是“读不动”

距离计算（尤其是 L2/内积）本质是大规模向量点积，常常受限于内存带宽而非算术单元。于是出现了工程上的典型取舍：

- float32 → float16：减半带宽，可能略损精度
- float → int8 量化：再降带宽，但需要尺度/零点与误差补偿
- PQ/OPQ：不是逐元素量化，而是“分块编码”，把向量压成极短码字

这些手段几乎都在为同一个目标服务：让“候选生成”阶段尽量少读原始向量。

## 2. 路由：你的请求先被拆成“很多个小请求”

单机向量索引做到一定规模后，最直观的问题是：内存放不下，更新太慢，查询不稳定。于是系统通常会把数据拆成：

- 分片（shard）：按 id 范围 / hash / tenant / partition key 分布到不同节点
- 段（segment）：节点内部的不可变数据块（类似 LSM 的 SSTable 概念），配合小的可变写入缓冲区

查询路由的核心动作：

1. 决定 fan‑out：要打到哪些分片

- 全局检索：打到所有分片（代价高，但通用）
- 分区检索：基于 partition key 只打少数分片（更快、更稳定）

2. 每个分片内，选择哪些 segments 可见

- 过滤掉已经被 tombstone 标记删除的段
- 根据时间窗/版本/权限等条件跳过一部分段

这一层的“黑盒技巧”是：尽量在“不碰向量”的情况下剪枝。

## 3. 黑盒核心：ANN 到底怎么把搜索空间缩小？

如果做精确 Top‑K（Exact KNN），最朴素的方法是对库中每条向量都算一次距离，然后取最小/最大 k 个。复杂度是 O(N·D)，N 大了基本不可接受。

ANN（Approximate Nearest Neighbor）的价值在于：不保证绝对最优，但用可控的召回损失换取数量级的加速。常见实现路线主要两大类：

- 图索引：HNSW / NSG / DiskANN（用“邻接图 + 局部扩展”搜索）
- 聚类倒排：IVF（先聚类成桶，再只扫少数桶），再叠加压缩：PQ/OPQ

下面按“查询时发生了什么”来拆。

### 3.1 HNSW：像在地图上“爬山”找最近点

直觉：把所有向量点连成一张“近邻图”。查询时你不看全量点，而是从某个入口出发，沿着“更接近 q 的邻居”一步步走，探索一小片区域就能逼近最近邻。

HNSW 的查询过程可以粗略理解为两阶段：

1. 高层粗定位（多层小世界图的上层）：快速把入口移动到 q 附近的区域
2. 底层精搜索（第 0 层）：用一个候选队列扩展邻居，直到预算耗尽

关键参数是 efSearch：它不是“返回多少”，而是“探索多少”。efSearch 越大，召回越高，延迟越大。

一个更贴近“查询时到底在做什么”的自然语言心智模型：

1. 先从索引的入口点 entrypoint 出发，在最高层开始做“贪心下山”：在这一层只沿着能让 q 距离更近的邻居移动，直到找不到更近的点为止；然后带着这个位置进入下一层，重复同样的过程，一直下到第 0 层。此时你大概率已经被带到了 q 附近的一片区域。

2. 到第 0 层后，不再只走一条路径，而是做“带预算的扩展搜索”。系统会维护两类集合：

- 一个“待扩展队列”：优先从当前看起来最接近 q 的点开始扩展它的邻居
- 一个“最好集合”（大小上限约等于 efSearch）：保存目前找到的最优候选点，用其中最差的那个作为剪枝门槛

3. 每次从待扩展队列里取出一个离 q 最近的候选点 v，枚举 v 的邻居 u：

- 如果 u 没访问过，就计算 u 到 q 的距离
- 如果“最好集合”还没满，或 u 比当前最差候选更好，就把 u 加入待扩展队列与“最好集合”

4. 当待扩展队列中最接近的点都已经“不可能打败”当前“最好集合”里的最差者时，就停止扩展（剪枝生效）。

5. 最后从“最好集合”里挑出最好的 k 个作为近似 Top‑K 结果。

黑盒点在于：这套机制把“看 N 个点”变成“看大约 efSearch 个点”，而 efSearch 远小于 N。代价是：如果图结构对数据分布不友好，或者过滤把可用点剪得太碎，搜索可能陷入局部最优。

工程上常见的进一步加速：

- 距离计算 SIMD 化：一次算多个维度
- 访问去重（visited bitset）：避免重复扩展
- 邻居列表缓存：减少随机内存访问

### 3.2 IVF：先找“桶”，再在桶里找人

IVF 的核心思想是把全库向量先聚成 C 个中心（centroids），每个向量只挂到一个（或少数）中心对应的倒排列表里。

查询时发生：

1. 计算 q 到所有中心的距离（或用近似方法找最近中心）
2. 选出前 nprobe 个中心
3. 只扫描这 nprobe 个中心的倒排列表

于是，扫描量从 N 变为：nprobe 对应的列表长度之和。

黑盒点在于 nprobe：它决定你愿意“探多少桶”。桶探得越多，召回越高，越慢。

### 3.3 PQ/OPQ：不读原向量，先用“压缩码”算个大概

即使你已经只扫少数桶，桶里的向量也可能还很多。读取原始向量做点积仍然贵。PQ（Product Quantization）把每条向量分成 m 个子向量，每个子向量用一个码本里的“码字 id”表示。这样一条向量可以压成 m 个字节（甚至更少）。

查询时的魔法：

1. 预先计算 q 的 m 个子向量到各自码本码字的距离表（lookup table）
2. 扫描候选时，不做 D 维点积，只做 m 次表查 + 加法

这会把每个候选的计算从 O(D) 降到 O(m)，并且内存读取从“读 D 个 float”变成“读 m 个 byte”。

代价是：这是近似距离。于是常见的组合拳变成：

- IVF 负责缩小范围
- PQ 负责极快打分
- 最后对少量候选做精确重排（读取原向量）

## 4. 过滤：为什么加了 filter 反而更慢？

很多人第一次用向量数据库都会踩到这个现象：不加过滤很快；一加 metadata filter，延迟抖动甚至变慢。

原因是过滤与 ANN 的基本矛盾：

- ANN 索引假设“空间连续”：邻近点在图里相连、在桶里集中
- 过滤条件可能把空间切成碎片：原本很近的点被过滤掉，索引引导你走向的区域变成“不可用”

工程上通常会提供两种过滤策略：

### 4.1 Pre‑filter：先过滤再检索（更准，可能更慢）

做法是先用倒排索引/位图/范围索引把可用 id 集合算出来（bitset），ANN 搜索过程中只接受满足条件的点。

好处：结果更干净，减少后处理。

坏处：ANN 扩展会频繁撞到“不可用点”，需要更大的 efSearch/nprobe 才能找齐 k 个有效结果，延迟抖动更明显。

### 4.2 Post‑filter：先检索再过滤（更快，可能缺结果）

先按向量相似性召回 M 个候选，再对候选做 filter，最后取 Top‑K。

好处：ANN 行为稳定，速度通常更好。

坏处：如果过滤太严格，M 里通过过滤的不足 k 个，你要么返回不足 k，要么扩大 M 重试。

因此很多系统会做“自适应过采样”：

- 初始召回 M = k × r（r 比如 20~200）
- 过滤后若不足 k，则增大 r 或增大 efSearch/nprobe 再跑一轮

黑盒点在于：你看到的是一个 k，但系统内部一直在为“找到 k 个有效点”做动态预算调整。

## 5. 重排：Top‑K 最后的“真实性审计”

当 ANN 召回给出候选 ids 后，系统往往会做一次（或多次）重排：

1. 拉取候选的原始向量（或更高精度的存储形式）
2. 用真正的 metric 算 exact distance / exact score
3. 对 exact score 做 Top‑K 选择

为什么需要重排？

- PQ/HNSW/IVF 都可能引入近似误差：候选集合对，但排名可能不准
- 过滤可能导致候选不足，需要扩大候选、再精排裁剪
- 分片合并时需要统一度量：每个分片返回的局部 Top‑K 只是局部最优

典型策略是“分片内多取一点，再在协调节点合并裁剪”：

- 每个分片取 k'（比如 k 的 2~10 倍）
- 协调节点合并成全局 Top‑K

这也是为什么你会看到一些系统的配置项叫：search_k、topk_per_shard、rerank_k 之类：它们控制的是“内部拿多少候选来换全局更稳的 Top‑K”。

## 6. Top‑K 是怎么“选”出来的：不是 sort，而是 heap

全量排序是 O(M log M)，但我们只要前 k。于是常用最大堆（size=k）维护当前 Top‑K：

- 扫描候选 (id, score)
- 堆未满就插入
- 堆满且 score 更好，就替换堆顶（最差的那个）

复杂度变成 O(M log k)，k 很小，所以非常划算。

在分布式场景下：

- 每个分片先做本地 Top‑k'
- 协调节点对多个有序列表做 k 路合并，或再次用堆裁剪到 k

## 7. 为什么同样的 q，每次返回略不一样？

向量检索的“不确定性”通常来自三个来源：

1. 近似：ANN 本身不保证绝对最优，尤其在参数偏小、过滤较严时更明显
2. 并发与段：写入和 compaction 会让可见 segments 变化，导致候选空间变化
3. 浮点与实现：不同硬件/指令路径（SIMD、GPU）可能带来极小的数值差异，临界点的排序会抖动

如果你追求更稳定的结果，通常需要：

- 增大 efSearch/nprobe 或启用更强的 rerank
- 固定随机入口（少见）或使用更稳定的索引构建参数
- 在业务侧做幂等与缓存（同 q 同 filter 的请求直接复用结果）

## 8. 把黑盒拆成可调旋钮：你到底该调哪些参数？

如果把整条链路简化成三个“旋钮”，它们分别对应三段成本：

1. 探索旋钮（ANN 探索预算）

- HNSW：efSearch
- IVF：nprobe

2. 候选旋钮（召回候选放大倍率）

- M = k × r，r 越大越准越慢

3. 精排旋钮（读取原向量的数量）

- rerank_k 越大，越能修正近似排序误差，但会增加内存/磁盘读取

经验上（不绑定任何具体产品）：

- 延迟优先：先降低探索预算与候选倍率，再看召回能否接受
- 召回优先：先开大探索预算（efSearch/nprobe），再考虑是否需要更大的 rerank
- 有过滤：优先考虑 post‑filter + 过采样；如果必须严格 pre‑filter，通常需要更大的探索预算

## 9. 最后用一句话总结黑盒密码

“q 向量进 → Top‑K 出”的真正秘密是：系统并不直接在全库上求 Top‑K，而是用 ANN 索引把搜索空间压缩到一个小候选集，然后在候选集上用更精确的计算与过滤把结果校正回来。你看到的 Top‑K 是一连串剪枝、近似和补偿之后的产物。

如果你愿意把这条链路的每一段都当成可解释的工程模块，那么向量数据库就不再是黑盒：它只是一个在“带宽、延迟、召回、更新成本”之间做权衡的系统。
