# 从人类文本到机器索引：分词器如何把一句话切成 Token ID

当用户输入一段文本时，大模型并不能直接“看懂”这些字符。分词器（Tokenizer）在这里充当了底层“翻译官”：它将用户输入的字符串片段，严格按照预先训练好的词表（Vocab）和合并规则（Merges），像拆解和查字典一样，最终转化为模型能够直接读取和计算的离散整数序列（Token IDs）。

## 1. 接收规范片段

分词器的核心拆解引擎接收的并非原始的一整段文本，而是一个**已经被初步切断的字符串数组**。
1. **黑盒输入**：比如原始文本 `"Hello world!"`，进入此阶段时，已经被预处理成了类似 `["Hello", " world", "!"]` 的数组（已清理乱码、统一编码）。
2. **边界隔离**：分词器会**逐个遍历**这个数组中的元素。片段内部会进行拆解与合并，但片段与片段之间互不干扰。这保证了 `"Hello"` 和 `" world"` 无论如何都不会被越界合并成一个 Token。

## 2. 规则匹配与拆解

将字符串片段拆解为模型可识别的 Token，本质上是一个**查表并循环合并**的过程（以 BPE 算法为例）。
1. **基础拆解**：比如片段 `"apple"`，首先会被打散成单个字符的数组 `['a', 'p', 'p', 'l', 'e']`。
2. **优先级合并**：引擎会查看相邻的字符对（如 `'a'` 和 `'p'`）。如果预训练好的 `Merges` 规则表中记录了 `'a' + 'p' -> 'ap'` 且优先级最高，它就会被合并。此时数组变成 `['ap', 'p', 'l', 'e']`。
3. **终止条件**：这是一个 `while` 循环，每次都挑 `Merges` 表中优先级最高的组合进行合并，直到数组里任何相邻的元素对都无法在表中找到记录为止。最终该片段的形态可能定格在 `['app', 'le']`。

## 3. ID 映射

完成字符串层面的拆解后，需要把字符串转换成机器能懂的数字。这一步对于程序员来说，就是一个简单的 **Hash Map 字典查表**操作。
1. **静态字典查表**：预训练阶段会生成一个庞大的 `Vocab` 字典（例如 `{"app": 1024, "le": 89, ...}`）。引擎将上一步得到的字符串数组，逐一去字典中取值（O(1) 复杂度）。
2. **转化为整数 ID**：`['app', 'le']` 会被直接映射成 `[1024, 89]`。这个数字 ID 本身不具备数学大小上的意义，它仅仅是一个**索引（Index）**，后续将被用来去 Embedding 矩阵里抽取对应的那一行向量。

## 4. OOV 兜底降级

面对词表未覆盖的生僻字、特殊 Emoji 或外语（即 OOV，Out of Vocabulary 问题），分词器必须保证计算链路不报错中断。
1. **字节级回退（Byte-Fallback）**：早期系统遇到未知符号会直接抛出 `<UNK>`（未知错误符），导致信息丢失。现代分词器遇到无法在 Vocab 中命中的字符时，会将其**降级拆解为计算机最底层的 UTF-8 字节**（如某个生僻字被拆成 `<0xE4>`, `<0xBD>`, `<0xA0>`）。
2. **信息保全（不报错）**：因为最初的词表里一定硬编码了 `0x00` 到 `0xFF` 这 256 个基础字节 Token。通过这种“降维打击”，分词器永远不会遇到无法解析的死胡同，只是会稍微增加这个生僻字符占据的 Token 数组长度。

## 5. 序列拼接与组装

所有独立片段分别完成上述映射与降级流程后，执行最终的上下文组装。
1. **展平数组（Flatten）**：各个片段分别转化出的数字数组，会按照原文的空间顺序拼接成一个完整的一维数组，类似 JavaScript 中的 `Array.prototype.flat()` 操作。
2. **交付下游**：最终产出的类似 `[15496, 31414, 296, 0]` 这样的整型数组，就是这段文本最底层的数字化表示。随后，它还会在首尾被插入一些特殊控制符（如代表开始的 `<BOS>`），最终正式送往 Embedding 层。
