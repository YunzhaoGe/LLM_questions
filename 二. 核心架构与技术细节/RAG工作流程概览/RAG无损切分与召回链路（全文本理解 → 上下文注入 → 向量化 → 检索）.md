# RAG 无损切分与召回链路（全文本理解 → 上下文注入 → 向量化 → 检索）

目标：从处理超长原始文档开始，串起下一代 RAG 解决“上下文丢失”的核心链路（重点拆解 Late Chunking 与 Contextual Retrieval 两种前沿方案）

---

## 1. 传统切分的痛点与破局思路（为什么需要新链路？）

1. 传统滑动窗口切分（Chunking）：按固定 Token 数（如 512）强行截断文档，最多加一点重叠（Overlap）（为了把长文档塞进 Embedding 模型的输入窗口）
2. 语义丢失与指代不明（痛点）：导致“他”、“该公司”等代词失去前文，或者核心前提条件被分到上一个 Chunk（这就使得向量检索时根本匹配不上 Query）
3. 破局核心（无损切分）：在把文本变成独立向量前，强迫 Chunk 先“看到”全局上下文（为了保留隐含的语义联系）

---

## 2. 路线一：Late Chunking 隐式上下文链路（Jina AI 方案）

目标：通过底层注意力机制让每个切片“感染”全文语境，走通从长文本处理到在线召回的全流程。

1. 全文本输入模型（Long-Context Encoding）：把多达 8K 甚至 1M Token 的完整文档一次性输入支持长上下文的 Embedding 模型（为了让底层 Transformer 的注意力机制跨越整个文档建立联系）
2. 提取 Token 级隐状态（Token-level Embeddings）：不急着输出单一的句子向量，而是提取每一个 Token 经过全局注意力计算后的特征向量（为了保留每个词在全局语境下的特定含义）
3. 边界划分与延迟池化（Late Pooling）：在拿到所有 Token 向量后，再根据标点或段落标记划定 Chunk 边界，对边界内的 Token 向量求平均（为了生成最终的 Chunk 向量，但此时的向量已经“自带全局视角”）
4. 向量库索引（Indexing）：把这些融合了全文语境的 Chunk 向量存入向量数据库（为了让后续能用相似度快速召回）
5. Query 的轻量级向量化（Asymmetric Embedding）：在线检索时，把用户的简短提问直接用常规方式转成向量，不需要走长窗口推理（为了保证提问时的极低延迟与算力节约）
6. 在线语义召回（Retrieval）：用轻量化的 Query 向量直接在向量库中做近邻搜索（因为 Chunk 向量已包含全局语境，即使 Query 问的是隐式指代也能精确命中）

---

## 3. 路线二：Contextual Retrieval 显式上下文链路（Anthropic 方案）

目标：利用大语言模型（LLM）直接把隐式上下文“写出来”并拼接给切片，走通从硬切分到多路召回的全流程。

1. 基础硬切分（Hard Chunking）：先用传统方法把长文档切碎（为了控制后续单次处理的文本粒度）
2. 开启大模型缓存（Prompt Caching）：把完整的长文档作为系统背景，写入 LLM 的上下文缓存中（为了在给成百上千个 Chunk 生成背景时，不重复消耗长文本的 API 计费与时间）
3. LLM 上下文生成（Context Generation）：基于已缓存的全文，把当前 Chunk 喂给大语言模型，让其生成 50-100 字的“该 Chunk 专属背景”（为了把隐式的代词、前提条件显式地写出来）
4. 文本拼接（Text Concatenation）：把“专属背景 + Chunk 原文”拼接成一段新文本（为了让后续的 Embedding 模型和词频引擎都能直接“看到”这些背景词）
5. 向量化与多路建库（Embedding & Indexing）：把拼接后的新文本输入 Embedding 模型得到向量存入向量库，同时强烈建议将其原文存入 BM25 关键词索引（为了兼顾语义匹配与精确词频匹配，显式上下文对 BM25 提升极大）
6. 在线混合检索（Hybrid Retrieval）：把用户 Query 转成向量，同时在 Vector 和 BM25 两路搜索，最后用 RRF（倒数秩融合）合并结果（为了兼顾“意思对”和“词对得上”）

---

## 4. 两种无损链路的工程取舍（成本与适用场景）

1. 离线建库成本（Indexing Cost）：Contextual Retrieval 会消耗极大的 LLM Token（整篇长文档要被重复输入 N 次去生成背景），而 Late Chunking 需要显存极大的长窗口 Embedding 模型（为了效果必须付出的基建代价）
2. 在线检索延迟（Latency）：两种路线在线检索时，速度与传统 RAG 几乎无异（因为复杂的上下文融合计算都转移到了离线建库阶段；这也是它们能在工业界落地的核心原因）
3. 落地选型建议：如果预算充足、追求极致准确率且极其依赖 BM25 关键词搜索，选 Contextual Retrieval；如果成本敏感、拥有长上下文 Embedding 模型部署能力，选 Late Chunking。
