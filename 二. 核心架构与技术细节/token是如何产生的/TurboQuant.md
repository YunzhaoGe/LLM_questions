# TurboQuant 压缩 KV Cache 的主线流程（写入缓存 → 主压缩 → 误差修正 → 注意力读取）

如果你已经知道 KV cache 是什么，那理解 TurboQuant 时，最重要的一点不是“它把 Key 和 Value 压得多小”，而是“它为什么敢在压得很小之后，还尽量不把注意力结果带偏”。因为 KV cache 不是普通存储数据，它后面还要继续参与 Query 和 Key 的匹配、参与 Value 的读取。也就是说，TurboQuant 不是单纯在做“把向量省空间”，而是在做“把会继续参与 attention 的向量压缩得足够小，同时尽量不破坏 attention 本身”。可以把这条链路理解成：先把原始 K/V 变成更适合低比特表示的形状，再用主要比特保存主体信息，然后用极少的额外信息修正最关键的误差，最后在真正做 attention 时，把这些压缩信息重新组织起来，让模型算出来的分数尽量接近原始结果。

---

## 1. 先明确目标：TurboQuant 压的不是普通向量，而是后面还要继续参与注意力的 K 和 V

1. 它压的对象是什么：在推理阶段，每个 token 产生的 Key 和 Value 会被持续写进 KV cache，后面新 token 来了之后，Query 要拿它们做匹配和读取。TurboQuant 处理的，就是这批会越积越多的 K/V。
2. 为什么它比普通量化更难：因为这些向量不是“存完就算了”，后面还要继续参与内积计算和加权汇总。压缩一旦改坏了它们的几何关系，attention 就会变形。
3. 所以真正要保住的是什么：不是“每个数都和原来一样”，而是“压缩后做出来的 attention score 和最终读取结果尽量别变”。也就是说，TurboQuant 保护的重点是**注意力行为**，而不只是向量外观。

---

## 2. 不是上来就硬量化：它会先把 K/V 变成更适合压缩的形状

1. 为什么不能直接低比特化：原始 K/V 的不同维度，分布往往不均匀，有些维度波动大，有些维度有离群值。如果直接逐维压缩，误差很容易集中在少数维度上。
2. TurboQuant 先做什么：它会先对输入向量做一次随机旋转。你可以把这理解成“先把原来不均匀的能量重新摊开”，让信息不要老挤在少数几个坐标上。
3. 这一步的意义：一旦分布被摊得更均匀，后面就更适合用简单、低成本的坐标级量化去压。也就是说，TurboQuant 不是靠更复杂的码本起家，而是先把问题整理成一个更容易压缩的问题。

---

## 3. 主体压缩：旋转后的 Key 会先被改写成低比特主码

1. 这一步到底发生了什么：经过随机旋转后，TurboQuant 不会继续保存完整的浮点向量，而是先把每一维映射到少量量化中心，再把这些中心写成更短的索引编码。也就是说，`Rk_i` 会先变成一份低比特的“主体表示”，必要时再从这份表示回填出一个粗粒度的主体重建向量 `Rk_i'`。
2. 这里“主体信息”具体指什么：它不是每一维都和原值一样，而是尽量保住这条 Key 的符号模式、幅度档位和大致几何关系。换句话说，模型先保住“这条向量大概长什么样、和谁更像”，而不是先保住“每个数字小数点后还差多少”。
3. 为什么这一步值钱：因为 KV cache 的大头开销，来自每个历史 token 都要长期保存完整的 K/V。只要这一层主体信息能先用极少 bit 表达出来，缓存体积就会立刻下降，后面才有空间再去处理更细的误差。

---

## 4. 主体重建参与打分时，问题会暴露在内积上

1. 一个很容易误解的点：很多人会觉得，只要主体重建向量 `Rk_i'` 和原始 `Rk_i` “看起来差不多”，attention 应该也差不多。这个说法并不完全对，因为 attention 真正用来比较的不是坐标外观，而是 Query 和 Key 的内积。
2. 偏差是怎么长出来的：内积本质上是在把每一维的乘积加总。于是，哪怕每一维只改了一点点，只要这些改动在乘积里没有互相抵消，而是朝着同一个方向累积，最后的 `Q·K` 分数就可能被系统性抬高或压低。
3. 为什么这很严重：注意力不是只看“误差大不大”，而是看“谁的分数更高”。一旦主体重建把两个候选 Key 的分数间隔改写了，模型面临的就不是“信息少一点”，而是“可能把注意力落到另一个位置上”。

---

## 5. 残差会被显式拆出来：因为它正好对应主体压缩没保住的那部分细节

1. 第一步压完之后还剩什么：会剩下一份“原始向量减去主体重建向量”的残差，也就是 `e_i = Rk_i - Rk_i'`。你可以把它理解成：主体压缩先保住了大轮廓，而残差就是那些没被主码本吃进去、但又真实存在的细节差异。
2. 为什么这一步必须单独成立：因为第四步已经说明，attention 最怕的不是“向量长得没那么像”，而是“内积被带偏”。而这部分内积偏差，恰恰就来自主体重建和原始向量之间那一点残差。
3. 所以 TurboQuant 的思路不是“主体压完就结束”，而是把残差当成一个正式对象单独拿出来。后面不是去把整条向量重新精确存一遍，而是继续处理这份残差对 attention 最敏感的那部分影响。

---

## 6. QJL 介入：它不追求重建残差本身，而是纠正残差对内积的影响

1. 这一步具体在做什么：TurboQuant 不会把残差再按原样存下来，而是对残差再做一次非常便宜的编码，公开材料里这部分叫 QJL。你可以先把它理解成：不是完整保存 `e_i`，而是保存一份足以参与纠偏的轻量表示。
2. 为什么这种做法有效：因为 attention 最敏感的是 `Q·K` 的结果，不是残差向量本身是不是被逐坐标准确恢复。只要这份轻量表示能把“残差把内积往哪个方向推了多少”补回来，最终打分就可能已经很接近原始结果。
3. 这一步的本质：不是再做一次普通压缩，也不是试图把第一阶段的误差彻底抹掉，而是在给主体压缩留下的偏差打一层便宜但有针对性的补丁。它保护的不是残差的外观，而是残差对 attention 的作用。

---

## 7. 真正写进 KV cache 的，是“主码 + 纠偏信息”这套组合表示

1. 压缩后缓存里到底放什么：从工程角度看，写进 KV cache 的不再是原始浮点 K/V，而是一套更紧凑的组合表示。对 Key 来说，至少包含两层：一层是主体压缩后的主码，另一层是后面用来纠偏的残差辅助信息。
2. 为什么这比“只做一种量化”更合理：如果只存主码，分数容易偏；如果又把所有残差完整存回去，压缩收益会被吃掉。TurboQuant 的做法相当于把信息分层：主码负责省空间，纠偏信息负责把 attention 拉回正确方向。
3. 所以它节省空间的来源不只是一点：既有低比特主码带来的直接收益，也有“只为关键偏差补少量信息”带来的间接收益。真正省下来的，不是某个局部技巧，而是整套缓存表示的总成本。

---

## 8. 读取阶段不是简单反量化：Query 会联合使用主码和纠偏信息

1. 新 token 到来时会发生什么：当前 token 产生 Query 后，需要和历史 KV cache 中的 Key 做匹配，并根据分数去读取 Value。也就是说，缓存里的压缩表示会再次进入 attention，而不是只被当成静态存档。
2. TurboQuant 下怎么做：它不是天真地先把所有压缩内容完整还原成原向量再照常计算，而是让主码和纠偏信息一起参与打分过程。前者先给出一个低成本的主体估计，后者再把最关键的内积偏差往回拉。
3. 这一步的关键理解：TurboQuant 的目标不是完美重建每个历史 Key，而是让 Query 和这些 Key 做出来的相关性尽量接近原始结果；对 Value 侧来说，目标也不是逐点复刻，而是让最后读出来的上下文表示不要明显变形。

---

## 9. 它为什么能做到“压得很狠，但效果不容易掉”

1. 第一层原因：随机旋转先把向量变得更均匀，降低了少数维度支配误差的风险，所以低比特主体压缩不容易一下子压坏结构。
2. 第二层原因：主体压缩负责保住“大轮廓”，QJL 负责修正“最影响内积的那点偏差”，两者分工明确，不是把所有任务都压给同一种量化器。
3. 第三层原因：它优化的目标更贴近 attention 的真实需求。也就是说，它不是只问“向量像不像原来”，而是进一步问“模型还会不会看向同样的位置、读出差不多的内容”。

---

## 10. 但“几乎无损”不等于“绝对零误差”：它保住的是下游行为，不是逐点完美重建

1. 要先把这个口径说准：TurboQuant 的亮点是，在极低 bit 下仍然尽量保持下游任务质量和注意力行为，而不是承诺每个 K/V 坐标都能精确恢复到原值。
2. 这两者有什么区别：前者强调“模型最终做事的结果别变”，后者强调“每个数字必须和原来一样”。在推理系统里，真正重要的通常是前者。
3. 所以正确的理解应该是：TurboQuant 压缩 KV cache 的成功，不在于它把向量还原得像显微镜下完全一致，而在于它把 attention 需要的几何关系和统计关系保得足够好。

---

## 11. 这套流程为什么特别适合在线 KV cache，而不是只适合离线压缩

1. KV cache 有什么特殊性：它是推理过程中一边生成一边写入的，不能先离线收集一大堆数据，再慢慢训练一个专属码本。
2. TurboQuant 为什么适合这个场景：它强调的是在线、数据无关的压缩思路。也就是说，不需要为了某个具体任务或具体数据集先准备一整套昂贵的离线量化流程。
3. 这意味着什么：它更符合真实推理服务的工作方式。来了一个新 token，就能立即把新的 K/V 压进缓存；之后新 Query 来了，也能直接在这种压缩表示上继续做 attention。

---

## 12. 一句话复盘

如果按主线流程来理解，TurboQuant 压缩 KV cache 不是“把历史 K/V 粗暴地改成低精度数字”这么简单。更准确的流程是：先把每个新产生的 K/V 通过随机旋转整理成更适合压缩的形状；再用主要比特预算把向量的主体信息压下来，显著缩小缓存体积；接着把第一阶段留下的残差单独拿出来，用 QJL 这类极轻量方式编码成纠偏信息；真正写入 KV cache 的，是“主体压缩结果 + 误差修正信息”这一套紧凑表示；等后面 Query 来做 attention 时，再利用这两部分信息一起参与匹配和读取，让内积关系、注意力排序和最终上下文汇总尽量接近原始结果。也就是说，TurboQuant 真正保住的，不是每一个浮点数原封不动，而是“模型仍然大致看向同样的位置，读出大致相同的内容”。
