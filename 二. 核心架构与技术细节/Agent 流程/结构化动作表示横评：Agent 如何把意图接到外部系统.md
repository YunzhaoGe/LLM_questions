# 结构化动作表示横评：Agent 如何把意图接到外部系统

如果把 Agent 看成“会调用外部能力的 LLM”，那真正决定它能不能落地的，往往不是模型本身，而是模型和外部系统之间那一层动作表示。用户给出一个任务，LLM 先输出 token，但外部系统并不理解 token。它理解的是另一套东西：函数名和参数、命令及参数、浏览器动作序列、工作流节点、事件消息、界面点击目标。所谓“结构化动作表示”，本质上就是把语言意图改写成外部系统可以消费的动作对象。

这一层之所以值得单独做横评，是因为很多讨论把不同东西混在一起了。有人在比较 function calling 和 MCP，有人在比较 JSON 工具调用和 computer use，还有人在比较 CLI Agent 和 GUI Agent。实际上，它们并不都在同一层：有的是能力暴露协议，有的是动作表示格式，有的是执行面本身。要把这件事讲清楚，最好把焦点收缩到一个问题：**当 LLM 决定“下一步要做什么”之后，它到底把这个决定组织成什么对象，再交给外部系统。**

这篇文章不再沿着某条具体任务主线展开，而是直接把焦点固定在“动作对象本身”。也就是说，后文关心的不是某个案例里先查日历还是先发邮件，而是每一种结构化动作表示在抽象层面到底长什么样、和什么执行世界天然匹配、为什么会稳定或脆弱。

---

## 1. 先定比较对象：我们比较的不是工具，而是“动作如何被表示出来”

1. 这一层真正比较的对象是什么：不是模型能力大小，也不是工具数量多少，而是“动作对象长什么样”。也就是说，我们比较的是外部系统最终接收到的那一层表示：它到底是函数调用、命令对象、查询对象、界面动作序列，还是工作流节点。
2. 为什么这一步要先说清：因为同样叫“Agent 调外部能力”，底下可能完全不是一种东西。一个系统可能把动作写成函数调用，另一个系统写成 CLI 命令，还有一个系统写成 GUI 点击计划。它们调用的世界不同，失败方式也不同。
3. 所以后文的横评标准会围绕同一组问题展开：这种动作表示到底是什么、它天然适合接哪类外部系统、它最典型的应用场景是什么、它的优点是什么、它的硬伤又是什么。

---

## 2. 第一类：函数调用 / JSON 工具调用，是今天最标准的结构化动作表示

1. 这类方式是什么：它把动作表示成一个“工具名 + 参数”的调用对象。它最核心的特征不是用了 JSON，而是动作已经被压缩成一个离散的、带边界的函数语义：调用哪个能力、传什么参数、预期返回什么类型的结果。JSON 只是今天最常见的承载格式，不是本质。
2. 放到“销售助手准备客户会前简报”这个场景里，它会长成几个彼此独立的调用对象：第一个对象的核心是“调用日历工具”，参数是日期与参会人；第二个对象的核心是“调用 CRM 工具”，参数是客户 ID；第三个对象的核心是“调用邮件草稿工具”，参数是收件人、主题和摘要要点。读者看到这种模样，会直觉地意识到：每一步都是一个边界清楚的函数动作。
3. 它的优缺点很鲜明：优点是结构稳定、参数可校验、权限容易控、失败结果可解析；缺点是强依赖已有工具定义，动作空间必须提前设计好。它特别适合“世界已经准备好语义化接口”的场景，不适合没有 API 的桌面和网页环境。

---

## 3. 第二类：Schema 驱动的多步动作对象，比单次函数调用更像真正的任务单

1. 这类方式是什么：它不是只表示“做一个动作”，而是表示“一组有组织关系的动作”。这个对象里往往同时包含动作序列、依赖关系、成功条件、失败回退、审批点或阶段状态。它本质上已经不只是工具调用，而是在表示一个可执行的小流程。
2. 放到“新员工入职自动化”这个场景里，它不会长成四五个散乱调用，而会长成一份有骨架的动作对象：里面先列出步骤顺序，再标出“创建邮箱”完成后才能“开通权限”，“设备登记”失败时要转人工，“直属经理确认”是审批点。读者看到这种模样，会直觉地意识到：这里表达的不是单个动作，而是一份小流程。
3. 它的优缺点在于：优点是更适合多步任务、天然便于审计、便于插入审批节点；缺点是表达成本更高、生成难度更高，而且一旦 schema 设计得太重，模型很容易在“填结构”上消耗大量上下文。它适合链路长、可追责要求高的系统，不适合极轻量的单步工具调用。

---

## 4. 第三类：CLI / 命令对象，是最通用也最危险的一类动作表示

1. 这类方式是什么：LLM 最终交给运行时的不是业务函数，而是一个进程级动作对象。它表达的核心意思是：请调用操作系统中的某个可执行工具，并把参数传进去。它面对的不是业务语义世界，而是 shell、进程、参数、标准输出和退出状态。
2. 放到“本地代码库修复和验证”这个场景里，它会长成一条条命令对象：先是“运行测试工具”的命令，参数是目标测试集；接着是“搜索代码”的命令，参数是关键词和目录；最后是“构建或格式化”的命令，参数是目标模块。读者看到这种模样，会直觉地意识到：这里不是在调业务函数，而是在驱动操作系统里的工具链。
3. 它的优缺点同样非常极端：优点是通用、扩展性强、能快速接入已有工具链；缺点是语义不稳定、输出格式松散、环境依赖重、安全风险大。和函数调用相比，CLI 不是在调用“明确语义动作”，而是在调用“通用工具箱”，所以自由度更大，但也更难控。

---

## 5. 第四类：SQL / 查询对象，本质上是在直接操作结构化状态

1. 这类方式是什么：动作不是“调用一个服务”，而是直接表达对结构化状态的操作。它可以是查询、过滤、聚合，也可以是更新、插入和删除。本质上，它把外部世界看成一份可直接读写的数据状态，而不是一组高层业务动作。
2. 放到“运营想找出过去 7 天下单下降最明显的客户”这个场景里，它会长成一个查询对象：先定义时间范围，再指定客户集合，再选择订单金额和订单次数这类字段，最后声明按降幅排序。读者看到这种模样，会直觉地意识到：这里表达的不是操作界面，而是直接操作一份结构化状态。
3. 它的优缺点是：优点是状态清晰、结果结构化、适合分析和查询类任务；缺点是动作语义太贴近存储层，不适合表达真正的业务副作用。换句话说，SQL 很适合“读世界”和“改数据”，但不适合“操作界面”或者“协调多个外部系统”。

---

## 6. 第五类：GUI / 浏览器动作序列，是没有 API 时最强的兜底表示

1. 这类方式是什么：它把动作表示成一串面向界面的操作意图，而不是面向系统语义的调用。最常见的原语是点击、输入、滚动、等待、聚焦窗口、选择元素、确认按钮。它的本质不是“调用某个能力”，而是“在已有界面上完成某个动作”。
2. 放到“在没有开放 API 的老旧财务后台里下载月报”这个场景里，它会长成一串界面动作：先聚焦浏览器窗口，再点击“报表中心”，再选择月份下拉框，再点击下载按钮，必要时还要等待页面刷新。读者看到这种模样，会直觉地意识到：这里没有稳定的业务字段，动作只能贴着当前界面往下走。
3. 它的优缺点是：优点是通用性极强，几乎任何有人类界面的系统都能接；缺点是脆弱、慢、依赖观察、很容易被界面变化打断。和函数调用相比，它不是在调用系统提供的“语义动作”，而是在模拟人类完成动作，所以灵活，但不稳定。

---

## 7. 第六类：Workflow / DAG / 状态机节点，适合把动作表示成“推进系统状态”

1. 这类方式是什么：Agent 输出的不是单个操作，而是一个“状态推进对象”。它的作用不是直接完成动作，而是让系统进入下一个流程节点、激活下一段 DAG，或者把任务推进到某个状态机阶段。它关注的是系统状态迁移，而不是单步操作细节。
2. 放到“合同审批”这个场景里，它会长成一个状态推进对象：当前阶段是“法务审核”，下一阶段是“财务会签”，进入下一阶段的条件是材料齐全且法务通过，如果失败则回退到“补材料”。读者看到这种模样，会直觉地意识到：这里表达的是任务在流程图里向前走，而不是某个按钮被点了一次。
3. 它的优缺点在于：优点是可靠、便于治理、适合多角色协作；缺点是灵活性不高，动作粒度往往比较粗，不适合那些需要即时探索或临场决策的场景。它特别像“把 Agent 的动作交给了流程引擎”，而不是交给单个工具。

---

## 8. 第七类：事件 / 消息对象，更适合异步世界，而不是即时交互世界

1. 这类方式是什么：动作被表示成一个事件或消息对象，然后交给下游系统异步消费。它不承诺“现在立刻做完”，而是在表达“有一件事应该被后续系统处理”。这类对象通常更像投递，而不像调用。
2. 放到“批量生成客户周报”这个场景里，它会长成一个消息对象：事件名是“开始生成周报”，负载里带着客户批次、时间范围、模板版本和回调地址。消息一旦发出，下游系统再异步拉数据、渲染模板、存盘并回传完成通知。读者看到这种模样，会直觉地意识到：这里不是立即执行，而是在可靠投递一个异步任务。
3. 它的优缺点是：优点是松耦合、易扩展、适合高吞吐和异步处理；缺点是反馈延迟大、调试复杂、对人类来说不够直观。它更适合“把任务交出去”，不适合“边看边点边决策”。

---

## 9. 第八类：领域专用动作 DSL，优点是高约束，缺点是通用性差

1. 这类方式是什么：系统不让模型直接输出通用动作，而是提供一套专门为某个行业或任务域设计的动作语言。它的本质是先把“允许存在的动作集合”限定死，再让模型只在这组原语里表达意图。它比 JSON tool call 更窄，比 GUI/CLI 更受控。
2. 放到“保险理赔初审”这个场景里，它会长成一套受限原语的组合：先执行“读取投保记录”，再执行“检查事故材料”，如果命中异常规则就只能输出“转人工复核”，否则才能输出“初审通过”或“初审拒绝”。读者看到这种模样，会直觉地意识到：这里动作不是开放生成的，而是在一套预先定义好的合法语言里拼出来的。
3. 它的优缺点是：优点是安全、可控、校验成本低、便于评测；缺点是扩展性弱，一旦业务变化，就得重新设计动作语言。它适合“目标稳定”的系统，不适合开放世界任务。

---

## 10. 如果把这些方式放在一张横评图里，真正该比的不是花样，而是四个维度

1. 第一维是语义稳定性：函数调用、SQL、领域 DSL 通常最高，因为动作边界清晰；CLI 和 GUI 通常更低，因为它们更多依赖运行环境和外部工具行为。越靠近结构化接口，越容易稳定。
2. 第二维是世界覆盖度：GUI 和 CLI 覆盖面最广，因为它们能接入大量没有 API 的系统；函数调用和工作流节点覆盖面较窄，但一旦世界已经结构化，它们的成功率通常更高。也就是说，覆盖度越高，往往意味着约束越弱。
3. 第三维是治理与审计成本：Schema 工作流、领域 DSL、函数调用最容易审计；CLI 和 GUI 最难，因为它们的动作语义更散。第四维是开发成本：GUI 看似“什么都能接”，但维护很重；DSL 最安全，但前期设计最贵。横评真正要比较的，应该是这四个维度的权衡，而不是简单问“哪种最先进”。

---

## 11. 一个最容易误解的点：MCP 不是动作表示本身，而是能力暴露层

1. 很多人会把 MCP 放进“结构化动作表示”的并列列表里，这个说法不完全对。MCP 更像是能力如何被挂出来、被发现、被描述的协议层，而不是动作最终长成什么样的表示层。
2. MCP 能做的是把资源、工具、约束和可调用能力暴露给模型和运行时；但真正发出去的动作，仍然可能是函数调用对象、资源读取请求、甚至后续被翻译成 GUI 操作序列。
3. 所以如果要做横评，MCP 更适合作为“这些动作表示如何获得能力上下文”的背景层，而不适合和 JSON 调用、CLI、GUI 放在同一行直接比较。

---

## 12. 一句话复盘

如果只盯着“LLM 输出了什么”，很容易把所有 Agent 都看成一种东西；但如果把焦点放到“它用什么结构化动作表示去衔接外部系统”，差异就会一下子清楚起来。函数调用最标准，适合结构化世界；Schema 和工作流对象更适合长链任务和审计；CLI 最通用，但风险也最大；SQL 直接作用于结构化状态；GUI 是没有 API 时最强的兜底方案；事件消息适合异步协作；领域 DSL 最安全，但最不通用。也就是说，真正该问的不是“哪一种动作表示最好”，而是“你的外部世界到底长什么样，你希望 Agent 以多高的稳定性、覆盖度和治理成本去接它”。

## 高密度横评表：不同结构化动作表示到底长什么样
