# 从 token 到 action：Computer use 分镜版与执行面关系图

近期 Agent 的热点，不是模型又多背了多少知识，而是它开始接到真实执行面：会读屏、会点按钮、会改文件、会产出草稿，必要时还会等你确认再执行关键动作。表面上看，这像是“模型学会了用电脑”。但如果把链路拆开看，真正发生的事其实更具体：LLM 只负责生成 token，运行时把这些 token 收束成动作候选，再把动作候选接到 GUI、CLI、API、文件系统、权限层这些真实执行面上。

如果只讲“模型如何接外部执行面”，读者容易看到的是抽象架构；如果只讲“信息图怎么画”，又容易丢掉底层接口的位置。这篇文章把两件事合在一起：一方面沿着同一个 Computer use 案例，把对象状态变化拆成连续分镜；另一方面在每个分镜里明确指出，MCP、API、CLI、GUI、文件系统、权限层分别站在链路的哪一段。

本文固定一个 mini-case。任务单记作 T0：读取下周一与客户 A 的会议安排；搜索客户 A 最近一周的公开新闻；把三条最重要的结论写进共享盘中的《客户A-周会.pptx》；最后生成一封发给销售团队的邮件草稿。后文始终追踪同一组对象：任务单 T0、会话状态 S、动作候选 A、能力目录 C、观察包 O、执行环境 E，以及最终落地到外部系统里的文件和邮件草稿。

## 一、第一张图要先画任务单，而不是画工具

Computer use 的起点不是工具调用，而是任务单 T0 从自然语言变成可执行状态。

1. 这一章的主对象是任务单 T0。它的前态是一段自然语言请求，后态是会话状态 S0。S0 至少包含四类字段：目标列表、当前上下文、可用能力、风险边界。对 T0 来说，这意味着系统已经知道它要碰日历、浏览器、共享盘和邮件客户端，也知道哪些动作只能生成草稿，哪些动作需要确认。
2. 这一步必须先发生，因为后面的每个执行面都只会看到局部动作，不会看到全局目标。GUI 只知道当前该点什么，API 只知道当前要读什么数据，文件系统只知道当前要打开什么文件。只有把 T0 先压成 S0，后面的局部动作才不会失去主线。
3. 这一章进入下一步的接口是“动作空间初始化”。信息图如果要画，第一张图应当是 T0 在左，S0 在右，中间标出四个槽位：目标、上下文、能力、风险边界。此时还不需要画浏览器，因为世界尚未被真正触碰。

## 二、第二张图要画能力目录，因为模型先知道有什么把手可用

在 token 变成动作之前，系统先要把“世界上有哪些可操作把手”暴露出来。

1. 这一章的主对象是能力目录 C0。它的前态是运行时内部已经接好的各类接口，后态是模型上下文中一组可被引用的动作原语。这里要区分几层：MCP 位于能力暴露层，负责把资源、工具、约束以统一方式挂出来；API 位于结构化执行层，适合读取日历或发起搜索请求；CLI 位于进程执行层，适合调起本地脚本或工具链；GUI 位于环境执行层，适合操作没有 API 的桌面和网页界面。
2. 这一步必要，因为如果没有 C0，模型输出的 token 只能停留在“我想去搜索新闻”这种语言层直觉，无法知道它应该走 API、CLI 还是 GUI。尤其在近期的 Computer use 热点里，系统越来越强调显式能力边界：模型不是凭空会用电脑，而是在一份受控的能力清单中选动作。
3. 进入下一步的接口是“动作候选生成”。这一章如果画信息图，最合适的是把 C0 画成一排执行面插槽：MCP 在最上层做暴露总线，下面分出 API、CLI、GUI、Filesystem、Permission 五条支路，表示它们不是同一层对象，但都会在后续被接入 A 的路由过程。

## 三、第三张图要画动作候选，因为 token 先变成提议，再变成执行

LLM 的 token 流不会直接落到鼠标事件上，它会先被收束成动作候选 A1。

1. 这一章的主对象是动作候选 A1。它的前态是模型连续输出的 token，后态是可解析的动作表示。对 T0 来说，A1 可能是“先读取下周一会议安排，再搜索客户 A 新闻，再定位并编辑《客户A-周会.pptx》”。这里的关键不是动作已经执行，而是动作已经从文本叙述变成可验证的结构。
2. 这一步必要，因为 token 本身没有执行语义。模型可能输出“打开浏览器并修改 PPT”，但运行时必须进一步判断：这里的“打开”是 GUI 动作，还是通过文件系统直接载入文档；“修改 PPT”是通过 Office 插件、GUI 操作，还是走某个本地脚本。没有动作候选层，token 会把所有执行面混成一句自然语言。
3. 进入下一步的接口是“路由决策”。信息图在这一格里应当画两层：上层是一束 token 流，下层是一个被规范化的动作卡片 A1。卡片上最好明确标记三件事：目标对象、预期执行面、成功判据。这样下一格才能接“为什么这一步走 API，不走 GUI”。

## 四、第四张图要画路由器，因为同一个目标会落到不同执行面

动作候选进入运行时后，不是统一执行，而是先做执行面选择。

1. 这一章的主对象是路由决策 R1。它的前态是抽象动作卡片 A1，后态是带执行面标签的动作队列。对 T0 而言，“读取下周一会议安排”更适合走 API，因为日历数据天然结构化；“搜索客户 A 最近一周公开新闻”可能走浏览器或搜索 API；“编辑《客户A-周会.pptx》”可能走 GUI 或 Office 插件；“生成邮件草稿”可能走邮件客户端 API，也可能走桌面 GUI。
2. 这一步必要，因为不同执行面的代价完全不同。API 最稳定，返回的是结构化状态；CLI 通用但容易受环境影响；GUI 最接近人类操作面，也最脆弱；文件系统适合直接改持久状态，但不等于完成业务动作。把这些路径分开，是为了避免模型把所有问题都丢给 Computer use 的 GUI 能力。
3. 进入下一步的接口是“观察请求”。这章的图最适合做一张分流图：A1 从中间进入，向右分成 API、CLI、GUI、Filesystem 四条支线，顶部再画 Permission 作为横跨所有支路的拦截层。这样后续每条路返回的 O，才不会被误看成同一种观察。

## 五、第五张图要画观察包，因为 GUI 不是调用函数，而是在环境里找落点

当动作被路由到 GUI 或 Browser 时，系统必须先拿到观察包 O1。

1. 这一章的主对象是观察包 O1。它的前态是执行环境 E0 的真实界面状态，后态是运行时返回给模型的一组中间表示，常见包括截图、窗口树、焦点位置、DOM 片段、按钮标签和错误提示。对 T0 来说，O1 可能显示浏览器已经打开、搜索框有焦点、共享盘目录里已有《客户A-周会.pptx》。
2. 这一步必要，因为 GUI 执行面不像 API 那样自带稳定字段。模型想点“搜索”，前提是它先看到搜索框在哪里、当前窗口是不是正确应用、文件是否已经打开。没有 O1，动作就没有落点，只能停留在“看上去合理”的语言计划。
3. 进入下一步的接口是“界面绑定动作”。信息图在这一格里应当把 GUI 和 API 区分开画：API 支路只需要结构化响应框，GUI 支路则需要一张界面截图旁边挂一层标注框，标出按钮、输入框、窗口标题和当前焦点，强调这是面向环境的观察，不是面向函数的返回值。

## 六、第六张图要画原子操作，因为 Computer use 真正碰世界时只会做很小的动作

一旦 O1 确认了落点，动作候选就会被压缩成执行环境里的原子事件序列。

1. 这一章的主对象是已执行动作 A2。它的前态是“面向某个界面目标的动作表示”，后态是执行环境 E1 中的一组原子操作，例如切到前台窗口、移动指针、点击输入框、输入关键词、等待页面刷新、双击共享盘中的演示文件。对 T0 来说，系统不是真的执行了“搜索客户 A 新闻”这句大动作，而是执行了若干个连续的小动作。
2. 这一步必要，因为 GUI 世界只能理解很小的事件，不能理解高层业务语义。API 可以直接接“读取日历事件”，CLI 可以直接接“运行某个工具”，但 GUI 只能接受点击、输入、滚动、等待这些原子操作。Computer use 的核心难点，就在于高层意图与低层动作之间的这道落差。
3. 进入下一步的接口是“环境状态更新”。这一章画图时，最适合把 A2 拆成时间轴：点击前、点击后、页面刷新后。这样信息图读者会清楚看到，token 没有直接变成“完成搜索”，而是变成了一串作用于 E 的局部状态变化。

## 七、第七张图要画回传观测，因为不同执行面返回的是不同类型的证据

Agent 之所以像在持续行动，不是因为一次想得很远，而是因为每走一步都拿到新的 O。

1. 这一章的主对象是回传观测 O2。它的前态是各执行面完成当前动作后的结果，后态是会被重新编码进上下文的观测片段。API 返回的是结构化字段，CLI 返回的是进程输出和退出状态，Filesystem 返回的是文件内容或版本变化，GUI 返回的是新截图和界面树。对 T0 来说，O2 可能包含三条新闻标题、一页已修改的 PPT 页面，以及一封未发送的邮件草稿。
2. 这一步必要，因为 LLM 只能继续消费 token，不能直接消费“刚刚点击成功”“文件句柄仍然有效”这种机器状态。运行时必须把不同执行面返回的异构结果重新压成统一的可读上下文，否则下一轮推理无法在同一尺度上比较“新闻已抓齐没有”“文档写入是否成功”“邮件草稿是否生成”。
3. 进入下一步的接口是“状态回写”。如果这张图要画得像分镜，最关键的是把四种回传结果并列展示：API 响应框、CLI 输出框、文件 diff 卡片、GUI 截图卡片。这样读者一眼就能看出，不同执行面共用的是闭环逻辑，不共用的是观测形态。

## 八、第八张图要画权限层，因为真正危险的不是不会做，而是做错了也真的会生效

当动作将要触碰真实后果时，系统必须在动作和执行面之间插入权限层。

1. 这一章的主对象是待确认动作 A3。它的前态是“模型建议执行的下一步”，后态是“经过策略层审查后的允许动作、拒绝动作或待确认动作”。在 T0 中，读取新闻和生成邮件草稿通常可以自动完成，但真正发送邮件、覆盖正式文档或提交外部表单，往往需要用户确认。
2. 这一步必要，因为一旦 Agent 接到了真实执行面，错误就不再停留在语言层。它可能改坏正式文件，也可能把错误信息发给真实收件人。近期热点里的 Computer use 之所以把确认机制放得很重，原因就在这里：系统宁可多停一次，也不能让 token 直接穿透到高风险后果。
3. 进入下一步的接口是“终态提交或等待人工决策”。信息图里，这一格最好画成一条横跨所有执行面的策略带：API、CLI、GUI、Filesystem 都要经过它，只是风险等级不同。这样读者会明白 Permission 不是另一种执行面，而是贯穿整个链路的控制层。

## 九、第九张图要画终态，因为一条链路好不好，最终看外部世界留下了什么痕迹

当 T0 完成时，系统优化的不是“说得像会做事”，而是“外部状态真的被正确改变”。

1. 这一章的主对象是任务单 T0 的终态。它的前态是语言里的开放目标，后态是外部世界里的已落地状态：日历信息已读取，三条新闻已进入摘要列表，《客户A-周会.pptx》已更新，销售团队邮件草稿已创建。此时真正被保住的，是动作与外部对象之间的一致关系。
2. 这一步必要，因为只有看终态，才能反推整条链路是不是对齐。如果最终文件没更新、邮件草稿没生成、新闻抓取错位，那么前面的 token、动作候选、观察包、路由和原子操作，无论写得多漂亮，都只是没有落地的中间过程。
3. 这一步进入下一轮的接口是“可审计回放”。最适合的信息图收尾方式，不是再画一个大而全的架构图，而是把 T0 的完整链路压成一条可回放的状态线：T0 → S0 → C0 → A1 → R1 → O1 → A2 → O2 → A3 → T0 终态。这样读者会直观看到，MCP、API、CLI、GUI、Filesystem、Permission 并不是并列的概念词，而是在不同位置接力的不同层对象。

## 结尾

把“信息图分镜版”和“执行面关系图”合在一起看，Computer use 的主线就会变得很清楚：LLM 从头到尾只生成 token；运行时先把 token 收束成动作候选，再根据能力目录和当前观察决定这一步该走哪个执行面；真正碰世界时，GUI 负责操作界面，API 负责读取或写入结构化系统，CLI 负责驱动进程工具，文件系统负责修改持久状态，而权限层负责阻止高风险动作直接落地。MCP 不在这些执行面之中，它更像上层的能力暴露总线，把哪些把手可用、边界在哪里、输入输出长什么样，统一挂给模型和运行时。

- 如果要画信息图，主线应当画“对象状态变化”，不是画概念清单。
- 如果要理解接口分层，先分清暴露层、路由层、执行层、权限层，不要把 MCP 和 GUI 放在同一层比较。
- 如果要判断一个 Agent 是否真的接到了世界，最终看的是外部对象是否被正确改写，而不是模型语言是否足够像人在思考。
