**整体思路**

- 不再铺垫“为什么需要 Query Rewrite”，默认读者已经带着明确问题进来
- 直接从“这一步在 pipeline 里干什么、怎么做、产出什么”切入
- 每张图都保留 4 层信息：结论层、工程动作层、案例层、落地边界层
- 文案重点从“认知解释”切到“实现决策”，尽量回答程序员真正会问的问题
- 整体叙事固定为：输入问题 → 七步 rewrite → 输出结构化检索计划 → 上线 checklist

- 第 1 张：封面 + 七步总览

**分镜脚本**

- **第 1 张｜封面 + 七步总览**
  - 标题：RAG 里的 Query Rewrite，不是改写一句话，而是生成一套检索计划
  - 副标题：面向准备落地和优化自己 RAG 系统的程序员
  - 页面核心：
    - 输入不是 query string，而是用户问题
    - 中间不是一次润色，而是七步 rewrite pipeline
    - 输出不是一句改写后的话，而是 query plan
  - 七步总览：
    - 问题分型
    - 检索面拆分
    - 术语映射
    - query set 生成
    - route 绑定
    - 二次 rewrite
    - 结构化输出
  - 案例问题：
    - 为什么新用户可以领取新人优惠券，老用户却提示活动资格校验失败？
  - 画面建议：
    - 上半部分放一条从 User Question 到 Retrieval Plan 的流水线
    - 下半部分放案例问题和最终输出卡片缩略图
  - 页面目的：
    - 让读者一眼知道这套图是“怎么做”的，不是“为什么要做”的

- **第 2 张｜Step 1：判断是否进入 Rewrite**
  - 标题：先做问题分型，不是所有问题都值得走多步 rewrite
  - 这一步要解决什么：
    - 判断是否值得付出多 query、多路由、多轮召回的额外成本
  - 推荐输入：
    - 原始问题
    - 会话上下文
    - 可选的用户画像、业务线、时间条件
  - 核心判断信号：
    - 是否有多个对象或多个状态要比较
    - 是否同时出现动作、异常、原因追问
    - 是否可能跨多个知识源取证
    - 是否有明显时间变化、版本变化、权限变化
  - 最务实的第一版实现：
    - 先用规则表做粗分型
    - 再让 LLM 只输出少量类型标签，不直接生成 query
    - 标签尽量控制在 5 到 8 类，不要一开始就做得太细
  - 本案例判断结果：
    - 类型：对比型资格排查
    - 复杂度：高
    - 处理策略：进入多 query rewrite
  - 工程边界：
    - 如果是定义类、位置类、单规则类问题，直接单 query 检索通常更稳
    - 分型做错的代价很大，因为它会影响后面整条 retrieval path
  - 页面输出：
    - task_type
    - complexity
    - should_rewrite
  - 画面建议：
    - 左边是输入问题
    - 中间是小型判定表
    - 右边是分型结果卡片

- **第 3 张｜Step 2：拆检索面，不按句子拆，按召回目标拆**
  - 标题：一句自然语言，先拆成多个会影响召回的检索面
  - 这一步要解决什么：
    - 避免 embedding 把多个目标平均掉，导致每一层都沾一点但都不够深
  - 推荐拆分维度：
    - 对象维度：谁和谁
    - 行为维度：在哪个动作里
    - 现象维度：用户看到了什么
    - 任务维度：系统要回答原因、差异、定义还是变更
    - 证据维度：答案可能在哪类文档里
  - 值不值得拆的判断标准：
    - 去掉它之后，Top-K 是否明显换了一批结果
    - 去掉它之后，route 是否可能改向另一类知识源
  - 本案例拆法：
    - 对象面：新用户 vs 老用户
    - 行为面：领取新人优惠券
    - 现象面：活动资格校验失败
    - 任务面：查差异 + 查原因
    - 证据面：活动规则、资格配置、历史工单、发布说明
  - 最务实的第一版实现：
    - 可以先做 schema 抽取，让模型只填对象、动作、现象、任务四个槽位
    - 不要一开始就放开让模型自由发挥，否则字段漂移会很严重
  - 工程边界：
    - 不是拆得越细越好
    - 拆太碎会直接把 query set 做爆，召回成本和噪声都会上升
  - 页面输出：
    - retrieval_facets[]
    - evidence_types[]
  - 画面建议：
    - 中心是原问题
    - 四周展开成 5 个 facet 卡片
    - 每个 facet 下标注“是否改变召回范围 / 是否改变路由”

- **第 4 张｜Step 3：把用户语言映射成知识库语言**
  - 标题：rewrite 的第一层价值，是会说知识库里的话
  - 这一步要解决什么：
    - 用户说法和文档术语不一致时，纯向量检索很容易只打中表面相关 chunk
  - 推荐做法：
    - 保留原始用户词
    - 加一层同义扩展
    - 再加一层领域术语扩展
    - 最后给每类词打来源标签，避免后面混用
  - 本案例映射：
    - 用户词：领取新人优惠券 / 活动资格校验失败 / 新用户 / 老用户
    - 同义扩展：活动领取 / 优惠券发放 / 资格失败 / 参与校验失败
    - 领域术语：用户分群 / 资格规则引擎 / 活动参与门槛 / 首单用户标签
  - 适合积累的资产：
    - 用户说法 ↔ 正式术语词表
    - 别名表、旧名表、内部简称表
    - 领域映射表
  - 最务实的第一版实现：
    - 先做人审过的高频词表
    - 再用 LLM 做受控扩展，不要让它无限发散
  - 工程边界：
    - 映射是扩展，不是替换
    - 术语扩太多会把召回面放得过宽，导致 rerank 压力明显上升
  - 页面输出：
    - surface_terms[]
    - canonical_terms[]
    - domain_terms[]
  - 画面建议：
    - 左列用户词
    - 中列同义扩展
    - 右列知识库正式术语
    - 箭头上标“保留原词，不做替换”

- **第 5 张｜Step 4：从单条 query 变成 query set**
  - 标题：复杂问题不要硬塞进一条 query，要给不同检索任务不同 query
  - 这一步要解决什么：
    - 把“找原因、找差异、找变更”拆开，避免一个 query 同时承担三件事
  - 推荐 query 角色：
    - 主查询：直击用户核心问题
    - 对比查询：比较两个对象或两个状态
    - 变更查询：查最近规则、配置、版本是否变化
    - 校验查询：验证系统已经怀疑的某个具体原因
  - 本案例 query set：
    - 主查询：老用户领取新人优惠券时活动资格校验失败的原因
    - 对比查询：新用户与老用户在新人优惠券活动中的资格规则差异
    - 变更查询：最近版本是否调整过新人优惠券活动的用户资格配置
  - query 生成时程序员真正关心的约束：
    - 每条 query 的职责必须单一
    - 每条 query 必须能对应到预期证据类型
    - query 数量要有上限，第一版建议 2 到 4 条
  - 最务实的第一版实现：
    - 用固定 query role schema
    - 让模型只填每个 role 的 query 文本和目标，不要自由发明新 role
  - 工程边界：
    - query 越多不代表效果越好
    - 如果没有 route 和 rerank，盲目多 query 往往只会带来更多噪声
  - 页面输出：
    - query_set[]
    - query_role
    - target_evidence_type
  - 画面建议：
    - 一条问题拆成三到四条并行 query
    - 每条 query 下方挂一类预期 chunk

- **第 6 张｜Step 5：给 query 绑定 route 和检索策略**
  - 标题：真正的 rewrite，不只改词，还在悄悄决定往哪里搜
  - 这一步要解决什么：
    - 不同 query 去不同数据源、不同索引、不同 Top-K、不同 rerank 规则
  - route 设计要回答的四个问题：
    - 去哪个知识源
    - 用什么检索方式
    - 取多少候选
    - 候选回来后按什么规则重排
  - 本案例 route：
    - 主查询 → 历史工单、异常处理文档
    - 对比查询 → 活动规则文档、资格配置说明
    - 变更查询 → 发布说明、活动变更记录
  - 检索策略示例：
    - 规则文档更适合 keyword + embedding 混合召回
    - 工单和复盘更适合语义召回 + 时间过滤
    - 变更记录更适合加版本和时间条件
  - 最务实的第一版实现：
    - 先做一张 query_role × datasource 的静态对照表
    - 不要一开始就做全模型路由
  - 工程边界：
    - route 错了，比 query 写得一般更致命
    - 如果证据可能跨源，不要把路由做成互斥单选
  - 页面输出：
    - route_target[]
    - retrieval_strategy
    - top_k
    - rerank_hint
  - 画面建议：
    - query 卡片进入不同数据源池
    - 每条线上标注 Top-K、过滤条件、排序提示

- **第 7 张｜Step 6：首轮召回后做二次 rewrite**
  - 标题：高价值 rewrite，很多时候发生在第一轮召回之后
  - 这一步要解决什么：
    - 利用首轮结果里出现的高频实体、规则号、版本词，把 query 从泛问题收窄成知识库真术语
  - 首轮最值得抽的信号：
    - 高频实体
    - 高频错误词
    - 高频版本词
    - 高频正式术语
  - 本案例首轮信号：
    - 新人券活动资格规则
    - 资格规则 A12
    - v3.4 版本新增首单用户标签校验
    - 新用户命中标签，老用户不满足条件
  - 二次 rewrite：
    - 原主查询：老用户领取新人优惠券时活动资格校验失败的原因
    - 新主查询：老用户领取新人券时被资格规则 A12 拦截的原因
    - 原对比查询：新用户与老用户在新人优惠券活动中的资格规则差异
    - 新对比查询：老用户是否因不满足 v3.4 新增的首单用户标签校验而无法领取新人券
  - 最务实的第一版实现：
    - 不要让模型直接读完所有 chunk 再自由改写
    - 先做首轮信号提取，再基于信号受控收窄
  - 工程边界：
    - 二次 rewrite 的目标不是变长，而是变准
    - 收窄过度会把真正有价值的备选原因直接过滤掉
  - 页面输出：
    - high_confidence_signals[]
    - refined_query_set[]
  - 画面建议：
    - 上半部分是首轮 query 和召回结果
    - 下半部分是被信号收窄后的 refined query

- **第 8 张｜Step 7：最后输出的是结构化检索计划**
  - 标题：下游真正能消费的，不是句子，而是一份 query plan
  - 这一步要解决什么：
    - 让 retrieval、rerank、context assembly、answer generation 都能稳定消费 rewrite 结果
  - 一个够用的输出结构至少包含：
    - query 文本
    - query role
    - route target
    - term expansions
    - top_k / filter / rerank hint
    - high confidence signals
  - 本案例工程化输出：
    - 主查询：定位老用户领取新人优惠券失败的直接原因
    - 对比查询：比较新用户与老用户在活动资格规则上的差异
    - 变更查询：检查 v3.4 是否新增首单用户标签校验
    - 术语扩展：领取优惠券 / 活动领取 / 用户分群 / 资格规则引擎
    - 路由信息：主查询查工单，对比查询查活动规则文档，变更查询查发布记录
    - 高置信线索：资格规则 A12、首单用户标签、新用户命中、老用户不满足
  - 最务实的第一版实现：
    - 输出成结构化对象，不要只回一段自然语言
    - 字段越稳定，下游越好接
  - 工程边界：
    - 如果 rewrite 输出无法直接驱动下游模块，那它就还只是“提示词润色”
  - 页面输出：
    - retrieval_plan
  - 画面建议：
    - 右侧做成结构化 plan 卡
    - 左侧标注它将如何流向 retrieve、rerank、assemble

- **第 9 张｜第一版落地 checklist + 指标 + 失败点**
  - 标题：如果你今天就做第一版，最值得先把这些东西做出来
  - 第一版最小可用集合：
    - 问题分型器
    - facet 抽取 schema
    - 用户词到知识库词映射表
    - 主查询 / 对比查询 / 变更查询生成器
    - query_role × datasource 路由表
    - 基于首轮召回的二次收窄机制
  - 上线后优先看的指标：
    - Recall@K
    - Top-K 命中正确 chunk 的比例
    - 证据覆盖率
    - 最终答案正确率
    - 单次请求的检索成本和时延
  - 最常见的失败点：
    - 术语扩展过度，扩出了错误概念
    - query 拆太碎，召回噪声变多
    - route 做得过死，漏掉跨源证据
    - 二次 rewrite 收得太窄，把正确原因筛掉
  - 最建议的迭代顺序：
    - 先把分型、facet、query set 做稳定
    - 再补 route
    - 最后再做二次 rewrite
  - 收束金句：
    - Query Rewrite 的目标不是把句子改好看
    - 而是让正确 chunk 更稳定地进到上下文里
  - 画面建议：
    - 上半部分放 checklist
    - 中间放 metrics
    - 下半部分放 failure modes

**文案原则**

- 默认读者已经知道自己为什么来看这组图，不再重复做概念启蒙
- 每张图至少回答一个工程问题，而不只是复述概念
- 尽量写出输入、输出、判断标准、实现约束
- 可以比常规社媒卡片更密，但仍然要维持强结构
- 例子始终围绕“新用户可以领券，老用户资格失败”这一条主线

**视觉建议**

- 主色建议：深蓝 + 青绿，体现系统化和工程感
- 强调色建议：橙色，用于标注风险点、失败模式、关键规则
- 图形语言建议统一为：
  - 用户问题：输入卡片
  - facet：标签组
  - query：搜索条
  - datasource：知识源池
  - refined query：漏斗收窄后的卡片
  - retrieval plan：结构化对象卡
- 如果版面允许，尽量用“输入 → 动作 → 输出”的固定栅格，让程序员一眼能扫到关键信息
