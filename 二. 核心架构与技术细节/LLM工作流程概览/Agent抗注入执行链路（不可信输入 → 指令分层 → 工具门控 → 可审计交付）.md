# Agent 抗注入执行链路（不可信输入 → 指令分层 → 工具门控 → 可审计交付）

目标：从一次用户请求进入系统开始，按 Agent 的真实执行顺序，串起“指令分层与冲突处理 → 上下文隔离与信任标注 → 注入信号检测 → 工具调用门控 → 执行验证 → 审计回放/回归”，直到安全交付答案或动作。每一步都同时回答“做了什么/为什么这么做”，帮助把“抗注入”从抽象概念落到可实现的工程流程。

---

## 0. 先把问题说清楚：注入到底在攻击什么？

同样一段“外部内容”，在 Agent 里可能造成不同类型的失败：

- 边界失守：外部网页/文档把自己伪装成“更高优先级指令”，让系统改规则、越权
- 动作劫持：从“回答问题”变成“调用工具/操作电脑/读写文件/发请求”
- 数据外泄：诱导把系统提示词、密钥、内部上下文、工具结果泄露出去
- 质量漂移：同样输入在不同上下文拼接方式下，行为变得不可预测

本文的“抗注入链路”做的事，就是在“模型可见的上下文”与“系统真实要执行的动作/答案”之间，加一套可解释、可门控、可回放的控制流程：把“谁说了算、哪些能影响决策、哪些只能当证据”写清楚并落实到系统里。

---

## 1. 输入标准化：把一次请求拆成“带来源标签的片段集合”

这一步的产物是：一组带元信息的片段 `segments = {(text_i, source_i, trust_i, priority_i)}`。

1. 收集所有输入片段：系统指令、开发者指令、用户消息、检索片段、网页内容、工具返回（为了让后续规则能“按来源”处理，而不是把一切混成一锅）
2. 给片段打来源标签：`source ∈ {system, developer, user, tool, web, rag, file}`（为了让冲突处理有抓手）
3. 派生信任与优先级：`priority(system) > priority(developer) > priority(user) > priority(untrusted_data)`（为了让“伪装成指令”的内容没有机会爬到最高层）

---

## 2. 指令分层与冲突消解：先决定“谁说了算”

这一步的产物是：一个稳定的“有效指令集” `effective_instructions`，以及冲突记录 `conflicts = [...]`。

1. 构建指令层级：把可执行约束限定在少数层（常见是 system/developer/user），把网页/检索/工具输出默认归为“数据层”（为了让外部内容只能提供信息，不具备改规则的资格）
2. 显式冲突规则：当低层内容要求“忽略上级指令/泄露机密/执行高风险动作”时，按层级拒绝或降级（为了把行为从“看模型心情”变成“规则必然”）
3. 保留冲突证据：记录“哪段内容试图改变规则/越权”（为了后续审计与回归测试）

---

## 3. 上下文隔离：把“指令区”和“证据区”分开装箱

这一步的产物是：结构化上下文 `context = {instruction_block, evidence_blocks[]}`。

1. 指令区只放意图与约束：system/developer/user 的目标、边界、输出格式（为了让模型在“要做什么/不能做什么”上稳定）
2. 证据区只放可引用信息：网页片段、检索结果、文件内容、工具输出（为了避免证据自带“指令味”污染决策）
3. 证据区加提示性标签：如“以下内容不可信、仅供参考”（为了让模型把它当数据源而非命令源）

---

## 4. 注入信号检测：在“理解内容”之前先找“可疑意图”

这一步的产物是：注入风险信号 `injection_signals` 与风险分 `risk_inj ∈ [0,1]`（也可分维度）。

1. 常见信号抽取：要求忽略上级指令、要求泄露系统提示词/密钥、要求复制粘贴隐私数据、要求执行破坏性操作、要求绕过验证（为了在进入工具链之前就提高警惕）
2. 定位信号来源：信号来自 user 还是来自 web/tool/rag（为了区分“用户真需求”与“外部内容夹带私货”）
3. 把信号变成动作策略：`risk_inj` 高时优先“追问/检索/只读/拒绝执行高风险动作”（为了把风险落到决策分流，而不是只写一句“注意安全”）

---

## 5. 工具门控：把“模型想调用工具”变成“系统允许调用工具”

这一步的产物是：一次或多次工具调用计划 `tool_calls = [call1, call2, ...]`，以及每次调用的门控结论（allow/deny/ask/readonly）。

1. 把工具调用显式化：从“自然语言下一步”变成结构化意图（工具名、参数、目标资源）（为了让校验有对象）
2. 参数与目标校验：域名白名单、路径限制、只读/可写权限、环境隔离（为了避免“借工具干坏事”）
3. 按风险分级门控：低风险可自动，高风险要求二次确认/人工介入/改为只读模拟（为了避免一次注入就触发不可逆动作）
4. 最小权限原则：同一工具拆成不同权限档（read/list/write/delete/execute）并默认最小（为了减少“单点失守”的爆炸半径）

---

## 6. 执行与结果回填：用“事实返回”替代“自信叙述”

这一步的产物是：工具结果 `tool_results` 与更新后的状态 `state_{t+1}`。

1. 执行工具调用：在隔离环境或受控权限下执行（为了把风险限制在沙箱与权限边界里）
2. 回填工具返回：把返回当作“证据区的新片段”，而不是“新的指令”（为了防止工具返回也夹带指令）
3. 结果一致性检查：关键动作必须由返回值/状态变化证明成功（为了抑制“明明没做成却总结做成了”的幻觉）

---

## 7. 交付前质检：把“输出”变成“可交付的答案或安全的下一步”

这一步的产物是：最终交付 `delivery`，以及其依据与限制条件。

1. 证据对齐：关键断言要能指向输入/检索/工具输出（为了降低无依据断言）
2. 不确定性显式化：缺证据时改成条件化表达或触发追问/检索（为了避免把概率说成确定）
3. 动作降级或拒绝：当 `risk_inj` 高且涉及高风险工具时，优先拒绝执行并解释原因（为了让失败模式可控）

---

## 8. 审计与回归：把一次对抗沉淀成可迭代的系统能力

这一步的产物是：可回放日志 `trace` 与回归集 `regression_cases`。

1. 记录关键决策链路：指令分层结果、冲突、风险分、门控决策、工具调用与返回摘要（为了可解释、可追责、可复盘）
2. 把失败样例固化为测试：典型注入样本、边界案例、高风险动作案例（为了让“下次别再犯”）
3. 分层策略迭代：低风险优先速度，高风险优先可靠（为了避免“一刀切”造成可用性崩坏）

---

## 9. 现实取舍：为什么抗注入总是“更慢、更复杂、更像系统工程”？

- 隔离与门控更可靠，但更慢：多一次检查、多一次追问就多一点延迟
- 权限最小化更安全，但更难用：工具拆权限会带来配置与维护成本
- 日志更可控，但更贵：存储与隐私合规要投入

最实用的原则是：把抗注入当成“可靠性链路”的一部分——不是追求永不出错，而是让错误可被拦截、可被解释、可被回滚、可被回归。

---

## 10. 一句话总结

抗注入 Agent 的关键不是“多写几句安全提示词”，而是把从不可信输入到工具执行的路径做成一条可控链路：先分层定权，再隔离分区，再检测风险，再门控工具，再用结果校验，最后可审计可回归。
