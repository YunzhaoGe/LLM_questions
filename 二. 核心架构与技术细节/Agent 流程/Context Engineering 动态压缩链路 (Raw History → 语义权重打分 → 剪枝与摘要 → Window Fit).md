# Context Engineering 动态压缩链路 (Raw History → 语义权重打分 → 剪枝/摘要 → Window Fit)

目标：从海量的原始对话历史（Raw History）开始，拆解 Harness 如何在有限上下文窗口（Context Window）内，把“可用信息密度”最大化：语义权重打分 → 剪枝与摘要 → 窗口适配

---

## 1. 原始数据捕获 (Raw History)

1.  **全量记录（可选字段）**：记录用户指令、系统/开发者指令、工具调用参数、工具返回结果、系统报错等（为了保证可回放与可审计）。模型的“思考过程”并不总能/应该被记录，更稳妥的做法是记录“决策痕迹”（例如选了哪个工具、为什么回退、使用了哪些证据）（为了兼顾可解释性与合规/安全）。
2.  **结构化存储**：将文本转为结构化对象（例如包含 `role`, `content`, `timestamp`, `metadata`, `source_id` 的消息）（为了后续能按角色、来源、证据 ID、时间段进行切片与复用）。
3.  **敏感信息最小化**：对日志中的密钥、个人信息、内部链接等进行脱敏或不落盘（为了避免把上下文管理做成数据泄露通道，并降低合规成本）。

---

## 2. 语义权重打分 (Semantic Weighting)

1.  **分层优先级**：高权重常驻（system/developer 指令、目标与约束、State、未解决错误、用户纠正）；中权重可压缩（关键工具摘要、决策依据）；低权重可外置（冗长原文、重复信息）（为了在固定窗口内优先保住“会影响下一步动作”的信息）。
2.  **打分信号**：角色优先级、新近性、与目标相关度、是否被用户显式引用、是否包含可执行要素（路径/参数/错误栈）、是否为真源数据（为了把“重要”变成可计算的排序依据）。
3.  **输出产物**：得到一份“保留/压缩/外置”的排序清单与 Token 预算分配（为了让后续剪枝与拼装变成确定性策略，而不是临时拍脑袋）。

---

## 3. 剪枝与摘要 (Pruning & Summarization)

这一步的产物是：一组经过压缩但保留核心语义的 Message 列表。

1.  **剪枝与截断**：删除低权重消息；对超长工具输出截断并保留 `source_id`/指纹（为了物理减少 Token 占用，同时保持可追溯）。
2.  **可核验摘要**：摘要保留关键字段、参数、结论与证据指针，并区分事实与假设（为了减少摘要引入的失真）。
3.  **State 收敛与防漂移**：将分散信息写入最新 State，并在关键节点用真源数据刷新 State；对关键事实保留引用/片段（为了避免多轮压缩后状态漂移）。

---

## 4. 窗口适配 (Window Fit)

1.  **Token 计数与预算**：拼装前计算 Token，并为 system、目标与约束、State、证据摘要等分配硬预算（为了保证关键区块不被挤掉）。
2.  **拼装顺序**：`System/Developer 指令 + 目标与约束 + 最新 State + 最近高相关消息 + 证据摘要 + 可选背景`（为了让模型先看到规则，再看到可操作状态与证据）。
3.  **超长处理**：先丢低权重背景，再把长证据降级为摘要+`source_id`，其余内容外置到冷存储（为了在不破坏约束与 State 的前提下完成 Window Fit）。

---

## 5. 冷存储回捞 (Recall)

冷存储的价值不在于“存”，而在于“能把需要的那一段精确拉回来”。回捞策略决定了压缩系统是否真正闭环。

1.  **回捞路径**：指针回捞（`source_id`/路径/对象 key 精确取回）与语义回捞（向量/关键词召回候选片段）（为了在“精确核对”和“模糊回忆”两类需求下都能取到真源）。
2.  **触发条件**：用户显式引用、决策需要证据但窗口内缺失、出现不一致信号（互斥结论/重复询问）（为了在证据不足时及时补给，而不是让模型硬编）。
3.  **回捞后处理**：回捞内容立刻进入同一条压缩链路，写回 State 并保留 `source_id`；上下文优先采用真源片段+引用指针（为了避免回捞把窗口撑爆或引入转述污染）。

---

## 6. 质量门槛与回退 (Quality Gates & Fallback)

动态压缩不是一次性的“写摘要”，而是一条会反复运行的生产链路。没有质量门槛，系统会在多次压缩后逐步漂移。

1.  **硬门槛**：压缩后必须仍包含目标与约束、验收标准、最新 State、未解决错误、用户纠正与偏好（为了避免目标漂移与重复试错）。
2.  **校验与一致性检查**：摘要必须可核验（参数/路径/错误/`source_id`），State 更新需检测矛盾、旧值覆盖新值、约束字段缺失（为了在漂移扩大前止损）。
3.  **回退与降级**：优先回退到真源或上一版 State，证据不足时输出“需要回捞的范围/`source_id`”，并对重复摘要设置阈值转为指针保留（为了控制误差累积并显式化不确定性）。

---

## 7. 工程级 Context 优化

1.  **前缀复用/Prompt Caching（若服务支持）**：复用稳定前缀，减少重复传输与计费（为了降低延迟与成本）。
2.  **结构化 State 作为主轴**：让对话历史服务于 State 更新与决策，而不是让模型从叙事里反推状态（为了提升稳定性）。
3.  **缓存与预算治理**：缓存检索/解析/昂贵工具结果并用 `source_id` 去重；为证据与背景设置硬预算与降级策略（为了降低重复成本并保持可预测质量）。

一句话总结：Context Engineering 不是简单的截断，而是**用可计算的策略，把关键信息变成高密度、可核验、可回捞的工作记忆**；并通过回捞闭环与质量门槛，让系统在多轮压缩后仍保持目标一致性与执行稳定性。
