# RAG 场景下，Query Rewrite 到底是怎么一步步改写出来的？

很多人一说 Query Rewrite，脑子里想的还是一件很轻的事：

用户提了一个问题，系统把句子改得更像搜索词，再拿去检索。

这个理解在普通搜索里勉强够用，但放到 RAG 里就明显不够了。因为 RAG 最终要打中的，不是“一个网页结果”，而是一批真正能拿来回答问题的 chunk。你 query 里少了一个关键约束，打偏的可能不是一点，而是整批召回结果。

所以在 RAG 里，Query Rewrite 的本质不是“润色一句话”，而是：**把一句模糊的人类问题，逐步改造成一组更容易召回正确 chunk 的检索输入。**

这篇文章不讲抽象模块，也不讲大而全框架，只走一条主线：

**从一个真实风格的用户输入出发，看看 RAG 系统是怎么一步步把它 rewrite 成可执行检索计划的。**

---

## 一、先把场景固定下来

假设你在做一个企业知识库助手，后面接了这些数据：

- 产品帮助中心文档
- 活动规则说明
- 发布说明
- 历史事故复盘
- 历史工单

用户提问：

> “为什么新用户可以领取新人优惠券，老用户却提示活动资格校验失败？”

这句话对人类来说其实挺清楚的，但对 RAG 来说，它同时混了好几层意图：

- 用户分群对比
- 业务动作
- 异常现象
- 原因追问

如果你把这整句话直接拿去做一次 embedding 检索，很可能会发生三件事：

- 召回很多“优惠券领取流程”，因为“领取优惠券”是高频词
- 召回很多“活动资格是什么”的解释文档，因为“活动资格校验失败”被当成通用概念
- 漏掉真正关键的人群规则和活动配置变更文档，因为问题里没有明确写出知识库里使用的术语

这也是为什么 RAG 场景下的 rewrite 非做不可。它解决的不是“句子不优美”，而是“chunk 根本打不准”。

---

## 二、第一步：先判断这是不是一个“一次召回就够”的问题

**这一步是什么：** 先判断问题复杂度，决定是否进入 rewrite 流程。

**为什么要做：** 因为不是每个问题都值得走多步 RAG 链路。像“某个按钮在哪里”这种问题，一次召回通常就够了；但“新用户可以领券、老用户不行、还涉及资格校验失败”这种问题，天然就是一个多因素问题。如果还走单 query 检索，大概率会把多种意图混在一起。

**怎么做：** 第一版最务实的判断标准很简单：

- 是否有对比结构，比如“一个可以，一个不行”，因为这通常意味着问题里至少有两个检索对象，单次召回很容易把两边证据混在一起
- 是否同时包含场景、异常、原因追问，因为这通常不是“找定义”，而是“找原因链条”，单个 query 很难同时覆盖现象和成因
- 是否明显需要跨多类知识源求证，因为这通常意味着答案不在同一种 chunk 里，而是分散在规则文档、工单、变更记录等不同来源中

这是一套在 RAG 工程里很常见的启发式判断。不同团队会写成不同版本，但底层逻辑通常差不多：**只要问题同时具备“多对象”“多意图”或“多来源”中的两项以上，就很值得先进入 rewrite，而不是直接一次召回。**

更通用地说，开发在自己的项目里做这一步时，最实用的方法通常不是训练一个很复杂的分类器，而是先做一张小型判定表：

- 哪些问题默认走单轮召回，比如定义、位置、单一规则说明
- 哪些问题默认进入 rewrite，比如对比、排查、追因、变更相关
- 每类问题各自对应哪些可观测信号，比如是否有对比词、是否有报错词、是否有时间变化词

第一版只要能把“简单直接搜”和“复杂先 rewrite”这两类稳定分开，就已经很有工程价值。

这个例子里三条几乎全中，所以系统会先给它打上标签：

- 这是一个对比型资格排查问题
- 不适合一次召回
- 需要先 rewrite，再做分路检索

**这一步的输出是什么：** 系统先得到一个前置结论：这题不能拿整句直接去搜，必须先拆检索意图。

---

## 三、第二步：把一句话拆成 RAG 真正关心的几个检索面

**这一步是什么：** 把用户自然语言拆成多个对召回有意义的检索面。

**为什么要做：** 因为在 RAG 里，一句长问题通常混着多种检索目标。如果不拆，embedding 容易把整句话平均掉，最后每一层意思都沾一点，但没有任何一层打得够深。

**怎么做：** 真正在做项目时，重点不是记住“这道题拆成了四个面”，而是先学会一个更通用的拆分方法：**不要按句子拆，要按检索目标拆。**

最常见、最有指导意义的做法，是先问这句话里有没有下面几类维度：

- **对象维度**：系统到底在比较谁、定位谁、排查谁。常见于地区、用户群、产品线、服务、版本
- **行为维度**：问题发生在什么动作里。常见于下单、开户、支付、登录、审批
- **现象维度**：用户表面上看到了什么。常见于报错、失败、超时、结果不一致、性能变慢
- **任务维度**：用户到底要系统干什么。常见于找定义、找流程、找原因、找差异、找变更
- **证据维度**：答案通常应该去哪类材料里找。常见于规则文档、工单、发布说明、事故复盘、FAQ

你可以把它理解成一个很实用的判断框架：

- 如果某个维度会改变召回范围，就值得单独拆出来
- 如果某个维度会改变该查哪类知识源，也值得单独拆出来
- 如果某个维度只是修饰语气，不影响召回和路由，就不值得单独拆

所以不同任务里，维度并不是固定四个，而是跟场景一起变的。

比如：

- 在**故障排查**里，更常拆对象、现象、错误信号、时间、版本
- 在**规则问答**里，更常拆对象、地区、规则主题、生效范围、时间
- 在**流程问答**里，更常拆角色、动作、前置条件、步骤阶段
- 在**差异比较**里，更常拆比较对象、比较维度、差异点、变化时间

放到这个例子里，系统至少应该拆出四个检索面：

- 用户分群对比面：新用户 vs 老用户
- 业务行为面：领取新人优惠券
- 异常现象面：活动资格校验失败
- 原因追问面：为什么会这样

为什么这里拆这四个，而不是别的？

- 因为“新用户 vs 老用户”会直接改变召回到哪一批人群规则文档
- 因为“领取新人优惠券”决定系统应该查活动参与和发券流程，而不是泛营销知识
- 因为“活动资格校验失败”决定要优先找异常说明和工单，而不是普通操作文档
- 因为“为什么会这样”说明用户要的是成因，不是定义或步骤

拆完之后，系统才能继续判断：哪些内容应该进主查询，哪些应该拆成子查询，哪些只适合作为扩展词。

更通用地说，开发在自己的场景里做维度拆分时，可以固定用一个三步顺序：

- 先拆会改变召回范围的维度，比如对象、地区、版本、时间
- 再拆会改变文档类型的维度，比如规则、流程、异常、变更
- 最后再决定哪些只是扩展词，不必单独成面

如果拿不准某个维度值不值得拆，一个很好用的判断问题是：**去掉它之后，Top-K 会不会明显换一批结果。** 如果会，它大概率就值得保留成一个独立检索面。

**这一步的输出是什么：** 系统不再面对一句混合问题，而是拿到四个可管理的检索面。这一步之后，RAG 才有机会把召回路径分开。

---

## 四、第三步：把用户语言翻译成知识库语言

**这一步是什么：** 把用户说法映射成知识库里真正出现的术语。

**为什么要做：** 用户语言和知识库语言，经常根本不是一套口径。用户会说“领券失败”“资格卡住了”，但知识库里可能写的是“活动参与校验”“用户分群准入拦截”“活动门槛开关”。如果不做术语映射，召回往往只会打中表面相关 chunk，而打不中真正有价值的规则文档。

**怎么做：** 这一步通常做两层扩展：

- 同义映射：领取新人优惠券 → 活动领取、优惠券发放、营销活动参与
- 领域映射：活动资格校验失败 → 用户分群准入失败、活动门槛拦截、资格规则校验未通过

这里有一个很重要的边界：映射不是替换，而是扩展。系统不应该把“活动资格校验失败”直接替换成“用户分群准入失败”，而是应该同时保留两套词，让后面的召回面更宽、排序更准。

更通用地说，开发在不同业务里做术语映射时，最值得先积累三类资产：

- 用户常用说法和知识库正式说法之间的同义词表
- 历史别名、内部简称、旧版本名称之间的别名表
- 某类口语问题通常会落到哪组专业术语上的领域映射表

第一版不用追求全覆盖。先把最常打偏召回的几十组高频词配好，通常就足以明显改善效果。

**这一步的输出是什么：** 系统得到一组术语扩展：

- 用户词：领取优惠券、活动资格校验失败、新用户、老用户
- 知识库词：活动领取、用户分群、资格规则引擎、活动参与门槛

这一步之后，rewrite 才真正开始“会说知识库的话”。

---

## 五、第四步：不要只改写一句话，而要拆成多条 Query

**这一步是什么：** 把原问题拆成主查询、对比查询、变更查询。

**为什么要做：** 这是 RAG 和普通搜索最大的差别之一。普通搜索很多时候只需要一个改写后的 query；但 RAG 往往更适合同时准备几条功能不同的 query，分别召回，再合并证据。因为这个例子本质上不是单跳问题，而是：

- 找老用户的资格失败原因
- 找新用户与老用户的规则差异
- 找最近有没有活动配置变更

如果把这些都塞进同一条 query，谁都不够强。

**怎么做：** 对这个例子，一个更合理的拆法是：

- 主查询：老用户领取新人优惠券时活动资格校验失败的原因
- 对比查询：新用户与老用户在新人优惠券活动中的资格规则差异
- 变更查询：最近版本是否调整过新人优惠券活动的用户资格配置

注意，这里不是为了“多搜几次而多搜几次”，而是因为这三条 query 对应的 chunk 类型完全不同：

- 主查询更容易打中异常说明和工单
- 对比查询更容易打中活动规则文档和资格配置差异
- 变更查询更容易打中发布说明和活动变更记录

更通用地说，开发在自己的系统里拆 query 时，可以先把 query 的角色固定成少数几类：

- 主查询：围绕用户核心问题直接召回
- 对比查询：当问题包含两个对象或两个状态时使用
- 变更查询：当问题含有“最近为什么变了”时使用
- 校验查询：当系统已经怀疑某个原因，需要单独验证时使用

不是每个问题都需要四种 query，但如果一条 query 同时承担“找现象、找差异、找变更”三件事，通常就应该拆。

**这一步的输出是什么：** 系统得到的已经不是一个 rewrite 句子，而是一个小型 query set。这一步会直接改变 RAG 的召回面。

---

## 六、第五步：给每条 Query 指定更适合的召回目标

**这一步是什么：** 为不同 query 绑定不同数据源和召回策略。

**为什么要做：** 在 RAG 里，问题经常不是“要不要搜”，而是“该往哪里搜”。同一句 query 如果统一搜全库，通常会带来很多噪声 chunk。而如果你先给 query 做路由，效果通常会稳定很多。

**怎么做：** 对这个例子，比较合理的路由是：

- 主查询优先去历史工单和异常处理文档
- 对比查询优先去活动规则文档和资格配置说明
- 变更查询优先去发布说明和活动变更记录

这一步的工程意义非常直接：

- 不是所有 query 都进同一个库分片
- 不是所有 query 都用同一个 Top-K
- 不是所有 query 都适合用同样的重排序规则

这说明 rewrite 在 RAG 里不只是改写文本，它还在悄悄决定检索路线。

更通用地说，开发在自己的项目里做路由时，最实用的起点通常不是模型路由，而是一张“query 角色 × 数据源类型”的对照表。

比如：

- 规则类 query 优先查制度文档、配置说明
- 异常类 query 优先查工单、故障手册、复盘
- 变更类 query 优先查发布记录、公告、版本说明
- 流程类 query 优先查帮助中心、SOP、操作指引

这样做的好处是，rewrite 结果一出来，下游就知道先去哪里搜，而不是统一把所有 query 扔进全库。

**这一步的输出是什么：** 系统拿到的是一组“query + route”：

- 每条 query 去哪个数据源
- 更可能召回什么类型的 chunk
- 哪类结果后面应该更优先进入重排序

---

## 七、第六步：看首轮召回结果，再做一次二次 Rewrite

**这一步是什么：** 基于首轮召回结果，进一步收窄 query。

**为什么要做：** 这是 RAG 里最像真实系统的一步。很多人理解 rewrite，只想到“检索前重写一次”。但复杂问题里，真正有价值的 rewrite 往往发生在首轮召回之后。因为你看过第一批 chunk 后，才真正知道知识库里到底用了哪套术语、哪个规则词反复出现、哪个版本变更最关键。

**怎么做：** 假设首轮召回后，系统反复看到这些信息：

- 新人券活动资格规则
- 资格规则 A12
- v3.4 版本新增首单用户标签校验
- 新用户命中标签，老用户不满足条件

那第二轮 rewrite 就会明显比第一轮更收敛。

原来的主查询：

- 老用户领取新人优惠券时活动资格校验失败的原因

可以进一步收窄成：

- 老用户领取新人券时被资格规则 A12 拦截的原因

原来的对比查询：

- 新用户与老用户在新人优惠券活动中的资格规则差异

也可以进一步收窄成：

- 老用户是否因不满足 v3.4 新增的首单用户标签校验而无法领取新人券

这一步的关键，不是让 query 更长，而是让它更贴近知识库里真实出现的 chunk 语言。

更通用地说，开发在自己的系统里做二次 rewrite 时，最值得重点观察四类首轮信号：

- 高频实体，比如地区名、服务名、规则号、模块名
- 高频错误词，比如错误码、失败阶段、异常类别
- 高频版本词，比如版本号、开关名、发布日期
- 高频正式术语，比如文档里真实使用的概念表达

只要某类词在首轮结果里高频出现、但在原问题里没有出现，它通常就是第二轮 rewrite 最有价值的收窄抓手。

**这一步的输出是什么：** 系统得到一组二次收窄后的 query。它们明显更贴近首轮召回中的高价值术语，这时后续召回会稳定很多。

---

## 八、第七步：最终输出的，不是一句话，而是一套 RAG 检索计划

**这一步是什么：** 把整个 rewrite 过程的结果整理成可直接驱动 RAG 管线的输入。

**为什么要做：** 因为 RAG 下游要做的不只是召回，还包括去重、重排序、chunk 组装、证据排序和最终回答。如果 rewrite 输出只是一句自然语言，下游其实很难稳定使用它。

**怎么做：** 对这个例子，一个更像工程产物的最终输出，通常会包含这些层次：

- 主查询：定位老用户领取新人优惠券失败的直接原因
- 对比查询：比较新用户与老用户在活动资格规则上的差异
- 变更查询：检查 v3.4 是否新增首单用户标签校验
- 术语扩展：领取优惠券 / 活动领取 / 用户分群 / 资格规则引擎
- 路由信息：主查询查工单，对比查询查活动规则文档，变更查询查发布记录
- 首轮高置信候选：资格规则 A12、首单用户标签、新用户命中、老用户不满足

如果再往下游走一步，这些信息还会继续影响：

- 哪些 chunk 先送去重排序
- 哪些 chunk 应该按“用户分群差异”组织
- 哪些 chunk 应该保留为“活动配置变更证据”

更通用地说，开发在设计 rewrite 输出时，最好不要只输出一个 query 字符串，而是至少保留四层信息：

- query 本身
- query 的角色
- query 的路由目标
- query 依赖的高置信线索

只有这样，下游的检索器、重排序器、上下文组装器才知道应该怎么消费这次 rewrite 的结果。否则 rewrite 做得再好，后面也只能把它当成一条普通搜索词来用。

**这一步的输出是什么：** 最终输出已经不是“改写后的问题”，而是一套可以直接驱动 RAG 召回与证据组织的检索计划。

---

## 九、把这条 RAG 主线再压缩一次

如果把整个流程压缩一下，RAG 场景下的 Query Rewrite 实际上是在做七件事：

- 先判断这是不是一个不能一次召回解决的问题
- 再把一句话拆成多个检索面
- 再把用户语言翻译成知识库语言
- 再把问题拆成多条功能不同的 query
- 再给每条 query 分配更适合的数据源
- 再根据首轮召回结果做二次收窄
- 最后输出一套能驱动 RAG 的检索计划

你会发现，RAG 里的 rewrite 和普通搜索最大的差别就在这里：

**它不是把句子“改好看”，而是把召回路径“改正确”。**

## 十、如果你今天就要做第一版 RAG Rewrite

第一版最值得做的，不是一个万能智能体，而是这五样东西：

- 一个问题分型器，判断是否需要 rewrite
- 一套检索面拆分规则
- 一套用户词到知识库词的映射表
- 一套主查询 / 对比查询 / 变更查询的拆分规则
- 一个基于首轮召回做二次收窄的简单机制

只要这五样东西在，用户那句“为什么新用户可以领取新人优惠券，老用户却提示活动资格校验失败”进来之后，你的系统就已经不再是在“换个说法”，而是在构造一套更容易打中正确 chunk 的 RAG 输入。

---

## 结语：为什么 RAG 里的 Query Rewrite 会更重要

在普通理解里，Query Rewrite 好像只是给搜索框润色一下输入。

但在 RAG 场景里，它真正做的是三件更重的事：

- 把一个大问题拆成多个更容易命中 chunk 的 query
- 把用户语言翻译成知识库语言
- 把 query 和数据源、召回目标、二次收窄过程绑在一起

所以，如果要把这篇文章压成一句话，我会这样说：

**RAG 场景里的 Query Rewrite，本质上不是改写一句问题，而是把一句模糊问题拆成一套更容易召回正确 chunk 的检索计划。**
