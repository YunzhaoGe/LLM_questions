# LLM 输入预处理链路（Token ID → 对齐/掩码 → 向量张量）

把 Token ID 进入主干网络之前的准备链路拆开来看：上下文窗口检查 → batch 对齐与 Padding → 有效位置标记与因果约束 → Embedding 查表 → 交接主干网络。整个过程可以概括为：先检查长度边界，再整理输入形状，然后补充可见性约束，最后把离散编号映射成连续向量。

---

## 1. 上下文窗口：先确认输入总长度是否落在上限内

1. 长度检查：统计这次输入的总 Token 数量，并与模型的上下文窗口上限比较（为了先确认这次前向传播在物理上是否可执行）
2. 预算构成：把当前用户输入、system prompt、历史消息等内容一起算进总长度（为了避免把“窗口大小”误解成只约束当前这一条输入）
3. 超长处理：一旦超出上限，就必须先截断、裁剪或重组（为了在后续张量化之前先满足显存与计算边界）

---

## 2. 批次对齐：把变长序列整理成统一的二维编号表

1. 构造 batch：把多条样本放到同一批次里处理，而不是完全逐条单独前向；这里的 batch 指一次前向传播中被同时处理的多条输入样本（为了提高并行效率）
2. Padding 补齐：将较短序列补到当前 batch 的统一长度；Padding 是为缺失位置填入统一占位符的对齐操作（为了让输入可以拼成规则矩阵）
3. 保留编号形态：这一步得到的仍然是二维 Token ID 表，而不是向量张量（为了先解决形状问题，再进入连续表示空间）

---

## 3. 有效位置标记：区分真实 Token 与补位位置

1. 标记真实内容：为每个位置准备有效性信息，区分真实 Token 和 PAD 占位（为了避免模型把 PAD 位置当成有效内容）
2. 约束计算范围：后续注意力计算只应把真实位置当成可参与对象（为了防止无效位置污染打分结果）
3. 约束并不删除张量：这些位置通常仍保留在输入形状里，只是在后续计算中被判定为无效；也就是说，位置仍然存在于张量中，不等于它仍然参与后续注意力计算（为了说明“保留位置”和“参与计算”是两件事）

---

## 4. 因果约束：限制当前位置访问右侧未来

1. 自回归规则：当前位置只能访问自己以及左边历史，不能访问右边尚未生成的位置（为了保证“预测下一个 Token”这件事成立）
2. 因果掩码：系统会提前构造可见范围约束，通常表现为一个下三角可见、上三角屏蔽的 mask 矩阵，用来限制每个位置能访问的上下文（为了把“只能访问左侧历史与当前位置”变成可执行规则）
3. 真正生效时机：这类约束通常会在后续注意力打分阶段执行，而不是在输入阶段直接删除右侧位置（为了区分“规则准备”和“规则执行”）

---

## 5. Embedding：把二维编号表换成三维向量张量

1. 查表映射：根据每个 Token ID，从 Embedding 矩阵中取出对应的一行向量（为了把离散编号转换成连续表示）
2. 批量展开：原来二维表里的每个整数位置，都会被替换成一条高维向量；这里的“高维”表示每个位置不再对应单个标量，而是对应一组连续特征值（为了让每个位置都具备可计算的特征表示）
3. 张量成形：输入形状从 `Batch × Seq` 变成 `Batch × Seq × Hidden_size`；其中 `Hidden_size` 表示模型为每个 Token 分配的特征维度（为了交付主干网络能够直接接手的输入张量）

---

## 6. 交接主干网络：真正送进去的是向量张量与约束信息

1. 输入主体：Embedding 输出的三维向量张量会作为主干网络的直接输入（为了提供每个位置的初始连续表示）
2. 配套约束：有效位置标记、因果约束和窗口边界会与输入一起进入后续计算链路（为了让注意力机制按正确边界工作）
3. 角色边界：Embedding 负责把编号映射成向量，约束信息负责规定这些向量后续如何参与计算；前者定义输入表示，后者定义计算边界与可见性规则（为了避免把“向量化”和“上下文规则”混成一件事）

---

## 7. 一句话复盘

Token ID 在进入主干网络前，通常要先完成四类准备：先检查上下文窗口是否允许这次输入，再把变长序列补齐成统一的二维编号表，同时准备好有效位置标记和因果约束，最后才通过 Embedding 把编号映射成三维向量张量。也就是说，主干网络接到的不是单独的 Embedding 输出，而是由向量输入与约束条件共同组成的完整输入。
